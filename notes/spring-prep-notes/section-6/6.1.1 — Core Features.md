## `@SpringBootApplication` ‚Äî already covered in 3.1.1

Three composed annotations: `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`. Main class at root package. Component scanning flows downward.

One addition worth noting here ‚Äî `@SpringBootApplication` accepts `exclude` to disable specific auto-configurations:

```java
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

This is the same as `@EnableAutoConfiguration(exclude = ...)` ‚Äî just more concise. Covered in more depth in 6.2.3.

## `CommandLineRunner` vs `ApplicationRunner` ‚Äî already covered in 3.1.1

The single exam-tested difference: argument type.

- `CommandLineRunner` ‚Üí `run(String... args)` ‚Äî raw strings
- `ApplicationRunner` ‚Üí `run(ApplicationArguments args)` ‚Äî parsed, structured

```java
@Component
@Order(1)
public class StartupTask implements CommandLineRunner {

    @Override
    public void run(String... args) throws Exception {
        // args[0], args[1] etc ‚Äî raw command line strings
        // e.g. --server.port=8080 comes through as the string "--server.port=8080"
    }
}

@Component
@Order(2)
public class StartupLogger implements ApplicationRunner {

    @Override
    public void run(ApplicationArguments args) throws Exception {
        // structured access
        Set<String> optionNames = args.getOptionNames();   // --key=value pairs
        List<String> nonOptionArgs = args.getNonOptionArgs(); // positional args
        boolean hasDebug = args.containsOption("debug");
        List<String> profiles = args.getOptionValues("spring.profiles.active");
    }
}
```

Both run after the context is fully started, before the application is considered ready. Both can be ordered with `@Order`. Both can throw exceptions ‚Äî an exception causes the application to exit.

## Embedded server ‚Äî already covered in 3.1.1

Default is Tomcat. Switch by excluding Tomcat and adding Jetty or Undertow. Key properties:

```properties
server.port=8080          # change port
server.port=0             # random port
server.servlet.context-path=/api  # URL prefix for all endpoints
```

## Fat JAR packaging ‚Äî already covered in 3.1.4

`spring-boot-maven-plugin` repackages to include all dependencies. Run with `java -jar`. The plugin is required ‚Äî without it, `mvn package` produces a thin JAR.

## `SpringApplication` customisation

Beyond the basic `SpringApplication.run()`, you can customise the startup:

```java
public static void main(String[] args) {
    SpringApplication app = new SpringApplication(OrderApplication.class);

    app.setBannerMode(Banner.Mode.OFF);          // disable the ASCII art banner
    app.setLazyInitialization(true);             // all beans lazy by default
    app.setDefaultProperties(Map.of(
        "server.port", "9090"                    // default properties (lowest precedence)
    ));
    app.addListeners(new ApplicationStartedListener());  // add event listener

    app.run(args);
}
```

Or using the builder:
```java
new SpringApplicationBuilder(OrderApplication.class)
    .bannerMode(Banner.Mode.OFF)
    .profiles("prod")
    .run(args);
```

## Spring Boot application events

Spring Boot fires a sequence of events during startup. The exam doesn't test these deeply but you should recognise them:

```
ApplicationStartingEvent         ‚Äî very early, before context created
ApplicationEnvironmentPreparedEvent ‚Äî Environment ready, context not yet
ApplicationContextInitializedEvent  ‚Äî context created, beans not loaded
ApplicationPreparedEvent         ‚Äî context loaded, not refreshed
ApplicationStartedEvent          ‚Äî context refreshed, runners not yet called
ApplicationReadyEvent            ‚Äî runners called, app ready to serve requests
ApplicationFailedEvent           ‚Äî startup failed
```

`ApplicationReadyEvent` is the one worth knowing ‚Äî it fires when the application is fully ready to handle requests. Use it for post-startup tasks that shouldn't run in `CommandLineRunner` (e.g. tasks that need the full application to be serving traffic).

```java
@Component
public class ReadyListener implements ApplicationListener<ApplicationReadyEvent> {

    @Override
    public void onApplicationEvent(ApplicationReadyEvent event) {
        System.out.println("Application fully ready");
    }
}
```

## Spring Initializr

The exam mentions Spring Initializr (`start.spring.io`) as the standard project bootstrapping tool. Know what it does:

- Generates a Maven or Gradle project skeleton
- Pre-configured with selected dependencies
- Includes `spring-boot-starter-parent` as the parent POM
- Creates the main application class with `@SpringBootApplication`
- Sets up the standard `src/main/java` / `src/test/java` structure

Nothing deep here ‚Äî the exam just tests that you know it exists and what it produces.

## Plain-English summary

- `@SpringBootApplication` = `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`. Can `exclude` auto-configs.
- `CommandLineRunner` = raw `String[]`. `ApplicationRunner` = parsed `ApplicationArguments`. Both run post-startup, orderable with `@Order`.
- Default embedded server = Tomcat. Switch by excluding and adding alternative starter.
- Fat JAR requires `spring-boot-maven-plugin`. Run with `java -jar`.
- `SpringApplication` is customisable ‚Äî banner, lazy init, default properties, event listeners.
- `ApplicationReadyEvent` fires when app is fully ready to serve requests.
- Spring Initializr = project skeleton generator at `start.spring.io`.
