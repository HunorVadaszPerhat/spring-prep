# Objective 6.3 — Spring Boot Actuator

## What Actuator is

Actuator adds production-ready operational features to your Spring Boot application — health checks, metrics, environment inspection, thread dumps, and more — all exposed via HTTP endpoints or JMX. Add the starter and you get a suite of monitoring and management endpoints with minimal configuration.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

---

## Default endpoints — what's exposed out of the box

⚠️ **EXAM TRAP**: This is HIGH VALUE. Know exactly which endpoints are exposed by default over HTTP.

**Over HTTP — exposed by default:** `health` and `info` only.

**Over JMX — exposed by default:** all endpoints.

```properties
# See what's exposed by default
# HTTP: only /actuator/health and /actuator/info
# JMX: all endpoints
```

All other endpoints (`metrics`, `env`, `beans`, `conditions`, etc.) exist but are **not** exposed over HTTP by default. You must explicitly enable them.

---

## Exposing endpoints

```properties
# Expose specific endpoints over HTTP
management.endpoints.web.exposure.include=health,info,metrics,env

# Expose all endpoints over HTTP (wildcard)
management.endpoints.web.exposure.include=*

# Expose all then exclude sensitive ones
management.endpoints.web.exposure.include=*
management.endpoints.web.exposure.exclude=env,beans

# Change the base path (default is /actuator)
management.endpoints.web.base-path=/management

# Move actuator to a different port (useful for securing separately)
management.server.port=8081
```

⚠️ **EXAM TRAP**: Exposing an endpoint and enabling it are different things. All endpoints are **enabled** by default (except `shutdown`). Enabling controls whether the endpoint exists. Exposing controls whether it's accessible over HTTP/JMX. An endpoint can be enabled but not exposed — it exists but can't be reached.

```properties
# Enable/disable specific endpoints
management.endpoint.health.enabled=true       # default
management.endpoint.shutdown.enabled=true     # shutdown is disabled by default
management.endpoint.beans.enabled=false       # disable beans endpoint entirely

# Disable all endpoints then selectively enable
management.endpoints.enabled-by-default=false
management.endpoint.health.enabled=true
management.endpoint.info.enabled=true
```

## Key built-in endpoints

| Endpoint | Description |
|---|---|
| `/actuator/health` | Application health status |
| `/actuator/info` | Application information |
| `/actuator/metrics` | Micrometer metrics |
| `/actuator/env` | Environment properties |
| `/actuator/beans` | All Spring beans |
| `/actuator/conditions` | Autoconfiguration conditions report |
| `/actuator/mappings` | All `@RequestMapping` paths |
| `/actuator/loggers` | Logger levels — readable and writable |
| `/actuator/threaddump` | JVM thread dump |
| `/actuator/heapdump` | JVM heap dump |
| `/actuator/shutdown` | Graceful shutdown (disabled by default) |
| `/actuator/httptrace` | Recent HTTP request traces |

## Health endpoint — detail levels

```properties
# Show full health details (default shows only UP/DOWN)
management.endpoint.health.show-details=always   # always show details
management.endpoint.health.show-details=when-authorized  # only authenticated users
management.endpoint.health.show-details=never    # default — status only
```

With `show-details=always`:
```json
{
  "status": "UP",
  "components": {

      "status": "UP",
```

## Custom health indicators

Implement `HealthIndicator` to add your own health check:

```java
@Component
public class PaymentGatewayHealthIndicator implements HealthIndicator {

    private final PaymentGateway paymentGateway;

    public PaymentGatewayHealthIndicator(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }

    @Override
    public Health health() {
        try {
            boolean reachable = paymentGateway.ping();
            if (reachable) {
                return Health.up()
                    .withDetail("gateway", "Stripe")
                    .withDetail("responseTime", "45ms")
                    .build();
            } else {
                return Health.down()
                    .withDetail("gateway", "Stripe")
                    .withDetail("reason", "Ping failed")
                    .build();
            }
        } catch (Exception e) {
            return Health.down(e).build();  // includes exception in details
        }
    }
}
```

The bean name determines the key in the health response — `PaymentGatewayHealthIndicator` → `paymentGateway` in the response.

## Micrometer metrics — HIGH VALUE

Actuator uses **Micrometer** as its metrics facade — the same relationship as SLF4J to logging. Micrometer provides a vendor-neutral API; you swap backends (Prometheus, Datadog, CloudWatch) without changing application code.

⚠️ **EXAM TRAP**: Know the four core Micrometer meter types — their names, what they measure, and when to use each.

### `Counter`

Monotonically increasing count. Records how many times something happened. Can only go up.

```java
@Service
public class OrderService {

    private final Counter orderCounter;

    public OrderService(MeterRegistry registry) {
        this.orderCounter = Counter.builder("orders.placed")
            .description("Number of orders placed")
            .tag("region", "eu-west")
            .register(registry);
    }

    public void placeOrder(Order order) {
        // business logic
        orderCounter.increment();       // increment by 1
        orderCounter.increment(3.0);    // increment by arbitrary amount
    }
}
```

Use for: request counts, error counts, events.

---

### `Gauge`

A value that can go up and down — a snapshot of a current state.

```java
@Service
public class QueueMonitor {

    public QueueMonitor(MeterRegistry registry, OrderQueue queue) {
        Gauge.builder("orders.queue.size", queue, OrderQueue::size)
            .description("Current number of orders in queue")
            .register(registry);
        // Micrometer calls queue::size on demand — always fresh value
    }
}
```

Use for: queue sizes, active connections, memory usage, cache sizes. The value is sampled when Micrometer reads it — the lambda/method reference is called each time.

⚠️ **EXAM TRAP**: `Gauge` is not incremented — it's sampled. You provide a supplier function that returns the current value. It can go up or down.

---

### `Timer`

Measures duration and count of events. Records both how long something took and how many times it happened.

```java
@Service
public class OrderService {

    private final Timer orderTimer;

    public OrderService(MeterRegistry registry) {
        this.orderTimer = Timer.builder("orders.processing.time")
            .description("Time to process an order")
            .tag("type", "standard")
            .register(registry);
    }

    public void processOrder(Order order) {
        orderTimer.record(() -> {
            // timed code block
            doProcessing(order);
        });
    }

    // Or manually
    public void processOrderManual(Order order) {
        Timer.Sample sample = Timer.start(registry);
        try {
            doProcessing(order);
        } finally {
            sample.stop(orderTimer);
        }
    }
}
```

Exposes: count, total time, max time, percentiles.

Use for: HTTP request duration, database query time, external API call duration.

---

### `DistributionSummary`

Records the distribution of a value — similar to `Timer` but for non-time values (sizes, amounts).

```java
@Service
public class OrderService {

    private final DistributionSummary orderValueSummary;

    public OrderService(MeterRegistry registry) {
        this.orderValueSummary = DistributionSummary.builder("orders.value")
            .description("Distribution of order values")
            .baseUnit("dollars")
            .register(registry);
    }

    public void placeOrder(Order order) {
        orderValueSummary.record(order.getTotal().doubleValue());
    }
}
```

Exposes: count, total amount, max, percentiles.

Use for: request payload sizes, order values, file sizes — anything you want distribution statistics on.

---

### The four meter types compared

| Type | Goes up/down | Measures | Use for |
|---|---|---|---|
| `Counter` | Up only | Event count | Requests, errors, events |
| `Gauge` | Up and down | Current state | Queue size, connections, memory |
| `Timer` | Up only | Duration + count | Method execution time |
| `DistributionSummary` | Up only | Value distribution | Payload sizes, monetary amounts |

## Custom metrics via `MeterRegistry`

Inject `MeterRegistry` directly for quick metric creation:

```java
@RestController
public class OrderController {

    private final MeterRegistry registry;

    public OrderController(MeterRegistry registry) {
        this.registry = registry;
    }

    @PostMapping("/orders")
    public ResponseEntity<Order> createOrder(@RequestBody Order order) {
        registry.counter("orders.created", "status", "success").increment();
        return ResponseEntity.ok(orderService.save(order));
    }
}
```

Tags add dimensions to metrics — the same metric name with different tag values can be queried separately. `"status", "success"` is a tag key-value pair.

## Securing Actuator endpoints

Actuator endpoints should be secured in production. Two approaches:

**Expose on a separate management port** (simplest for internal network security):

```properties
management.server.port=8081  # actuator on different port, not exposed publicly
```

**Secure via Spring Security:**

```java
// Exam style
@Configuration
public class ActuatorSecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http.requestMatcher(EndpointRequest.toAnyEndpoint())
            .authorizeRequests()
                .requestMatchers(EndpointRequest.to(HealthEndpoint.class,
                                                     InfoEndpoint.class)).permitAll()
                .anyRequest().hasRole("ACTUATOR_ADMIN")
                .and()
            .httpBasic();
    }
}
```

```java
// Modern style
@Bean
public SecurityFilterChain actuatorSecurity(HttpSecurity http) throws Exception {
    http.securityMatcher(EndpointRequest.toAnyEndpoint())
        .authorizeHttpRequests(auth -> auth
            .requestMatchers(EndpointRequest.to(HealthEndpoint.class)).permitAll()
            .anyRequest().hasRole("ACTUATOR_ADMIN")
        )
        .httpBasic(Customizer.withDefaults());
    return http.build();
}
```

## `info` endpoint — application information

The `/actuator/info` endpoint displays arbitrary information about the application. Populate it via properties:

```properties
info.app.name=Order Service
info.app.version=1.2.3
info.app.description=Handles order processing

# Git info (requires spring-boot-maven-plugin git commit info)
info.git.branch=main
info.git.commit.id=abc1234
```

Or programmatically:
```java
@Component
public class AppInfoContributor implements InfoContributor {

    @Override
    public void contribute(Info.Builder builder) {
        builder.withDetail("startup-time", Instant.now().toString());
        builder.withDetail("active-orders", orderService.countActive());
    }
}
```

## Plain-English summary

- Add `spring-boot-starter-actuator` to enable production endpoints.
- HTTP default: only `health` and `info` exposed. JMX default: all endpoints.
- Enable all HTTP endpoints: `management.endpoints.web.exposure.include=*`.
- Enabled ≠ exposed. All endpoints enabled by default except `shutdown`.
- `HealthIndicator` — implement to add custom health checks. Bean name becomes key in response.
- Four Micrometer types: `Counter` (count up), `Gauge` (current value, up/down), `Timer` (duration + count), `DistributionSummary` (value distribution).
- Inject `MeterRegistry` to create metrics programmatically.
- Secure actuator via separate management port or Spring Security with `EndpointRequest`.
