# Objective 6.2.3 — Overriding Default Configuration

## The three ways to override Boot's defaults

Spring Boot's autoconfiguration is designed to step aside whenever you provide your own configuration. There are three distinct mechanisms — each for a different scenario.

---

## Mechanism 1: Declare your own bean (`@ConditionalOnMissingBean`)

The most common approach. As covered in 6.2.2, autoconfigured beans are guarded by `@ConditionalOnMissingBean`. Declare a bean of the same type and Boot's default is suppressed.

```java
// Boot's default ObjectMapper is suppressed — yours takes over
@Configuration
public class JacksonConfig {

    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false)
            .setSerializationInclusion(JsonInclude.Include.NON_NULL);
    }
}
```

```java
// Boot's default DataSource is suppressed — yours takes over
@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource dataSource() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/orders");
        config.setMaximumPoolSize(20);
        return new HikariDataSource(config);
    }
}
```

⚠️ **EXAM TRAP**: This only works because Boot's autoconfigured beans use `@ConditionalOnMissingBean`. If you declare a bean and Boot's autoconfiguration doesn't use that condition, both beans exist and you get `NoUniqueBeanDefinitionException`. Always verify the autoconfiguration uses `@ConditionalOnMissingBean` before relying on this pattern.

---

## Mechanism 2: Use properties to configure defaults

Most autoconfigured beans read from `application.properties` — Spring Boot exposes hundreds of standard properties that drive autoconfiguration behaviour without requiring any Java code changes.

```properties
# Embedded server
server.port=9090
server.servlet.context-path=/api
server.tomcat.max-threads=200
server.tomcat.accept-count=100

# DataSource — drives DataSourceAutoConfiguration
spring.datasource.url=jdbc:postgresql://localhost:5432/orders
spring.datasource.username=postgres
spring.datasource.password=secret
spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.connection-timeout=30000

# JPA
spring.jpa.hibernate.ddl-auto=validate
spring.jpa.show-sql=false
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.PostgreSQLDialect

# Jackson
spring.jackson.serialization.write-dates-as-timestamps=false
spring.jackson.default-property-inclusion=non_null

# Logging
logging.level.com.example=DEBUG
logging.level.org.springframework.web=INFO
logging.file.name=application.log
```

This is the preferred approach for simple configuration changes — no Java code needed, environment-specific values can be externalised.

---

## Mechanism 3: Exclude autoconfiguration entirely

When you don't want Boot to configure something at all — not just override it, but disable it entirely.

**Via `@SpringBootApplication(exclude = ...)`:**

```java
@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,        // don't configure DataSource
    SecurityAutoConfiguration.class,          // don't configure Spring Security
    ErrorMvcAutoConfiguration.class           // don't configure default error page
})
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

**Via `@EnableAutoConfiguration(exclude = ...)`:**

```java
@Configuration
@EnableAutoConfiguration(exclude = DataSourceAutoConfiguration.class)
public class AppConfig { }
```

**Via `spring.autoconfigure.exclude` property:**

```properties
spring.autoconfigure.exclude=\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  org.springframework.boot.autoconfigure.security.servlet.SecurityAutoConfiguration
```

This property approach is useful when you need to disable autoconfiguration without changing code — for example in a specific deployment environment.

⚠️ **EXAM TRAP**: The `exclude` attribute takes the **fully qualified class name** (or the class literal in Java). You can't use short names. The property approach always requires fully qualified names.

---

## Choosing between the three mechanisms

| Scenario | Mechanism |
|---|---|
| Need a customised version of the autoconfigured bean | Declare your own `@Bean` |
| Need to tweak autoconfigured bean behaviour | Use `application.properties` |
| Don't want the autoconfiguration at all | Use `exclude` |

In practice: try properties first — it's the least invasive. If properties don't give you enough control, declare your own bean. If you need to prevent Boot from touching something entirely, use exclude.

## Customising with `WebMvcConfigurer` — MVC example

For Spring MVC, Boot provides `WebMvcAutoConfiguration`. Rather than replacing it entirely, you can extend it by implementing `WebMvcConfigurer`:

```java
@Configuration
public class MvcConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://frontend.example.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE");
    }

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoggingInterceptor())
            .addPathPatterns("/api/**");
    }

    @Override
    public void configureMessageConverters(List<HttpMessageConverter<?>> converters) {
        converters.add(new MappingJackson2HttpMessageConverter(customObjectMapper()));
    }
}
```

⚠️ **EXAM TRAP**: Implementing `WebMvcConfigurer` **adds to** Boot's MVC configuration — it doesn't replace it. `@EnableWebMvc` **replaces** Boot's MVC configuration entirely — it disables `WebMvcAutoConfiguration`. Almost never use `@EnableWebMvc` in a Boot application.

## Customising autoconfigured beans via `*Customizer` interfaces

Some autoconfigured beans support customisation via callback interfaces. Boot collects all customiser beans and applies them to the autoconfigured bean:

```java
// Customise embedded Tomcat without replacing the whole server configuration
@Bean
public WebServerFactoryCustomizer<TomcatServletWebServerFactory> tomcatCustomizer() {
    return factory -> {
        factory.setPort(9090);
        factory.addConnectorCustomizers(connector ->
            connector.setMaxPostSize(10 * 1024 * 1024)  // 10MB max POST
        );
    };
}
```

```java
// Customise Jackson ObjectMapper without replacing it
@Bean
public Jackson2ObjectMapperBuilderCustomizer jacksonCustomizer() {
    return builder -> builder
        .featuresToDisable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS)
        .modules(new JavaTimeModule());
}
```

This pattern is elegant — you get fine-grained control without declaring a full replacement bean. The customiser is applied on top of Boot's defaults.

## The `@ConditionalOnProperty` override pattern

You can gate your own configuration behind a property — letting operators enable/disable it at deployment time:

```java
@Configuration
@ConditionalOnProperty(name = "app.cache.provider", havingValue = "redis")
public class RedisCacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        return RedisCacheManager.builder(factory).build();
    }
}

@Configuration
@ConditionalOnProperty(
    name = "app.cache.provider",
    havingValue = "redis",
    matchIfMissing = true  // default to this if property not set
)
public class InMemoryCacheConfig {

    @Bean
    @ConditionalOnMissingBean
    public CacheManager cacheManager() {
        return new ConcurrentMapCacheManager();
    }
}
```

---

## Plain-English summary

- Three override mechanisms: declare your own bean, use properties, or exclude autoconfiguration entirely.
- Declaring your own bean works because autoconfigured beans use `@ConditionalOnMissingBean`.
- Properties are the least invasive — try these first for simple customisation.
- Exclude with `@SpringBootApplication(exclude = ...)` or `spring.autoconfigure.exclude` property. Requires fully qualified class names.
- `WebMvcConfigurer` adds to MVC config. `@EnableWebMvc` replaces it — avoid in Boot apps.
- `*Customizer` interfaces allow fine-grained customisation without full bean replacement.
