# Objective 6.2.1 — Defining and Loading Properties

## Property sources and the precedence order — HIGH VALUE

This is one of the most tested topics in Section 6. Spring Boot loads properties from many sources and combines them into a single `Environment`. When the same key appears in multiple sources, the **higher-precedence source wins**.

The full 17-level order from highest to lowest precedence:

```
1.  Command-line arguments                    (--server.port=9090)
2.  SPRING_APPLICATION_JSON (inline JSON)     (env var or system property)
3.  ServletConfig init parameters
4.  ServletContext init parameters
5.  JNDI attributes                           (java:comp/env)
6.  Java system properties                    (-Dserver.port=9090)
7.  OS environment variables                  (SERVER_PORT=9090)
8.  Profile-specific files outside JAR        (application-{profile}.properties)
9.  Profile-specific files inside JAR         (application-{profile}.properties)
10. Application properties outside JAR        (application.properties)
11. Application properties inside JAR         (application.properties)
12. @PropertySource annotations               (@PropertySource("classpath:custom.properties"))
13. Default properties                        (SpringApplication.setDefaultProperties())
```

⚠️ **EXAM TRAP**: Command-line arguments have the **highest** precedence — they beat everything including OS environment variables and profile-specific files. `application.properties` inside the JAR is near the bottom — it's the baseline that everything else overrides.

⚠️ **EXAM TRAP**: Profile-specific files (`application-prod.properties`) have **higher** precedence than the base `application.properties` — even when both are inside the JAR. The profile-specific file overrides the base file for any keys they share.

⚠️ **EXAM TRAP**: Properties files **outside** the JAR have higher precedence than the same file inside the JAR. Placing an `application.properties` in the same directory as the JAR overrides the bundled one — useful for production overrides without rebuilding.

---

## The key anchors to memorise

You don't need to memorise all 17 levels. Know these anchors:

```
HIGHEST:  Command-line args (--key=value)
          ↓
          OS environment variables
          ↓
          Profile-specific files (application-{profile}.properties)
          ↓
LOWEST:   application.properties inside JAR
          Default properties (setDefaultProperties)
```

---

## `application.properties` vs `application.yml`

Both are equivalent — Boot supports both formats. YAML is more readable for hierarchical config:

```properties
# application.properties
spring.datasource.url=jdbc:postgresql://localhost:5432/orders
spring.datasource.username=postgres
spring.datasource.password=secret
spring.jpa.hibernate.ddl-auto=update
server.port=8080
```

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/orders
    username: postgres
    password: secret
  jpa:
    hibernate:
      ddl-auto: update
server:
  port: 8080
```

⚠️ **EXAM TRAP**: You cannot use `@PropertySource` to load YAML files. `@PropertySource` only supports `.properties` files. This was covered in 1.3.1 and is a recurring exam question.

If you have both `application.properties` and `application.yml` in the same location, `application.properties` takes precedence — but having both is a code smell and should be avoided.

## Profile-specific property files

When a profile is active, Boot automatically loads `application-{profile}.properties` (or `.yml`) in addition to the base `application.properties`. Profile-specific values override base values for matching keys:

```
application.properties          ← always loaded
application-dev.properties      ← loaded when dev profile active
application-prod.properties     ← loaded when prod profile active
```

```properties
# application.properties (base — always loaded)
server.port=8080
spring.datasource.url=jdbc:h2:mem:default
app.feature.payments=false

# application-prod.properties (overrides for prod)
spring.datasource.url=jdbc:postgresql://prod-db:5432/orders
app.feature.payments=true
# server.port not overridden — stays 8080
```

When `prod` profile is active: `spring.datasource.url` = PostgreSQL URL, `server.port` = 8080 (from base), `app.feature.payments` = true.

## Command-line arguments

Passed as `--key=value` when running the JAR:

```bash
java -jar order-service.jar --server.port=9090 --spring.profiles.active=prod
```

These override everything. Useful for CI/CD pipelines and deployment scripts.

Programmatically disable command-line argument processing:
```java
SpringApplication app = new SpringApplication(OrderApplication.class);
app.setAddCommandLineProperties(false);  // ignore command-line args
app.run(args);
```

## OS environment variables

Spring Boot maps OS environment variables to property keys using relaxed binding — uppercase with underscores maps to dotted lowercase:

```bash
export SERVER_PORT=9090
export SPRING_DATASOURCE_URL=jdbc:postgresql://prod-db:5432/orders
export SPRING_PROFILES_ACTIVE=prod
```

These map to `server.port`, `spring.datasource.url`, and `spring.profiles.active` respectively.

⚠️ **EXAM TRAP**: OS environment variables are important for container deployments (Docker, Kubernetes) where you inject configuration via environment variables rather than property files. The relaxed binding makes this work automatically.

## `@ConfigurationProperties` with relaxed binding — recap

Covered in 1.3.1 but worth restating in the Boot context:

```java
@ConfigurationProperties(prefix = "payment")
public class PaymentProperties {
    private String apiUrl;      // matches payment.api-url, payment.api_url, PAYMENT_API_URL
    private int timeoutMs;      // matches payment.timeout-ms, payment.timeout_ms
    private boolean enabled;
}
```

Relaxed binding applies to `@ConfigurationProperties` only — not `@Value`. `@Value("${payment.apiUrl}")` requires exact key match.

## Property encryption and externalisation

For production secrets, properties should not be in `application.properties` committed to source control. Common patterns:

- OS environment variables (inject at deployment time)
- External config server (Spring Cloud Config — out of scope)
- Vault integration (out of scope)

The exam doesn't test these deeply — know that environment variables are the standard Boot-native way to inject sensitive config without putting it in files.

## `@Value` with property placeholders — recap

```java
@Value("${server.port}")                    // required — fails if missing
@Value("${server.port:8080}")               // default if missing
@Value("${app.name:My Application}")        // string default
@Value("${app.features.payments:false}")    // boolean default
```

The colon syntax provides a default value. Without a default, a missing property throws `IllegalArgumentException` at startup.

## Accessing the `Environment` directly

```java
@Component
public class ConfigReader {

    @Autowired
    private Environment environment;

    public void printConfig() {
        String port = environment.getProperty("server.port");
        String portWithDefault = environment.getProperty("server.port", "8080");
        boolean hasProperty = environment.containsProperty("app.feature.payments");
        String[] profiles = environment.getActiveProfiles();
    }
}
```

`Environment` gives programmatic access to all resolved properties across all property sources — respecting precedence order.

## Plain-English summary

- Command-line args win. `application.properties` inside JAR is near the bottom.
- Profile-specific files override base `application.properties` for matching keys.
- Files outside the JAR override files inside the JAR at the same level.
- `application.yml` and `application.properties` are equivalent — YAML is more readable for hierarchical config.
- `@PropertySource` loads `.properties` files only — not YAML.
- OS environment variables map to property keys via relaxed binding (`SERVER_PORT` → `server.port`).
- `@ConfigurationProperties` uses relaxed binding. `@Value` requires exact key match.
- `Environment` bean provides programmatic access to all resolved properties.
