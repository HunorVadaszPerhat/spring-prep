# Objective 6.2.2 — Autoconfiguration Mechanics

## What autoconfiguration does

When you add `spring-boot-starter-web` to your project, you don't declare a `DispatcherServlet`, a `Jackson ObjectMapper`, or a `TomcatServletWebServerFactory` bean. They just appear. This is autoconfiguration — Spring Boot detects what's on the classpath and what's already configured, then fills in sensible defaults for everything else.

The mechanism is built on two things: a registry of autoconfiguration classes, and the `@Conditional` annotation family that controls when each class applies.

---

## The autoconfiguration registry

Spring Boot needs to know which classes are autoconfiguration candidates. These are listed in a registration file — Boot reads this file at startup and evaluates each candidate.

**📝 EXAM NOTE (Boot 2.x):** `META-INF/spring.factories`

```
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration,\
  org.springframework.boot.autoconfigure.data.jpa.JpaRepositoriesAutoConfiguration,\
  org.springframework.boot.autoconfigure.jackson.JacksonAutoConfiguration
```

**✅ Modern (Boot 3.x):** `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`

```
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.data.jpa.JpaRepositoriesAutoConfiguration
org.springframework.boot.autoconfigure.jackson.JacksonAutoConfiguration
```

Same purpose, different file and format. Boot 3 moved away from `spring.factories` to a dedicated imports file.

⚠️ **EXAM TRAP**: The exam was written against Boot 2.5 — expect `spring.factories` in exam questions. Know both files exist and what they do, but answer `spring.factories` for exam-context questions.

## The `@Conditional` family

Each autoconfiguration class is guarded by one or more `@Conditional` annotations. Spring evaluates these conditions before deciding whether to apply the configuration. If any condition fails, the entire autoconfiguration class is skipped.

### `@ConditionalOnClass`

Applies only if the specified class is on the classpath:

```java
@Configuration
@ConditionalOnClass(DataSource.class)  // only if DataSource class is present
public class DataSourceAutoConfiguration {

    @Bean
    public DataSource dataSource() {
        return DataSourceBuilder.create().build();
    }
}
```

This is how Boot detects what you've added to the project. Add `spring-boot-starter-data-jpa` → Hibernate is on the classpath → `@ConditionalOnClass(LocalContainerEntityManagerFactoryBean.class)` passes → JPA autoconfiguration applies.

---

### `@ConditionalOnMissingBean` — HIGH VALUE

Applies only if no bean of the specified type already exists in the context. This is the cornerstone of Boot's "configure if you haven't already" philosophy.

```java
@Configuration
@ConditionalOnClass(ObjectMapper.class)
public class JacksonAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean  // only if no ObjectMapper bean exists yet
    public ObjectMapper objectMapper() {
        return new ObjectMapper();
    }
}
```

If you declare your own `ObjectMapper` bean, `@ConditionalOnMissingBean` fails — your bean exists, so Boot's default is skipped entirely. Your customisation wins.

⚠️ **EXAM TRAP**: `@ConditionalOnMissingBean` is how you **replace** an autoconfigured bean with your own. Declare a bean of the same type and Boot's default steps aside. This is the standard customisation pattern.

```java
// Your custom configuration — Boot's ObjectMapper auto-config is suppressed
@Configuration
public class JacksonConfig {

    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false)
            .setSerializationInclusion(JsonInclude.Include.NON_NULL);
    }
}
```

---

### `@ConditionalOnProperty`

Applies only if a specific property has a specific value:

```java
@Configuration
@ConditionalOnProperty(
    name = "app.cache.enabled",
    havingValue = "true",
    matchIfMissing = false  // default — skip if property not set
)
public class CacheAutoConfiguration {

    @Bean
    public CacheManager cacheManager() {
        return new ConcurrentMapCacheManager();
    }
}
```

`matchIfMissing = true` means the configuration applies even if the property isn't set — treating absence as the truthy case:

```java
@ConditionalOnProperty(
    name = "app.feature.payments",
    matchIfMissing = true  // applies unless explicitly set to false
)
```

⚠️ **EXAM TRAP**: `matchIfMissing` defaults to `false` — if the property is absent, the condition fails and the configuration is skipped. This trips people up when they expect a feature to be on by default.

---

### `@ConditionalOnWebApplication`

Applies only in a web application context:

```java
@Configuration
@ConditionalOnWebApplication(type = ConditionalOnWebApplication.Type.SERVLET)
public class WebMvcAutoConfiguration { }
```

Types: `SERVLET` (traditional Spring MVC), `REACTIVE` (WebFlux — out of scope), `ANY` (either).

---

### `@ConditionalOnBean`

The inverse of `@ConditionalOnMissingBean` — applies only if a bean of the specified type already exists:

```java
@Bean
@ConditionalOnBean(DataSource.class)  // only create if DataSource exists
public JdbcTemplate jdbcTemplate(DataSource dataSource) {
    return new JdbcTemplate(dataSource);
}
```

`JdbcTemplate` autoconfiguration only applies if a `DataSource` bean has already been created — which makes sense since `JdbcTemplate` depends on one.

---

### `@ConditionalOnMissingClass`

The inverse of `@ConditionalOnClass` — applies only if the specified class is NOT on the classpath:

```java
@Configuration
@ConditionalOnMissingClass("com.example.PremiumFeature")
public class BasicFeatureConfig { }
```

---

## How autoconfiguration is ordered

Autoconfiguration classes can declare ordering relative to each other:

```java
@AutoConfiguration(after = DataSourceAutoConfiguration.class)
public class JpaRepositoriesAutoConfiguration { }
```

`JpaRepositoriesAutoConfiguration` runs after `DataSourceAutoConfiguration` — it needs the `DataSource` bean to exist first.

📝 **EXAM NOTE**: In Boot 2.x this used `@AutoConfigureAfter` / `@AutoConfigureBefore`. Boot 3.x uses `@AutoConfiguration(after = ...)`. Know both exist.

## A complete autoconfiguration example

Here's what a real (simplified) autoconfiguration class looks like — brings together everything above:

```java
@AutoConfiguration
@ConditionalOnClass({DataSource.class, JdbcTemplate.class})  // JDBC on classpath?
@ConditionalOnSingleCandidate(DataSource.class)               // exactly one DataSource?
public class JdbcTemplateAutoConfiguration {

    @Bean
    @Primary
    @ConditionalOnMissingBean(JdbcOperations.class)  // no JdbcTemplate already?
    public JdbcTemplate jdbcTemplate(DataSource dataSource,
                                     JdbcProperties properties) {
        JdbcTemplate jdbcTemplate = new JdbcTemplate(dataSource);
        JdbcProperties.Template template = properties.getTemplate();
        jdbcTemplate.setFetchSize(template.getFetchSize());
        jdbcTemplate.setMaxRows(template.getMaxRows());
        return jdbcTemplate;
    }
}
```

Reading this: "If JDBC classes are on the classpath, there's exactly one `DataSource` bean, and no `JdbcOperations` bean exists yet — create a `JdbcTemplate`."

## Debugging autoconfiguration

Spring Boot Actuator exposes a `/actuator/conditions` endpoint (covered in 6.3) that shows which autoconfiguration classes were applied and which were skipped, with the reason for each decision. Invaluable for debugging why something isn't being configured.

Also available at startup with `--debug` flag:

```bash
java -jar order-service.jar --debug
```

Prints a conditions evaluation report to the console.

---

## Plain-English summary

- Boot reads autoconfiguration candidates from `spring.factories` (Boot 2.x) or `AutoConfiguration.imports` (Boot 3.x).
- `@ConditionalOnClass` — apply if class is on classpath. How Boot detects what you've added.
- `@ConditionalOnMissingBean` — apply if no bean of that type exists. How Boot backs off when you provide your own.
- `@ConditionalOnProperty` — apply if property has specific value. `matchIfMissing` defaults to false.
- `@ConditionalOnWebApplication` — apply only in web context.
- `@ConditionalOnBean` — apply only if a specific bean already exists.
- Override autoconfigured beans by declaring your own — `@ConditionalOnMissingBean` makes Boot's default step aside.
- `--debug` flag or `/actuator/conditions` shows which autoconfiguration applied and why.
