# Objective 1.2.1 — Define Spring Beans using Java Code

## <span style="background:yellow">What is a Bean?</span>

A **bean** is simply an object that Spring creates, configures, and manages. The IoC container owns its lifecycle — creation, wiring, destruction. Your job is to tell Spring *what* to create and *how* to create it.

---

## <span style="background:yellow">`@Configuration`</span> and <span style="background:yellow">`@Bean`</span>

`@Configuration` marks a class as a source of bean definitions. `@Bean` marks a method whose return value should be registered as a bean in the container.

```java
@Configuration
public class AppConfig {

    @Bean
    public PaymentGateway paymentGateway() {
        return new StripePaymentGateway();
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(paymentGateway()); // inter-bean method call
    }
}
```

**Bean name**: defaults to the method name. So `paymentGateway()` registers a bean named `"paymentGateway"`. You can override it: `@Bean("gateway")` or `@Bean(name = "gateway")`.

---

## <span style="background:yellow">The CGLIB proxy</span> — why inter-bean calls work correctly

This is one of the most important mechanics to understand for the exam.

When Spring sees `@Configuration`, it doesn't use your class directly. It creates a **CGLIB subclass** of it (a proxy). That proxy intercepts calls to `@Bean` methods. So when `orderService()` calls `paymentGateway()`, the proxy intercepts that call and returns the **already-created singleton** from the container — it does *not* invoke the method again and create a second instance.

```java
@Configuration
public class AppConfig {
    @Bean
    public PaymentGateway paymentGateway() {
        return new StripePaymentGateway(); // only called ONCE
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(paymentGateway()); // proxy returns existing singleton
    }

    @Bean
    public AuditService auditService() {
        return new AuditService(paymentGateway()); // same singleton — not a new instance
    }
}
```

⚠️ **EXAM TRAP**: If you use `@Configuration(proxyBeanMethods = false)` — or the lighter-weight `@Component` instead of `@Configuration` — the CGLIB proxy is *not* created. Inter-bean method calls become plain Java calls and **will create new instances each time**. The exam tests whether you know this distinction.

## `@Primary` and `@Qualifier`

When you have multiple beans of the same type, Spring doesn't know which one to inject. You have two tools to resolve this:

**`@Primary`** — marks one bean as the default choice when no other qualifier is specified.

```java
@Configuration
public class AppConfig {

    @Bean
    @Primary
    public PaymentGateway stripeGateway() {
        return new StripePaymentGateway();
    }

    @Bean
    public PaymentGateway paypalGateway() {
        return new PaypalPaymentGateway();
    }
}

@Service
public class OrderService {
    @Autowired
    private PaymentGateway paymentGateway; // gets stripeGateway — it's @Primary
}
```

**`@Qualifier`** — lets you name the exact bean you want at the injection point.

```java
@Service
public class OrderService {
    @Autowired
    @Qualifier("paypalGateway") // overrides @Primary, gets the paypal one
    private PaymentGateway paymentGateway;
}
```

⚠️ **EXAM TRAP**: `@Qualifier` at the injection point **overrides** `@Primary`. If both are in play, `@Qualifier` wins.

---

## Plain-English summary of what to hold onto

- `@Configuration` + `@Bean` = declare beans in Java code. Bean name = method name by default.
- `@Configuration` classes get CGLIB-proxied so inter-bean calls return the same singleton.
- `@Primary` = default when ambiguous. `@Qualifier` = explicit name, overrides `@Primary`.
- `@Component` / `proxyBeanMethods = false` disables the proxy — inter-bean calls create new instances.
