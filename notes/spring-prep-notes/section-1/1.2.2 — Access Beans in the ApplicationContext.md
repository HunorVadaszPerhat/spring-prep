# Objective 1.2.2 — Access Beans in the ApplicationContext

## The big picture

You already know Spring manages beans for you. But sometimes — in tests, in bootstrapping code, or in framework-level code — you need to reach into the container and pull a bean out directly. That's what `ApplicationContext.getBean()` is for.

The exam tests which overloads exist and what each one does. There are three.

---

## The three `getBean()` overloads

```java
@Configuration
public class AppConfig {
    @Bean
    public PaymentGateway paymentGateway() {
        return new StripePaymentGateway();
    }
}

// Bootstrapping the context (you'd normally let Boot do this)
ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);

// 1. By name only — returns Object, requires a cast
PaymentGateway gw = (PaymentGateway) ctx.getBean("paymentGateway");

// 2. By type only — no cast needed, throws if 0 or 2+ beans of that type exist
PaymentGateway gw = ctx.getBean(PaymentGateway.class);

// 3. By name AND type — no cast, also validates the type matches
PaymentGateway gw = ctx.getBean("paymentGateway", PaymentGateway.class);
```

What each one does and when it breaks:

**By name** — flexible, but you lose type safety and need a cast. If the name doesn't exist, throws `NoSuchBeanDefinitionException`.

**By type** — cleanest, no cast. Throws `NoSuchBeanDefinitionException` if none found. Throws `NoUniqueBeanDefinitionException` if more than one bean of that type exists — this is the common failure mode.

**By name + type** — safest option. Finds by name, then validates the type matches. Throws if name not found or type doesn't match.

⚠️ **EXAM TRAP**: `getBean(PaymentGateway.class)` with two `PaymentGateway` beans in the context throws `NoUniqueBeanDefinitionException` — even if one is `@Primary`. The `@Primary` preference applies during **injection**, not during explicit `getBean()` by type calls.

Actually — correction worth knowing: Spring *does* respect `@Primary` in `getBean(type)` calls in recent versions. But the exam-safe answer is: if you have ambiguity, use name+type or qualify properly. Don't rely on `@Primary` to save you in a `getBean()` call.

---

## Why you should rarely use `getBean()` in application code

Calling `getBean()` inside your business logic is **dependency lookup** — your class now has a direct dependency on the Spring container itself. This is exactly what IoC is supposed to eliminate.

```java
// BAD — your class knows about the container
@Service
public class OrderService {
    @Autowired
    private ApplicationContext ctx;

    public void process() {
        PaymentGateway gw = ctx.getBean(PaymentGateway.class); // lookup
    }
}

// GOOD — container pushes the dependency in
@Service
public class OrderService {
    private final PaymentGateway paymentGateway;

    public OrderService(PaymentGateway paymentGateway) { // injection
        this.paymentGateway = paymentGateway;
    }
}
```

The exam may present both patterns and ask which one correctly demonstrates dependency injection. The constructor version is the right answer — `getBean()` in application code is dependency lookup and breaks testability.

---

## Plain-English summary

- Three overloads: by name (needs cast), by type (no cast, fails on ambiguity), by name+type (safest).
- `NoSuchBeanDefinitionException` = bean not found. `NoUniqueBeanDefinitionException` = too many matches.
- Injection is always preferred over `getBean()` in application code. `getBean()` = dependency lookup = the anti-pattern Spring exists to replace.
