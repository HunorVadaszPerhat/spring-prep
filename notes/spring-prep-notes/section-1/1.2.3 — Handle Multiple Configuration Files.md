# Objective 1.2.3 — Handle Multiple Configuration Files

## Why you'd split configuration

In a real application, putting every bean definition in one `@Configuration` class gets unwieldy fast. Spring lets you split configuration across multiple classes and compose them together. The exam tests three mechanisms for doing this.

---

## `@Import` — composing `@Configuration` classes

`@Import` pulls another configuration class into the current one. Spring processes the imported class as if it had been discovered directly.

```java
@Configuration
public class DataConfig {
    @Bean
    public DataSource dataSource() {
        return new EmbeddedDatabaseBuilder().build();
    }
}

@Configuration
public class ServiceConfig {
    @Bean
    public OrderService orderService(DataSource dataSource) {
        return new OrderService(dataSource);
    }
}

@Configuration
@Import({DataConfig.class, ServiceConfig.class}) // pulls both in
public class AppConfig {
    // AppConfig is now the single entry point
}

// Bootstrap with just AppConfig — DataConfig and ServiceConfig are included
ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);
```

The imported classes don't need `@Configuration` on them to be imported — `@Import` works on any component class — but in practice you'll see it used with `@Configuration`.

⚠️ **EXAM TRAP**: `@Import` is a *pull* mechanism — you declare from the importing class what you want to bring in. It does not scan packages. That's `@ComponentScan`'s job.

---

## `@ImportResource` — mixing in XML config

The exam was written when XML config was still common. `@ImportResource` lets a Java config class include beans defined in an XML file.

```java
@Configuration
@ImportResource("classpath:legacy-beans.xml") // loads XML bean definitions
public class AppConfig {
    @Bean
    public OrderService orderService() {
        return new OrderService();
    }
}
```

📝 **EXAM NOTE**: You won't need to *write* XML config, but you need to recognise `@ImportResource` and know it bridges Java config and XML config.

---

## Multiple `@Configuration` classes registered directly

You can also just pass multiple configuration classes to the context constructor — no `@Import` needed. Spring merges them.

```java
ApplicationContext ctx = new AnnotationConfigApplicationContext(
    DataConfig.class,
    ServiceConfig.class,
    WebConfig.class
);
```

Spring builds one unified bean registry from all of them. Bean definitions across classes can reference each other freely — if `ServiceConfig` declares a bean that needs a `DataSource`, Spring will find the `DataSource` from `DataConfig` automatically via type-matching.

---

## How Spring merges multiple configuration sources

Regardless of which mechanism you use, the end result is the same: one flat bean registry. Spring doesn't keep configurations separate at runtime. All bean definitions from all sources are merged into the `ApplicationContext` together.

This means:
- A bean in `DataConfig` can be injected into a bean in `ServiceConfig` — Spring resolves it by type across the whole registry.
- If two configurations define a bean with the **same name**, the last one registered wins. This can cause subtle bugs when merging configs from multiple sources.

⚠️ **EXAM TRAP**: Bean name collisions across configuration files are resolved by **last-registered wins** — not an error by default. The exam may present a scenario where a bean definition is silently overridden.

---

## Plain-English summary

- `@Import` — explicitly pull in other `@Configuration` classes. Single entry point pattern.
- `@ImportResource` — bridge to XML config. Recognise it; don't need to write XML.
- Multiple configs passed directly to the context constructor — Spring merges them all.
- All configs merge into one flat registry at runtime. Name collisions = last one wins.
