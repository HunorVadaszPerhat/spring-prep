# Objective 1.4.3 — `@PostConstruct` and `@PreDestroy`

## What problem these solve

Sometimes a bean needs to do something after it's been fully constructed and all its dependencies injected — open a connection, validate config, warm up a cache. And sometimes it needs to clean up before the container shuts it down — close a connection, flush a buffer, release a resource.

You can't do this in the constructor because dependencies haven't been injected yet when the constructor runs. `@PostConstruct` and `@PreDestroy` solve this by hooking into the bean lifecycle at the right moments.

---

## `@PostConstruct`

Runs **after** the bean is instantiated and all dependencies are injected, but **before** the bean is handed to the rest of the application.

```java
@Component
public class DatabaseConnectionPool {

    @Autowired
    private DataSourceProperties properties;

    @PostConstruct
    public void init() {
        // Safe to use injected dependencies here
        System.out.println("Connecting to: " + properties.getUrl());
        // open connections, validate config, etc.
    }
}
```

⚠️ **EXAM TRAP**: `@PostConstruct` runs after injection is complete. If you try to use an injected dependency in the constructor, it may be null — the constructor runs before injection. Move that logic to `@PostConstruct`.

---

## `@PreDestroy`

Runs **before** the bean is removed from the container — during a graceful shutdown. Use it to release resources.

```java
@Component
public class DatabaseConnectionPool {

    @PreDestroy
    public void cleanup() {
        // Runs before the bean is destroyed
        // close connections, flush buffers, etc.
        System.out.println("Closing connection pool");
    }
}
```

⚠️ **EXAM TRAP**: `@PreDestroy` is **not called on prototype beans**. Spring creates prototypes and hands them off — it doesn't track them afterward, so it never calls their destroy callbacks. This is the same point from 1.2.5. The exam tests it here too.

---

## Package location — the JSR-250 detail

`@PostConstruct` and `@PreDestroy` are JSR-250 annotations, not Spring-specific.

- 📝 **EXAM NOTE (Boot 2.5 / Java EE)**: `javax.annotation.PostConstruct` / `javax.annotation.PreDestroy`
- ✅ **Modern (Boot 3.x / Jakarta EE)**: `jakarta.annotation.PostConstruct` / `jakarta.annotation.PreDestroy`

The exam was written against Boot 2.5, so expect `javax.annotation` in exam code. Don't be thrown off — it's the same annotation, different package.

---

## The alternatives Spring also supports

Spring provides two other mechanisms that do the same job. The exam expects you to recognise all three and know they're equivalent:

**`InitializingBean` / `DisposableBean` interfaces** — Spring-specific interfaces:

```java
@Component
public class CacheManager implements InitializingBean, DisposableBean {

    @Override
    public void afterPropertiesSet() {
        // same as @PostConstruct
    }

    @Override
    public void destroy() {
        // same as @PreDestroy
    }
}
```

**`@Bean` init/destroy attributes** — for beans you define in `@Configuration` classes:

```java
@Bean(initMethod = "init", destroyMethod = "cleanup")
public DatabaseConnectionPool connectionPool() {
    return new DatabaseConnectionPool();
}
```

This is the approach you use for third-party classes — you can't add `@PostConstruct` to a class you don't own, but you can point to its existing init/cleanup methods.

---

## Execution order when multiple mechanisms are combined

If a bean uses more than one mechanism, they fire in this order:

**Initialisation:**
1. `@PostConstruct`
2. `InitializingBean.afterPropertiesSet()`
3. `@Bean(initMethod = "...")`

**Destruction:**
1. `@PreDestroy`
2. `DisposableBean.destroy()`
3. `@Bean(destroyMethod = "...")`

⚠️ **EXAM TRAP**: The exam may ask which fires first when multiple mechanisms are present. `@PostConstruct` always wins for initialisation. `@PreDestroy` always wins for destruction.

---

## Plain-English summary

- `@PostConstruct` — runs after injection is complete. Safe to use dependencies here. Constructor is too early.
- `@PreDestroy` — runs before the bean is destroyed. Use for resource cleanup.
- Neither is called on prototype beans — Spring doesn't track prototype lifecycle after creation.
- Three equivalent mechanisms: `@PostConstruct`/`@PreDestroy`, `InitializingBean`/`DisposableBean`, `@Bean(initMethod/destroyMethod)`.
- For third-party classes, use `@Bean(initMethod = "...", destroyMethod = "...")`.
- Order when combined: `@PostConstruct` → `afterPropertiesSet()` → `initMethod`.
