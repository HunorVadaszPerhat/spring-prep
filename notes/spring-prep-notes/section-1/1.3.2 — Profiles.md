# Objective 1.3.2 — Profiles

## What profiles are for

Different environments need different beans or different property values. You don't want your production database bean active during tests, and you don't want your mock payment gateway active in production. Profiles let you tag beans and configuration with an environment name, then activate only the right ones at runtime.

---

## `@Profile` on beans

Tag a bean or entire `@Configuration` class with `@Profile` to make it conditional on which profiles are active.

```java
@Configuration
public class DataSourceConfig {

    @Bean
    @Profile("dev")
    public DataSource devDataSource() {
        return new EmbeddedDatabaseBuilder().build(); // H2 in-memory for dev
    }

    @Bean
    @Profile("prod")
    public DataSource prodDataSource() {
        DataSourceBuilder<?> builder = DataSourceBuilder.create();
        builder.url("jdbc:postgresql://prod-db:5432/orders");
        return builder.build();
    }
}
```

Only the bean matching the active profile is registered. The other one doesn't exist in the container at all — it's not just skipped at injection time, it's never created.

You can also put `@Profile` on an entire `@Configuration` class:

```java
@Configuration
@Profile("dev")
public class DevOnlyConfig {
    // all beans in here only exist when dev profile is active
}
```

---

## Profile expressions

`@Profile` accepts more than a simple name:

```java
@Profile("!prod")          // active when prod is NOT active
@Profile({"dev", "test"})  // active when dev OR test is active
@Profile("dev & cloud")    // active when BOTH dev AND cloud are active (Spring 5.1+)
```

⚠️ **EXAM TRAP**: The `!` (not) operator is commonly tested. `@Profile("!prod")` means the bean is registered in every environment *except* production.

---

## Activating profiles

**In `application.properties`:**
```properties
spring.profiles.active=dev
```

**As a JVM argument:**
```
-Dspring.profiles.active=prod
```

**As an environment variable:**
```
SPRING_PROFILES_ACTIVE=prod
```

Multiple profiles can be active at once:
```properties
spring.profiles.active=dev,cloud
```

---

## Profile-specific property files

Spring Boot automatically loads `application-{profile}.properties` (or `.yml`) when a profile is active. This is separate from `@Profile` on beans — it's purely about properties.

```
application.properties          ← always loaded (base config)
application-dev.properties      ← loaded when dev profile is active
application-prod.properties     ← loaded when prod profile is active
```

Profile-specific files **override** values in the base `application.properties`. So if both files define `server.port`, the profile-specific file wins.

⚠️ **EXAM TRAP**: The base `application.properties` is always loaded regardless of which profiles are active. Profile-specific files add to or override it — they don't replace it entirely.

---

## `@ActiveProfiles` in tests

In tests you activate profiles with `@ActiveProfiles` rather than a property:

```java
@SpringBootTest
@ActiveProfiles("test")
class OrderServiceIntegrationTest {
    // spring.profiles.active=test for this test class
}
```

You can activate multiple profiles:
```java
@ActiveProfiles({"test", "cloud"})
```

This is the exam-standard way to set profiles in tests. Don't set `spring.profiles.active` in `application.properties` for test purposes — use `@ActiveProfiles`.

---

## Default profile

If no profile is explicitly activated, Spring uses the `default` profile. Beans tagged with `@Profile("default")` are active only when nothing else is. You can change the default profile name:

```properties
spring.profiles.default=dev
```

---

## Plain-English summary

- `@Profile("name")` on a bean or `@Configuration` — that bean only exists when the named profile is active.
- `!` negates, `{a, b}` means OR, `a & b` means AND.
- Activate profiles via `spring.profiles.active` property, JVM arg, or env var.
- `application-{profile}.properties` is auto-loaded by Boot and overrides base `application.properties` values.
- Base `application.properties` is always loaded — profile files layer on top.
- In tests, use `@ActiveProfiles` — not a property file.
