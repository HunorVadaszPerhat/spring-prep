# Objective 1.1 ‚Äî Why Spring Exists: IoC, DI, and the Container

## Why Spring exists

Before Spring, Java enterprise apps (J2EE) required you to manually wire together every dependency. If `OrderService` needed a `PaymentGateway`, you'd write `new PaymentGateway()` inside `OrderService`. That creates **tight coupling** ‚Äî `OrderService` now controls the creation and lifecycle of its own dependencies, which makes it nearly impossible to swap implementations or test in isolation.

Spring's core answer to this is <span style="background:yellow">**Inversion of Control (IoC)**</span>: instead of your code creating its dependencies, you hand that responsibility to a framework ‚Äî the **IoC container**. Your class just declares what it needs, and the container figures out how to provide it.

The practical payoffs are:
- <span style="background:yellow">**Decoupling**</span> ‚Äî `OrderService` depends on an interface, not a concrete class. Swap the implementation without touching `OrderService`.
- <span style="background:yellow">**Testability**</span> ‚Äî inject a mock `PaymentGateway` in tests instead of the real one.
- <span style="background:yellow">**Enterprise integration**</span> ‚Äî transactions, security, caching, etc. can be layered on without polluting business logic.

---

## IoC vs Dependency Injection vs Dependency Lookup

<span style="background:yellow">IoC is the *principle*.</span> There are two ways to implement it:

<span style="background:yellow">**Dependency Injection (DI)**</span> ‚Äî the container *pushes* dependencies into your class. Your class is passive; it just receives what it needs. This is what Spring uses.

<span style="background:yellow">**Dependency Lookup**</span> ‚Äî your class *actively asks* the container for a dependency (e.g. `context.getBean(...)`). Your class is aware of the container. This breaks testability and is considered an anti-pattern for application code.

‚öÝÔ∏è <span style="background:yellow">**EXAM TRAP**</span>: Calling `ApplicationContext.getBean()` inside your business logic is dependency *lookup*, not injection ‚Äî and it's the thing Spring is trying to help you *avoid*. The exam may ask you to distinguish these.

## BeanFactory vs ApplicationContext

Both are IoC containers. `ApplicationContext` extends `BeanFactory` and adds production-grade features.

| Feature | BeanFactory | ApplicationContext                                    |
|---|---|-------------------------------------------------------|
| Basic DI | <span style="background:yellow">‚úÖ</span> | <span style="background:yellow">‚úÖ</span>              |
| Eager singleton init | <span style="background:yellow">‚ùå (lazy by default)</span> | <span style="background:yellow">‚úÖ (at startup)</span> |
| Event publishing | <span style="background:yellow">‚ùå</span> | <span style="background:yellow">‚úÖ</span>              |
| `MessageSource` (i18n) | <span style="background:yellow">‚ùå</span> | <span style="background:yellow">‚úÖ</span>              |
| `BeanPostProcessor` auto-registration | <span style="background:yellow">Manual</span> | <span style="background:yellow">‚úÖ Automatic</span>             |
| AOP integration | <span style="background:yellow">Limited</span> | <span style="background:yellow">‚úÖ</span>              |

In practice: **always use `ApplicationContext`**. `BeanFactory` exists mostly for constrained environments (embedded devices, etc.) where memory is critical. The exam may ask which one supports events or i18n ‚Äî the answer is always `ApplicationContext`.

## The three DI types

#### <span style="background:yellow">1. Constructor injection ‚Äî PREFERRED</span>
```java

@Service
public class OrderService {
    private final PaymentGateway paymentGateway;

    public OrderService(PaymentGateway paymentGateway) {  // Spring injects here
        this.paymentGateway = paymentGateway;
    }
}
```

#### <span style="background:yellow">2. Setter injection ‚Äî use when dependency is optional</span>
```java
@Service
public class OrderService {
    private PaymentGateway paymentGateway;

    @Autowired
    public void setPaymentGateway(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }
}
```

#### <span style="background:yellow">3. Field injection ‚Äî convenient but problematic</span>
```java
@Service
public class OrderService {
    @Autowired
    private PaymentGateway paymentGateway;  // Can't make final, hard to Inversion of Control
}
```

<span style="background:yellow">**Why constructor injection is preferred:**</span>
- The field can be `final` ‚Äî guarantees the dependency is never null after construction.
- Dependencies are explicit in the constructor signature ‚Äî easy to see what a class needs.
- Works with plain `new` in unit tests ‚Äî no Spring context required.
- With a single constructor in Spring 4.3+, `@Autowired` is optional ‚Äî Spring infers it.

Field injection hides dependencies and makes the class untestable without a Spring context or a reflection-based injector.
