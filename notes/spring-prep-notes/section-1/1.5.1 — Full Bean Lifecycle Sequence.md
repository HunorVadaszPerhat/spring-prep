# Objective 1.5.1 — Full Bean Lifecycle Sequence

## Why this matters

The exam asks questions like "at what point can a bean use its injected dependencies?" or "which callback fires before `@PostConstruct`?" — you need the sequence cold, not just the individual steps.

---

## The full sequence

Here it is in order, for a singleton bean from container startup to shutdown:

**Creation phase:**
1. **Instantiation** — Spring calls the constructor. Dependencies are NOT injected yet.
2. **Populate properties** — Spring injects dependencies (`@Autowired`, setter injection, `@Value`).
3. **`BeanNameAware.setBeanName()`** — Spring passes the bean's name to it (if implemented).
4. **`BeanFactoryAware.setBeanFactory()`** — Spring passes the `BeanFactory` reference (if implemented).
5. **`ApplicationContextAware.setApplicationContext()`** — Spring passes the `ApplicationContext` (if implemented).
6. **`BeanPostProcessor.postProcessBeforeInitialization()`** — runs for ALL beans. Fires before init callbacks.
7. **`@PostConstruct`** — bean's own init callback.
8. **`InitializingBean.afterPropertiesSet()`** — alternative init callback.
9. **`@Bean(initMethod = "...")`** — third init callback option.
10. **`BeanPostProcessor.postProcessAfterInitialization()`** — runs for ALL beans. This is where AOP proxies are created.

**Bean is now ready and in use.**

**Destruction phase (on container shutdown):**
11. **`@PreDestroy`** — bean's own destroy callback.
12. **`DisposableBean.destroy()`** — alternative destroy callback.
13. **`@Bean(destroyMethod = "...")`** — third destroy callback option.

## The sequence as code

```java
@Component
public class LifecycleBean implements BeanNameAware, ApplicationContextAware, 
                                      InitializingBean, DisposableBean {

    public LifecycleBean() {
        System.out.println("1. Constructor — no dependencies yet");
    }

    @Autowired
    public void setDependency(SomeDependency dep) {
        System.out.println("2. Dependency injected");
    }

    @Override
    public void setBeanName(String name) {
        System.out.println("3. BeanNameAware — my name is: " + name);
    }

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        System.out.println("5. ApplicationContextAware");
    }

    @PostConstruct
    public void postConstruct() {
        System.out.println("7. @PostConstruct");
    }

    @Override
    public void afterPropertiesSet() {
        System.out.println("8. InitializingBean.afterPropertiesSet()");
    }

    @PreDestroy
    public void preDestroy() {
        System.out.println("11. @PreDestroy");
    }

    @Override
    public void destroy() {
        System.out.println("12. DisposableBean.destroy()");
    }
}
```

---

## The exam-critical points in the sequence

**Step 1 vs Step 2**: The constructor runs before injection. This is why you can't use `@Autowired` fields in a constructor — they're null. Use `@PostConstruct` for post-injection logic.

**Steps 3–5 — the Aware interfaces**: These let a bean get a reference to container infrastructure. Rarely used in application code but the exam tests that you know they exist and when they fire — before `BeanPostProcessor`, before init callbacks.

⚠️ **EXAM TRAP**: The Aware callbacks fire **before** `BeanPostProcessor.postProcessBeforeInitialization()`. The naming is confusing — "before initialisation" sounds early, but the Aware interfaces are even earlier.

**Step 6 vs Step 10 — `BeanPostProcessor`**: Two hooks, wrapping the init callbacks. `postProcessBeforeInitialization` fires before `@PostConstruct`. `postProcessAfterInitialization` fires after all init callbacks — this is the step where Spring creates AOP proxies. The bean you get from the container after step 10 may be a proxy, not the original instance.

⚠️ **EXAM TRAP**: AOP proxies are created in `postProcessAfterInitialization` — step 10, the last step before the bean is ready. This is why AOP only works on beans retrieved from the container, not on objects you create with `new`.

**Steps 7–9 — init callback order**: If a bean uses multiple init mechanisms, they fire in this order: `@PostConstruct` → `afterPropertiesSet()` → `initMethod`. Same point from 1.4.3, now in the full context.

---

## Prototype beans — what's different

Prototype beans go through steps 1–10 (creation) every time they're requested. They do **not** go through steps 11–13 (destruction). Spring creates them and forgets about them. No `@PreDestroy`, no `DisposableBean.destroy()`.

---

## Plain-English summary

The sequence in plain English: construct → inject → aware callbacks → BeanPostProcessor before → init callbacks → BeanPostProcessor after → ready → (use) → destroy callbacks.

Key anchors to memorise:
- Constructor = no dependencies yet
- Aware interfaces = before BeanPostProcessor
- `@PostConstruct` = first init callback, after injection
- AOP proxy creation = `postProcessAfterInitialization` (step 10)
- Prototype beans = full creation, no destruction
- Destruction order mirrors init: `@PreDestroy` → `afterPropertiesSet` equivalent → `destroyMethod`
