# Objective 1.5.3 — Spring Proxies: JDK Dynamic vs CGLIB

## Why Spring uses proxies at all

Spring's core features — AOP, `@Transactional`, `@Cacheable`, scoped beans — all work by **wrapping your bean in a proxy**. When you call a method on a Spring-managed bean, you're often calling it on a proxy that intercepts the call, does some work (starts a transaction, checks the cache, applies advice), then delegates to your actual object.

This is transparent in normal use. The proxy and the real object look identical from the outside — they implement the same interface or extend the same class. But the mechanics matter because they determine what Spring *can and can't* intercept.

---

## The two proxy types

### JDK Dynamic Proxy

Creates a proxy that **implements the same interface(s)** as the target bean. The proxy is a separate object that implements the interface — it has no knowledge of the concrete class.

```java
// Your interface
public interface PaymentGateway {
    void processPayment(Order order);
}

// Your implementation
@Service
public class StripePaymentGateway implements PaymentGateway {
    public void processPayment(Order order) { ... }
}

// What Spring creates (conceptually):
// A JDK proxy that implements PaymentGateway
// and delegates to StripePaymentGateway
```

**Requirement**: The target bean must implement at least one interface. The proxy implements that interface.

**Limitation**: You can only inject the proxy as the interface type, not as the concrete class. If you try to inject `StripePaymentGateway` directly, Spring can't assign a `PaymentGateway` proxy to it.

---

### CGLIB Proxy

Creates a proxy by **subclassing the concrete class**. The proxy extends your class and overrides its methods to add interceptor logic.

```java
// Your class — no interface needed
@Service
public class OrderService {
    public void processOrder(Order order) { ... }
}

// What Spring creates (conceptually):
// A CGLIB subclass of OrderService
// that overrides processOrder() to add interceptor logic
```

**Requirement**: The class must not be `final`, and the methods to be proxied must not be `final`. You can't subclass or override final things.

**Advantage**: Works on any class — no interface required.

---

## Which one does Spring choose?

Spring decides based on whether the bean implements an interface:

- Bean **implements an interface** → JDK dynamic proxy by default
- Bean **does NOT implement an interface** → CGLIB (no choice)
- You can **force CGLIB** even when interfaces exist: `@EnableAspectJAutoProxy(proxyTargetClass = true)`

Spring Boot defaults to CGLIB for everything since Boot 2.0 — it sets `proxyTargetClass = true` globally. This avoids the "injecting by concrete type" footgun.

📝 **EXAM NOTE**: The exam was written against Boot 2.5 where CGLIB is the default. The question is more likely to test *when JDK proxy is used* (interface exists, `proxyTargetClass = false`) as an edge case.

---

## `@Configuration` always uses CGLIB

As covered in 1.2.1: `@Configuration` classes are always CGLIB-proxied regardless of any other settings. This is what makes inter-bean method calls return singletons. This proxy is separate from the AOP proxy mechanism — it's applied during context startup specifically to `@Configuration` classes.

---

## The limitations that flow from proxy mechanics

Both proxy types share the same fundamental constraint: **the proxy only intercepts calls that go through the proxy object**. This produces two important failure modes:

### Self-invocation bypasses the proxy

```java
@Service
public class OrderService {

    @Transactional
    public void processOrder(Order order) {
        validateOrder(order);   // calls this.validateOrder — bypasses proxy
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void validateOrder(Order order) {
        // @Transactional here has NO EFFECT when called from processOrder()
        // The call never goes through the proxy
    }
}
```

When `processOrder()` calls `validateOrder()`, it's calling `this.validateOrder()` — a direct call on the real object, not through the proxy. The `@Transactional` on `validateOrder()` is silently ignored.

⚠️ **EXAM TRAP**: Self-invocation is one of the top exam traps. Any time a method in the same class calls another method in the same class, proxy-based features (`@Transactional`, AOP advice, `@Cacheable`) on the called method are bypassed entirely.

### Private and final methods can't be proxied

- **JDK proxy**: can only intercept methods declared on the interface. Private methods aren't on the interface.
- **CGLIB proxy**: subclasses and overrides methods. Can't override `private` or `final` methods.

```java
@Service
public class OrderService {

    @Transactional
    private void processPayment() {
        // @Transactional silently ignored — proxy can't intercept private methods
    }
}
```

⚠️ **EXAM TRAP**: `@Transactional` on a private method has no effect. No error is thrown — it silently does nothing. This is in the Quick Reference at the top of your study prompt for good reason.

---

## Plain-English summary

- JDK proxy = implements the same interface. Requires an interface. Can't inject by concrete type.
- CGLIB proxy = subclasses the concrete class. Works without interfaces. Can't proxy final classes or methods.
- Spring Boot defaults to CGLIB. Force CGLIB explicitly with `proxyTargetClass = true`.
- `@Configuration` always uses CGLIB — this is what makes singleton inter-bean calls work.
- Self-invocation bypasses the proxy entirely — `@Transactional`, AOP advice, `@Cacheable` all silently fail.
- Private and final methods can't be proxied — annotations on them are silently ignored.
