# Objective 1.2.5 — Bean Scopes

## The five scopes

Spring beans have a **scope** that controls how many instances are created and how long they live.

| Scope | Instances | Lifetime |
|---|---|---|
| `singleton` | One per container | Entire application lifetime |
| `prototype` | New one every time requested | Until GC — Spring doesn't manage destruction |
| `request` | One per HTTP request | Duration of that request |
| `session` | One per HTTP session | Duration of that session |
| `application` | One per ServletContext | Entire app lifetime (similar to singleton) |

`singleton` is the default. If you don't specify a scope, you get a singleton.

`request`, `session`, and `application` are only valid in a web-aware Spring context. Using them outside of a web context throws an error.

---

## Declaring a scope

```java
@Configuration
public class AppConfig {

    @Bean
    @Scope("prototype")
    public ShoppingCart shoppingCart() {
        return new ShoppingCart();
    }
}

// Or with the constant (preferred — avoids typos)
@Bean
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public ShoppingCart shoppingCart() {
    return new ShoppingCart();
}
```

With `@Component`-style beans:

```java
@Component
@Scope("prototype")
public class ShoppingCart { }
```

---

## ⚠️ HIGH VALUE: The prototype-into-singleton trap

This is one of the most tested topics in Section 1. Understand the mechanics, not just the fix.

**The problem**: A singleton bean is created once. If it holds a reference to a prototype bean, that reference is set once at creation time and never updated. The prototype bean is effectively frozen as a singleton — you never get a fresh instance.

```java
@Configuration
public class AppConfig {

    @Bean
    @Scope("prototype")
    public ShoppingCart shoppingCart() {
        return new ShoppingCart(); // intended: new instance each time
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(shoppingCart()); // BUG: shoppingCart() called once
    }                                            // orderService holds one cart forever
}
```

Even with the CGLIB proxy in a `@Configuration` class, this still misfires — the proxy returns the singleton `OrderService` which was already constructed with one `ShoppingCart` instance baked in.

---

## Fix 1: `ScopedProxyMode.TARGET_CLASS`

Tell Spring to inject a **scoped proxy** instead of the real instance. The proxy is singleton-safe to hold, but every method call on it delegates to a fresh prototype instance behind the scenes.

```java
@Bean
@Scope(value = "prototype", proxyMode = ScopedProxyMode.TARGET_CLASS)
public ShoppingCart shoppingCart() {
    return new ShoppingCart();
}

@Bean
public OrderService orderService(ShoppingCart shoppingCart) {
    // shoppingCart is a proxy — each call through it gets a fresh instance
    return new OrderService(shoppingCart);
}
```

`ScopedProxyMode.TARGET_CLASS` uses CGLIB to subclass the bean. Use `ScopedProxyMode.INTERFACES` if the bean implements an interface and you want a JDK proxy instead.

---

## Fix 2: `ObjectFactory` or `Provider`

Instead of injecting the bean directly, inject a **factory** that creates a new instance on demand. The singleton holds the factory, and calls the factory each time it needs a fresh prototype.

```java
@Service
public class OrderService {

    private final ObjectFactory<ShoppingCart> cartFactory;

    public OrderService(ObjectFactory<ShoppingCart> cartFactory) {
        this.cartFactory = cartFactory;
    }

    public void processOrder() {
        ShoppingCart cart = cartFactory.getObject(); // fresh prototype each call
    }
}
```

`ObjectFactory` is Spring-specific. The JSR-330 equivalent is `javax.inject.Provider<T>` (or `jakarta.inject.Provider<T>` in Boot 3.x) — both work the same way.

⚠️ **EXAM TRAP**: The question will describe a singleton that has a stale prototype dependency and ask what's wrong or how to fix it. The two valid fixes are scoped proxy and `ObjectFactory`/`Provider`. Simply changing the scope annotation on the prototype bean without one of these fixes does nothing — the singleton still holds the original instance.

---

## Prototype destruction

One subtle difference from singleton: **Spring does not call `@PreDestroy` on prototype beans**. It creates them and then hands them off — lifecycle management after that point is your responsibility. Singleton beans get full lifecycle management including destruction callbacks.

---

## Plain-English summary

- Singleton = one instance, lives forever. Prototype = new instance every request, no destruction callback.
- Web scopes (request, session, application) only work in a web context.
- Injecting a prototype into a singleton = the prototype is created once and frozen. This is the trap.
- Fix with `ScopedProxyMode.TARGET_CLASS` (proxy delegates to fresh instance on each call) or `ObjectFactory`/`Provider` (pull a fresh instance when you need it).
- `@PreDestroy` is not called on prototype beans.
