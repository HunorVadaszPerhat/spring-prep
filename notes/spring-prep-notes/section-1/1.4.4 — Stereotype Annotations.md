# Objective 1.4.4 — Stereotype Annotations

## What stereotypes are

`@Component` is the generic marker that tells Spring "manage this class as a bean." The stereotype annotations — `@Service`, `@Repository`, `@Controller` — are all just `@Component` with a more specific name. They're **specialisations**, not replacements.

From a pure bean registration standpoint, all four do the same thing. The differences are semantic and, in one important case, functional.

---

## The four stereotypes

```java
@Component        // generic — use when no more specific stereotype fits
public class UtilityHelper { }

@Service          // business logic layer
public class OrderService { }

@Repository       // data access layer
public class OrderRepository { }

@Controller       // Spring MVC web layer — returns view names
public class OrderController { }

@RestController   // @Controller + @ResponseBody — returns data, not views
public class OrderApiController { }
```

`@RestController` is a composed annotation — it's `@Controller` + `@ResponseBody` combined. The exam tests this distinction in Section 3 but it's worth noting here.

---

## The one functional difference: `@Repository` and exception translation

`@Repository` does something the others don't — it activates **exception translation**. Spring wraps persistence-layer exceptions (checked `SQLException`, Hibernate exceptions, JPA exceptions) into Spring's unified `DataAccessException` hierarchy.

```java
@Repository
public class OrderRepository {

    @Autowired
    private JdbcTemplate jdbcTemplate;

    public Order findById(long id) {
        // If this throws a SQLException, Spring translates it
        // into a DataAccessException (unchecked) automatically
        return jdbcTemplate.queryForObject(...);
    }
}
```

Without `@Repository`, that translation doesn't happen and the raw persistence exception propagates up.

⚠️ **EXAM TRAP**: Exception translation is the **only functional difference** between `@Repository` and plain `@Component`. `@Service` and `@Controller` add no extra behaviour — they're purely semantic markers. The exam tests whether you know which stereotype activates exception translation.

---

## Why semantics matter even when behaviour is the same

Even though `@Service` and `@Component` behave identically at runtime, using the right stereotype:

- Makes the code readable — a new developer immediately knows what layer a class belongs to
- Enables tooling — IDEs and frameworks can treat layers differently
- Supports `@ComponentScan` filtering — you can exclude or include specific stereotypes:

```java
@ComponentScan(
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ANNOTATION,
        classes = Repository.class  // exclude all @Repository classes (e.g. in unit tests)
    )
)
```

This is a real pattern — excluding `@Repository` classes from the context in tests that use `@DataJpaTest` slices.

---

## How stereotypes are built — meta-annotations

Each stereotype is annotated with `@Component` itself. This is Spring's **meta-annotation** mechanism — an annotation annotated with another annotation inherits its behaviour.

```java
// From Spring's source — simplified
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Component   // ← @Service IS a @Component via meta-annotation
public @interface Service { }
```

This means `@ComponentScan` finds `@Service`, `@Repository`, and `@Controller` classes automatically — it looks for anything meta-annotated with `@Component`, not just `@Component` itself.

⚠️ **EXAM TRAP**: `@ComponentScan` discovers all stereotype annotations, not just `@Component`. A class annotated with `@Service` is picked up by scanning without needing an explicit `@Component`.

---

## Plain-English summary

- `@Service`, `@Repository`, `@Controller` are all specialisations of `@Component`. All register beans.
- `@Repository` is special — it activates exception translation (SQL/persistence exceptions → `DataAccessException`).
- `@Service` and `@Controller` are semantic only — no extra runtime behaviour.
- `@RestController` = `@Controller` + `@ResponseBody` combined.
- `@ComponentScan` finds all stereotype annotations via meta-annotation, not just `@Component`.
- Stereotypes enable filtering — you can include/exclude by annotation type in `@ComponentScan`.
