# Objective 1.5.4 — Bean Creation Order: `@DependsOn` and `@Lazy`

## How Spring normally determines creation order

Spring builds a dependency graph from your bean definitions and creates beans in the order that satisfies dependencies. If `OrderService` depends on `PaymentGateway`, Spring creates `PaymentGateway` first automatically — you don't need to do anything.

The problems arise when there's a dependency Spring **can't see** — a side-effect dependency, an initialisation order requirement that isn't expressed through injection, or a resource that must exist before a bean initialises.

---

## `@DependsOn`

Forces Spring to create one bean before another, even when there's no injection relationship between them.

```java
@Configuration
public class AppConfig {

    @Bean
    @DependsOn("databaseInitializer") // create databaseInitializer first
    public OrderRepository orderRepository() {
        return new OrderRepository();
    }

    @Bean
    public DatabaseInitializer databaseInitializer() {
        return new DatabaseInitializer(); // creates schema, inserts seed data
    }
}
```

In this example, `OrderRepository` doesn't inject `DatabaseInitializer` — it just needs the database schema to exist before it runs. `@DependsOn` expresses that hidden ordering requirement.

Multiple dependencies:
```java
@Bean
@DependsOn({"databaseInitializer", "cacheWarmer"})
public OrderRepository orderRepository() {
    return new OrderRepository();
}
```

Works on `@Component`-annotated classes too:
```java
@Component
@DependsOn("auditLogger")
public class SecurityService { }
```

⚠️ **EXAM TRAP**: `@DependsOn` controls **creation order only** — it does not inject the named bean. If you need the actual `DatabaseInitializer` instance, you still need `@Autowired`. `@DependsOn` just guarantees it exists and has been initialised before your bean is created.

---

## Circular dependencies

If bean A depends on bean B and bean B depends on bean A, Spring throws `BeanCurrentlyInCreationException` at startup — it can't resolve the circular reference with constructor injection.

```java
// This will fail at startup
@Component
public class A {
    public A(B b) { }  // A needs B
}

@Component
public class B {
    public B(A a) { }  // B needs A — circular!
}
```

The fix is to break the cycle — refactor to remove the circular dependency (the right answer), or use setter injection on one side (allows Spring to create both beans first, then inject):

```java
@Component
public class A {
    @Autowired
    public void setB(B b) { }  // setter injection breaks the constructor cycle
}
```

📝 **EXAM NOTE**: Spring Boot 2.6+ detects circular dependencies at startup and fails fast by default. Earlier versions (including the exam's Boot 2.5 baseline) allowed them in some cases with setter injection. Know that circular constructor injection always fails.

---

## `@Lazy`

By default, singleton beans are created **eagerly** — all at once during context startup. `@Lazy` defers creation until the bean is first requested.

```java
@Component
@Lazy
public class HeavyReportGenerator {
    // Expensive to initialise — don't create until actually needed
    public HeavyReportGenerator() {
        // imagine expensive setup here
    }
}
```

The bean is created the first time something requests it — either via injection or `getBean()`.

---

## `@Lazy` on injection points

You can also put `@Lazy` on an injection point rather than the bean definition. This injects a lazy proxy instead of the real bean — the real bean is created only when a method on the proxy is first called.

```java
@Service
public class ReportService {

    @Autowired
    @Lazy  // don't create HeavyReportGenerator until a method is called on it
    private HeavyReportGenerator reportGenerator;
}
```

This is useful when you want most of your context to start eagerly but need to defer one expensive bean.

---

## `@Lazy` on `@Configuration` classes

Putting `@Lazy` on a `@Configuration` class makes all `@Bean` methods within it lazy by default:

```java
@Configuration
@Lazy  // all beans in this class are lazy unless overridden
public class OptionalFeaturesConfig {

    @Bean
    public FeatureXService featureX() { ... } // lazy

    @Bean
    @Lazy(false)  // override — this one is eager
    public FeatureYService featureY() { ... }
}
```

---

## The trade-off with `@Lazy`

Eager initialisation (the default) fails fast — misconfigured beans cause startup errors you see immediately. `@Lazy` pushes that failure to first use, which could be in production under load.

⚠️ **EXAM TRAP**: `@Lazy` does not change the bean's scope — a lazy singleton is still a singleton. It's created once, just not until first requested. The exam may try to conflate laziness with prototype scope — they're unrelated.

---

## Plain-English summary

- Spring normally determines creation order from the dependency graph automatically.
- `@DependsOn` forces ordering when there's a hidden dependency Spring can't see. Controls order only — doesn't inject.
- Circular constructor injection always fails. Setter injection can break the cycle.
- `@Lazy` defers singleton creation until first use. Default is eager (all at startup).
- `@Lazy` on an injection point injects a proxy — real bean created on first method call.
- Lazy singletons are still singletons — created once, just later. Not the same as prototype.
