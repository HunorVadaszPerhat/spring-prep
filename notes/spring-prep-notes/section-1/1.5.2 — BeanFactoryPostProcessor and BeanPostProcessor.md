# Objective 1.5.2 — `BeanFactoryPostProcessor` and `BeanPostProcessor`

## The conceptual distinction first

These two interfaces sound similar but operate at completely different points and on completely different things. Getting them confused is one of the most common mistakes on this section.

- **`BeanFactoryPostProcessor`** — operates on **bean definitions** (the metadata/recipe), before any beans are instantiated.
- **`BeanPostProcessor`** — operates on **bean instances** (actual objects), after each bean is created.

Think of it this way: `BeanFactoryPostProcessor` edits the blueprint before construction starts. `BeanPostProcessor` inspects or wraps the finished object after it's built.

---

## `BeanFactoryPostProcessor`

Fires after the container has read all bean definitions but **before it instantiates any beans**. It receives the `ConfigurableListableBeanFactory`, which gives it access to all registered `BeanDefinition` objects.

```java
@Component
public class CustomBeanFactoryPostProcessor implements BeanFactoryPostProcessor {

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory factory) {
        // All bean definitions are loaded — no beans created yet
        BeanDefinition bd = factory.getBeanDefinition("orderService");
        
        // Can modify the definition — e.g. change scope, property values
        bd.setScope(BeanDefinition.SCOPE_PROTOTYPE);
        
        // Can also read metadata about what's registered
        System.out.println("Bean count: " + factory.getBeanDefinitionCount());
    }
}
```

**What it can do:**
- Modify bean definitions before instantiation (change scope, add property values)
- Register new bean definitions
- Read metadata about what's in the container

**What it cannot do:**
- Instantiate beans itself — doing so causes them to miss later lifecycle steps

⚠️ **EXAM TRAP**: `BeanFactoryPostProcessor` must be instantiated **before** all other beans so it can process their definitions. Spring handles this automatically, but it means a `BeanFactoryPostProcessor` cannot have dependencies on regular beans injected into it — those beans haven't been created yet.

---

## The most important built-in `BeanFactoryPostProcessor`

You've been using one all along without knowing it:

**`PropertySourcesPlaceholderConfigurer`** — this is what resolves `${...}` placeholders in `@Value` annotations. It runs as a `BeanFactoryPostProcessor`, reads property sources, and replaces placeholder strings in bean definitions with their actual values — before beans are instantiated.

Spring Boot auto-configures this for you. In plain Spring, you'd register it manually:

```java
@Bean
public static PropertySourcesPlaceholderConfigurer propertyConfigurer() {
    return new PropertySourcesPlaceholderConfigurer();
}
```

📝 **EXAM NOTE**: Notice the `static` keyword. `BeanFactoryPostProcessor` beans must be declared as `static @Bean` methods in `@Configuration` classes. This prevents the `@Configuration` class itself from being fully instantiated just to create the post-processor — which would defeat the purpose of having it run early.

---

## `BeanPostProcessor`

Fires **after each individual bean is instantiated and injected** — once before init callbacks, once after. It wraps or inspects individual bean instances.

```java
@Component
public class LoggingBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        // Fires before @PostConstruct / afterPropertiesSet
        System.out.println("About to init: " + beanName);
        return bean; // must return the bean (or a replacement)
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // Fires after all init callbacks
        // This is where AOP proxies are created
        System.out.println("Finished init: " + beanName);
        return bean; // can return a proxy instead of the original
    }
}
```

**What it can do:**
- Inspect, modify, or wrap any bean instance
- Return a completely different object (a proxy) instead of the original bean
- This return-a-proxy capability is exactly how Spring AOP and `@Transactional` work

⚠️ **EXAM TRAP**: Both methods **must return the bean**. If you return `null`, Spring will throw a `NullPointerException` during context startup. The return value replaces the bean in the container — return the original bean or an intentional proxy.

## Side by side comparison

| | `BeanFactoryPostProcessor` | `BeanPostProcessor` |
|---|---|---|
| Operates on | Bean **definitions** (metadata) | Bean **instances** (objects) |
| Fires | After definitions loaded, before instantiation | After each bean is instantiated |
| Receives | `ConfigurableListableBeanFactory` | Individual bean instance + name |
| Fires once or per bean | Once total | Once per bean (×2 — before and after init) |
| Can create proxies | No | Yes |
| Example built-in | `PropertySourcesPlaceholderConfigurer` | `AutowiredAnnotationBeanPostProcessor` |

`AutowiredAnnotationBeanPostProcessor` is the built-in `BeanPostProcessor` that processes `@Autowired` annotations — it's what actually performs dependency injection.

---

## Plain-English summary

- `BeanFactoryPostProcessor` = edits blueprints before construction. Runs once. Can't depend on regular beans.
- `BeanPostProcessor` = inspects/wraps finished objects. Runs per bean, twice (before and after init callbacks).
- `PropertySourcesPlaceholderConfigurer` is a `BeanFactoryPostProcessor` — resolves `${...}` before beans are created.
- `BeanPostProcessor` methods must return the bean. Returning null breaks the context.
- AOP proxies are created in `postProcessAfterInitialization` — this is how `@Transactional` and AOP advice get wired in.
- `static @Bean` is required when declaring a `BeanFactoryPostProcessor` in a `@Configuration` class.
