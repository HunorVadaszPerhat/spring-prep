# Objective 1.3.1 — External Properties

## Why external properties exist

Hardcoding values like database URLs, port numbers, or API keys inside your classes means recompiling to change them. External properties let you separate configuration from code — the same JAR runs in dev, staging, and production with different values injected at runtime.

---

## `@Value`

The most direct way to inject a single property value into a bean.

```java
@Component
public class PaymentConfig {

    @Value("${payment.api.url}")           // reads from property sources
    private String apiUrl;

    @Value("${payment.timeout:5000}")      // default value after the colon
    private int timeoutMs;

    @Value("${payment.enabled:true}")
    private boolean enabled;
}
```

The `${...}` syntax is **property placeholder** resolution — Spring looks up the key in its property sources and substitutes the value. If the key isn't found and there's no default, Spring throws `IllegalArgumentException` at startup.

⚠️ **EXAM TRAP**: `${...}` is property placeholder syntax. `#{...}` is SpEL (Spring Expression Language) — covered in 1.3.3. They look similar but do different things. The exam will test whether you can tell them apart.

---

## `@ConfigurationProperties`

For groups of related properties, `@Value` on individual fields gets noisy. `@ConfigurationProperties` binds a whole prefix of properties to a class at once.

```java
@ConfigurationProperties(prefix = "payment")
public class PaymentProperties {
    private String apiUrl;
    private int timeout;
    private boolean enabled;

    // getters and setters required for binding
}
```

With `application.properties`:
```properties
payment.api-url=https://api.stripe.com
payment.timeout=5000
payment.enabled=true
```

To activate it, either annotate the class with `@Component`, or register it explicitly:

```java
@Configuration
@EnableConfigurationProperties(PaymentProperties.class)
public class AppConfig { }
```

Then inject it normally:
```java
@Service
public class PaymentService {
    private final PaymentProperties props;

    public PaymentService(PaymentProperties props) {
        this.paymentProperties = props;
    }
}
```

---

## Relaxed binding

`@ConfigurationProperties` supports **relaxed binding** — the property key format in the file doesn't have to exactly match the field name. Spring normalises both sides before matching.

| Field name | Accepted property key formats |
|---|---|
| `apiUrl` | `api-url`, `api_url`, `APIURL`, `API_URL` |
| `timeoutMs` | `timeout-ms`, `timeout_ms`, `TIMEOUT_MS` |

⚠️ **EXAM TRAP**: Relaxed binding applies to `@ConfigurationProperties` only — **not** to `@Value`. With `@Value("${payment.apiUrl}")`, the key must match exactly. The exam tests this distinction.

---

## Property precedence (Spring Core level)

Spring resolves properties from multiple sources in a defined order. Higher in the list wins:

1. JVM system properties (`-Dpayment.timeout=3000`)
2. OS environment variables (`PAYMENT_TIMEOUT=3000`)
3. `@PropertySource` annotations on `@Configuration` classes
4. `application.properties` / `application.yml`

The full 17-level Boot-specific precedence order (command-line args at the top) is covered in **6.2.1** — that's the one the exam tests in detail. For now, the key principle is: **more specific / runtime sources beat compile-time file-based sources**.

---

## `@PropertySource`

Loads a properties file into the `Environment` so `@Value` can read from it.

```java
@Configuration
@PropertySource("classpath:payment.properties")
public class AppConfig { }
```

📝 **EXAM NOTE**: `@PropertySource` does not support YAML files out of the box. It only handles `.properties` files. This is a known exam question.

---

## Plain-English summary

- `@Value("${key}")` — inject a single property. Add `:default` for a fallback. Exact key match required.
- `@ConfigurationProperties(prefix = "...")` — bind a group of properties to a class. Supports relaxed binding. Needs getters/setters.
- Relaxed binding = camelCase, kebab-case, underscore, UPPER_CASE all map to the same field. `@ConfigurationProperties` only — not `@Value`.
- `@PropertySource` loads `.properties` files. Does not support YAML.
- Full property precedence detail is in 6.2.1 — that's where the exam goes deep on it.
