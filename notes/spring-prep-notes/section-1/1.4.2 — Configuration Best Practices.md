# Objective 1.4.2 — Configuration Best Practices

## The core question

You now know two ways to define beans: `@Bean` methods in a `@Configuration` class, and `@Component` with scanning. The exam expects you to know when each approach is appropriate — not just that both exist.

---

## `@Component` (and stereotypes) — use when you own the class

When you're writing the class yourself, annotate it directly. This is the most common case in application code.

```java
@Service
public class OrderService {
    // You wrote this class — annotate it directly
}

@Repository
public class OrderRepository {
    // You wrote this class — annotate it directly
}
```

Spring finds these via component scanning. No `@Configuration` class needed. The annotation lives right next to the code it describes.

---

## `@Bean` — use when you don't own the class

When you need to register a third-party class as a bean, you can't add `@Component` to it — you don't control its source. `@Bean` in a `@Configuration` class is the answer.

```java
@Configuration
public class InfrastructureConfig {

    @Bean
    public ObjectMapper objectMapper() {
        // ObjectMapper is from Jackson — you can't annotate it
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
    }

    @Bean
    public RestTemplate restTemplate() {
        // RestTemplate is from Spring — also can't annotate it
        return new RestTemplate();
    }
}
```

⚠️ **EXAM TRAP**: This is the primary distinction the exam tests. If the question describes registering a third-party or framework class as a bean, `@Bean` in a `@Configuration` class is the correct answer. `@Component` scanning only works on classes you control.

---

## `@Bean` — also use when construction is complex

Even for classes you own, if creating the bean requires conditional logic, external resources, or multi-step construction, `@Bean` gives you full Java control:

```java
@Bean
public DataSource dataSource(@Value("${db.url}") String url,
                             @Value("${db.pool.size:10}") int poolSize) {
    HikariConfig config = new HikariConfig();
    config.setJdbcUrl(url);
    config.setMaximumPoolSize(poolSize);
    config.addDataSourceProperty("cachePrepStmts", "true");
    return new HikariDataSource(config);
}
```

You can't express this kind of construction logic with `@Component` alone.

---

## Java config vs XML config

The exam was written when XML config was still relevant. The practical rule is:

- **Java config** (`@Configuration` + `@Bean`) — preferred for new code. Type-safe, refactorable, IDE-friendly.
- **XML config** — legacy. Recognise it on the exam, know `@ImportResource` bridges it into Java config. Don't write it.

📝 **EXAM NOTE**: If a question shows XML bean definitions, read it carefully — it's testing whether you can interpret the older syntax. Don't try to modernise it in your head. Answer based on what the XML says.

---

## Annotation config vs explicit wiring — the practical trade-off

| | `@Component` + scanning | `@Bean` explicit |
|---|---|---|
| Verbosity | Low — just annotate | Higher — need a config class |
| Works on third-party classes | ❌ | ✅ |
| Complex construction logic | ❌ | ✅ |
| Visibility of wiring | Implicit | Explicit |
| Testability | Good | Good |

Neither is universally better. Real applications use both — `@Component` for your own classes, `@Bean` for infrastructure and third-party beans.

---

## One config class or many?

The exam may ask about organising `@Configuration` classes. The practical answer:

- Split by concern: `DataConfig`, `SecurityConfig`, `WebConfig`, `ServiceConfig`
- Compose with `@Import` or let Boot's component scanning find them all
- Avoid one massive config class — it becomes hard to navigate and harder to test slices of

There's no single right answer the exam enforces here, but if asked what the best practice is, **split by concern and use `@Import` or scanning to compose** is the safe answer.

---

## Plain-English summary

- `@Component` (and stereotypes) — use on classes you own. Requires component scanning to work.
- `@Bean` — use for third-party classes you can't annotate, or when construction logic is complex.
- The exam's primary test: third-party class = `@Bean`. Own class = either, but `@Component` is simpler.
- XML config = legacy. Recognise `@ImportResource` as the bridge. Don't write XML.
- Split `@Configuration` classes by concern in real applications.
