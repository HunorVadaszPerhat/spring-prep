# Objective 1.6.2 — Implementing Aspects: `@Aspect` and `@EnableAspectJAutoProxy`

## Enabling AOP in Spring

Spring doesn't scan for aspects by default. You need to switch it on explicitly.

```java
@Configuration
@EnableAspectJAutoProxy  // activates Spring's AOP proxy creation mechanism
public class AppConfig { }
```

With Spring Boot, `@EnableAspectJAutoProxy` is auto-configured when `spring-boot-starter-aop` is on the classpath. You don't need to declare it manually in a Boot application — but you need to know it exists and what it does for the exam.

`@EnableAspectJAutoProxy` registers a `BeanPostProcessor` that detects `@Aspect` classes and creates proxies for any beans they target. This is the weaving step from 1.6.1 — it happens during context startup via `postProcessAfterInitialization`.

## Declaring an aspect

```java
@Component  // must be a Spring bean so the container finds it
@Aspect     // marks this as an aspect — Spring's AOP infrastructure inspects it
public class LoggingAspect {

    // Pointcut + advice combined in one method
    @Before("execution(* com.example.service.*.*(..))")
    public void logBefore(JoinPoint joinPoint) {
        System.out.println("Calling: " + joinPoint.getSignature().getName());
    }
}
```

Two annotations are required on the class:
- `@Aspect` — tells Spring's AOP infrastructure to inspect this class for pointcut and advice declarations
- `@Component` (or any stereotype) — registers it as a Spring bean so it gets picked up

⚠️ **EXAM TRAP**: `@Aspect` alone is not enough. Without `@Component` (or an explicit `@Bean` declaration), Spring won't find the aspect at all. The class must be in the container as a bean *and* be marked as an aspect.

## `JoinPoint` — accessing method context in advice

Most advice types receive a `JoinPoint` parameter that gives you information about the intercepted method call:

```java
@Before("execution(* com.example.service.*.*(..))")
public void logBefore(JoinPoint joinPoint) {
    String methodName = joinPoint.getSignature().getName();
    Object[] args = joinPoint.getArgs();
    Object target = joinPoint.getTarget(); // the real object, not the proxy

    System.out.println("Method: " + methodName + ", args: " + Arrays.toString(args));
}
```

`@Around` advice uses `ProceedingJoinPoint` instead — a subtype of `JoinPoint` that adds the `proceed()` method. Covered fully in 1.6.4.

## The proxy limitations — complete picture

These all flow from the single root cause covered in 1.5.3 and 1.6.1: Spring AOP uses runtime proxies.

### Self-invocation

```java
@Service
public class OrderService {

    public void processOrder(Order order) {
        validateOrder(order); // direct this.validateOrder() — bypasses proxy
    }

    @Transactional // silently ignored when called from processOrder()
    public void validateOrder(Order order) { ... }
}
```

The call to `validateOrder()` is `this.validateOrder()` — it goes directly to the real object, not through the proxy. Any advice targeting `validateOrder()` is never triggered.

⚠️ **EXAM TRAP**: Self-invocation is the single most tested AOP limitation. If the exam shows a method calling another method in the same class, and asks whether advice/`@Transactional` applies — the answer is no.

**Workarounds** (know these exist, not deep detail):
- Inject the bean into itself: `@Autowired private OrderService self;` then call `self.validateOrder()`
- Restructure to move the method to a different bean
- Use full AspectJ with load-time weaving (out of scope for exam)

### Private methods

```java
@Service
public class OrderService {

    @Transactional
    private void processPayment() { // proxy can't override private methods
        // @Transactional silently ignored
    }
}
```

CGLIB subclasses the class and overrides methods. Private methods can't be overridden — the proxy simply can't intercept them. No error, no warning — silently ignored.

### Final classes and methods

```java
@Service
public final class OrderService { // CGLIB can't subclass final classes
    // No proxy created — AOP won't work at all
}
```

```java
@Service
public class OrderService {

    @Transactional
    public final void processOrder() { // CGLIB can't override final methods
        // @Transactional silently ignored
    }
}
```

### Static methods

Static methods don't belong to instances — there's no object to proxy. Advice targeting static methods is silently ignored.

## `proxyTargetClass` — forcing CGLIB

```java
@EnableAspectJAutoProxy(proxyTargetClass = true)
```

Forces CGLIB proxying even when the bean implements an interface. Spring Boot sets this to `true` by default. The exam may ask what `proxyTargetClass = true` does — it forces CGLIB subclassing instead of JDK interface proxying.

---

## Summary of what Spring AOP cannot intercept

| Scenario | Can Spring AOP intercept? |
|---|---|
| Public method on a Spring bean called from outside | ✅ |
| Method called from within the same class (self-invocation) | ❌ |
| Private method | ❌ |
| Static method | ❌ |
| Method on a final class | ❌ |
| Final method | ❌ |
| Method on an object created with `new` | ❌ |

---

## Plain-English summary

- `@EnableAspectJAutoProxy` activates AOP. Auto-configured by Boot with `spring-boot-starter-aop`.
- `@Aspect` + `@Component` both required on an aspect class. `@Aspect` alone isn't enough.
- `JoinPoint` gives access to method name, args, and target inside advice methods.
- Self-invocation bypasses the proxy — the most tested limitation.
- Private, static, and final methods/classes cannot be proxied — annotations on them are silently ignored.
- `proxyTargetClass = true` forces CGLIB. Spring Boot default.
