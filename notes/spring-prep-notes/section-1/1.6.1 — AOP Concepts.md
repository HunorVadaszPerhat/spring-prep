# Objective 1.6.1 — AOP Concepts

## What problem AOP solves

Some concerns cut across many classes — logging, transaction management, security checks, performance monitoring. Without AOP, you end up copy-pasting the same boilerplate into every method that needs it. Change the logging format and you're editing fifty classes.

AOP lets you define that cross-cutting logic **once** and declare where it applies. The business logic stays clean; the cross-cutting concern is applied separately.

---

## The core vocabulary

The exam expects you to match definitions to terms precisely. Here they are with concrete examples.

**Join point** — a specific point in program execution where advice *could* be applied. In Spring AOP, join points are always **method executions**. Every method call on a Spring-managed bean is a potential join point.

**Pointcut** — an expression that selects which join points to actually target. It's the filter. "Apply advice to all methods in the `com.example.service` package" is a pointcut.

**Advice** — the code that runs at the selected join points. The what. "Log the method name and arguments before execution" is advice.

**Aspect** — the module that combines a pointcut and advice together. The complete definition of a cross-cutting concern. One aspect class can contain multiple pointcuts and multiple advice methods.

**Weaving** — the process of applying aspects to target objects. Creates the proxied version of your bean with the advice baked in.

**Target object** — the real object being proxied. The thing your advice is wrapping around.

**AOP proxy** — the object that Spring actually gives you instead of the target. Looks like the target from the outside, intercepts calls, applies advice, then delegates to the target.

---

## How they relate — a concrete example

```
Pointcut:  "all methods annotated with @Transactional"
Advice:    "before the method runs, open a transaction;
            after it returns, commit;
            if it throws, rollback"
Aspect:    the class that declares both the pointcut and the advice
Join point: each specific method call that matches the pointcut at runtime
Weaving:   Spring creating a proxy of your service bean that applies this advice
```

This is exactly how `@Transactional` works under the hood — it's an AOP aspect built into Spring.

## Spring AOP vs AspectJ — the scope difference

⚠️ **EXAM TRAP**: Spring AOP and full AspectJ are different things. The exam tests Spring AOP specifically.

| | Spring AOP | Full AspectJ |
|---|---|---|
| Weaving mechanism | Runtime proxies | Bytecode weaving (compile-time or load-time) |
| Join point types | Method execution only | Method, constructor, field access, etc. |
| Works on | Spring-managed beans only | Any Java object |
| Intercepts private methods | ❌ | ✅ |
| Intercepts static methods | ❌ | ✅ |
| Intercepts final classes | ❌ | ✅ |

Spring uses AspectJ's **annotation syntax** (`@Aspect`, `@Before`, pointcut expressions) but implements the behaviour through runtime proxies — not bytecode weaving. This is an important distinction. You get the familiar AspectJ annotations but with Spring's proxy-based limitations.

---

## Weaving timing in Spring

Spring AOP weaves at **runtime** — specifically during context startup, in `BeanPostProcessor.postProcessAfterInitialization()` (the step you saw in 1.5.1). By the time your application is running and starts calling methods, the proxies are already in place.

This is different from compile-time weaving (AspectJ modifies bytecode during compilation) and load-time weaving (AspectJ modifies bytecode as classes are loaded by the JVM). Spring does neither — it creates proxy objects at runtime.

---

## Why method-execution only matters

Because Spring AOP uses proxies, it can only intercept calls that **go through the proxy**. This means:

- Only method calls are join points — field access doesn't go through the proxy
- Only calls from **outside** the bean go through the proxy — internal `this.method()` calls bypass it
- Only Spring-managed beans can be advised — objects created with `new` don't have proxies

These limitations all flow from the same root cause: proxy-based interception.

---

## Plain-English summary

- **Join point** = a moment in execution where advice could apply. In Spring: always a method call.
- **Pointcut** = the expression that selects which join points to target.
- **Advice** = the code that runs at matched join points.
- **Aspect** = the class combining pointcut(s) and advice.
- **Weaving** = applying aspects to create proxied beans. Spring does this at runtime.
- Spring AOP uses AspectJ annotation syntax but runtime proxies — not bytecode weaving.
- Spring AOP: method execution only, Spring beans only, no private/static/final interception.
- Self-invocation bypasses the proxy — advice on internally called methods is silently skipped.
