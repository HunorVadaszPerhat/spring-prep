# Objective 1.2.4 — Handle Dependencies Between Beans

## The core question

When one bean needs another bean to function, how does Spring wire them together inside a `@Configuration` class? There are three patterns the exam tests.

---

## Pattern 1: Inter-bean method calls

The most direct approach — one `@Bean` method calls another `@Bean` method directly.

```java
@Configuration
public class AppConfig {

    @Bean
    public PaymentGateway paymentGateway() {
        return new StripePaymentGateway();
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(paymentGateway()); // direct method call
    }
}
```

This works *because* of the CGLIB proxy covered in 1.2.1. The call to `paymentGateway()` is intercepted by the proxy, which returns the existing singleton rather than creating a new one.

This only works inside a `@Configuration` class. If the class is `@Component` or has `proxyBeanMethods = false`, the proxy doesn't exist and you get a fresh instance on every call.

---

## Pattern 2: `@Bean` method parameters

Instead of calling another `@Bean` method directly, declare the dependency as a **parameter** on your `@Bean` method. Spring resolves it by type from the container and passes it in.

```java
@Configuration
public class AppConfig {

    @Bean
    public PaymentGateway paymentGateway() {
        return new StripePaymentGateway();
    }

    @Bean
    public OrderService orderService(PaymentGateway paymentGateway) {
        // Spring sees the parameter, finds a PaymentGateway bean, injects it
        return new OrderService(paymentGateway);
    }
}
```

This pattern is **safer and more flexible** than inter-bean method calls because:
- It works even when configs are split across multiple classes — Spring resolves across the whole registry.
- It doesn't rely on the CGLIB proxy mechanics.
- The dependency is explicit in the method signature.

⚠️ **EXAM TRAP**: Parameters on `@Bean` methods are resolved by type, exactly like `@Autowired`. If there are two beans of the same type, Spring will throw `NoUniqueBeanDefinitionException` unless you add `@Qualifier` on the parameter.

```java
@Bean
public OrderService orderService(@Qualifier("stripeGateway") PaymentGateway paymentGateway) {
    return new OrderService(paymentGateway);
}
```

---

## Pattern 3: `@Autowired` on `@Bean` method parameters (explicit annotation)

You can add `@Autowired` explicitly to a `@Bean` method parameter — but you don't need to. Spring autowires `@Bean` method parameters automatically. The annotation is optional and changes nothing about the behaviour.

```java
@Bean
public OrderService orderService(@Autowired PaymentGateway paymentGateway) {
    // @Autowired here is redundant — Spring does this anyway
    return new OrderService(paymentGateway);
}
```

The exam may show this and ask whether it's valid. It is — it just isn't necessary.

---

## Cross-configuration dependencies

The parameter pattern is how you handle dependencies when beans are defined in **different** `@Configuration` classes:

```java
@Configuration
public class DataConfig {
    @Bean
    public DataSource dataSource() {
        return new EmbeddedDatabaseBuilder().build();
    }
}

@Configuration
public class ServiceConfig {
    @Bean
    public OrderService orderService(DataSource dataSource) {
        // DataSource comes from DataConfig — Spring finds it across the whole registry
        return new OrderService(dataSource);
    }
}
```

You cannot use the inter-bean method call pattern here — `ServiceConfig` can't call `DataConfig`'s methods. The parameter pattern is the correct approach for cross-configuration wiring.

---

## Plain-English summary

- **Inter-bean method call** — works inside one `@Configuration` class. Relies on CGLIB proxy returning the singleton. Doesn't work across separate config classes.
- **`@Bean` method parameter** — Spring resolves by type from the whole registry. Works across config classes. Preferred approach.
- **`@Autowired` on parameters** — valid but redundant. Spring autowires `@Bean` parameters automatically.
- Cross-config dependencies always use the parameter pattern — you can't call methods across config classes.
