# Objective 3.2.1 — HTTP Verbs, `@Valid`, `@RestControllerAdvice`, `@ExceptionHandler`

## HTTP verb annotations — complete picture

Covered partially in 3.1.3 — here's the full set with the semantics the exam expects you to know:

```java
@RestController
@RequestMapping("/orders")
public class OrderController {

    @GetMapping("/{id}")        // retrieve — safe, idempotent
    public ResponseEntity<Order> findById(@PathVariable Long id) { ... }

    @PostMapping                // create — neither safe nor idempotent
    public ResponseEntity<Order> create(@RequestBody @Valid Order order) { ... }

    @PutMapping("/{id}")        // full replace — idempotent
    public ResponseEntity<Order> replace(
            @PathVariable Long id,
            @RequestBody @Valid Order order) { ... }

    @PatchMapping("/{id}")      // partial update — not necessarily idempotent
    public ResponseEntity<Order> update(
            @PathVariable Long id,
            @RequestBody Map<String, Object> updates) { ... }

    @DeleteMapping("/{id}")     // delete — idempotent
    public ResponseEntity<Void> delete(@PathVariable Long id) { ... }
}
```

**Idempotent** means calling the operation multiple times produces the same result as calling it once. `PUT` and `DELETE` are idempotent — deleting something twice has the same end state as deleting it once. `POST` is not — creating an order twice creates two orders.

## `@Valid` — bean validation

`@Valid` triggers JSR-303/380 bean validation on the annotated parameter. Validation annotations on the entity are evaluated before the method body executes.

```java
public class Order {

    @NotNull(message = "Customer name is required")
    @Size(min = 2, max = 100)
    private String customerName;

    @NotNull
    @DecimalMin(value = "0.01", message = "Total must be positive")
    private BigDecimal total;

    @NotNull
    @Future(message = "Delivery date must be in the future")
    private LocalDate deliveryDate;
}
```

```java
@PostMapping
public ResponseEntity<Order> create(@RequestBody @Valid Order order) {
    // if validation fails, MethodArgumentNotValidException is thrown
    // before this method body executes
    Order saved = orderService.save(order);
    return ResponseEntity.created(URI.create("/orders/" + saved.getId())).body(saved);
}
```

⚠️ **EXAM TRAP**: When `@Valid` validation fails, Spring throws `MethodArgumentNotValidException` — which produces a 400 Bad Request by default. This exception is handled by `DefaultHandlerExceptionResolver` unless you provide your own `@ExceptionHandler`.

`@Validated` is Spring's variant of `@Valid` — adds support for validation groups. The exam primarily tests `@Valid`.

## `@ExceptionHandler` — handling exceptions in a controller

`@ExceptionHandler` on a method in a `@Controller` or `@RestController` handles exceptions thrown by handler methods in **that same controller**:

```java
@RestController
@RequestMapping("/orders")
public class OrderController {

    @GetMapping("/{id}")
    public Order findById(@PathVariable Long id) {
        return orderService.findById(id)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }

    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(OrderNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse(ex.getMessage()));
    }
}
```

The limitation: `@ExceptionHandler` inside a controller only handles exceptions from **that controller**. For global exception handling across all controllers, use `@RestControllerAdvice`.

---

## `@RestControllerAdvice` — global exception handling

`@RestControllerAdvice` is a composed annotation:

```java
@RestControllerAdvice = @ControllerAdvice + @ResponseBody
```

`@ControllerAdvice` makes the class apply to all controllers. `@ResponseBody` means the `@ExceptionHandler` methods write to the response body (not a view name) — same as `@RestController`.

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(OrderNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)  // sets status code
    public ErrorResponse handleNotFound(OrderNotFoundException ex) {
        return new ErrorResponse("NOT_FOUND", ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidationFailure(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .toList();
        return new ErrorResponse("VALIDATION_FAILED", errors.toString());
    }

    @ExceptionHandler(Exception.class)  // catch-all
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleGeneric(Exception ex) {
        return new ErrorResponse("INTERNAL_ERROR", "An unexpected error occurred");
    }
}
```

⚠️ **EXAM TRAP**: `@RestControllerAdvice` applies globally — to all controllers. `@ExceptionHandler` inside a controller applies only to that controller. If the same exception type has handlers in both places, the **controller-local handler wins** for that controller.

---

## Exception handler resolution order

When an exception is thrown, Spring searches for an `@ExceptionHandler` in this order:

1. `@ExceptionHandler` methods in the **same controller** — most specific match wins
2. `@ExceptionHandler` methods in `@RestControllerAdvice` classes — most specific match wins
3. Built-in `HandlerExceptionResolver` implementations (handle framework exceptions like 404, 405)

Within each level, Spring prefers the **most specific exception type**. An `@ExceptionHandler(OrderNotFoundException.class)` beats `@ExceptionHandler(RuntimeException.class)` when an `OrderNotFoundException` is thrown.

## `@ResponseStatus` on exception classes

An alternative to `@ExceptionHandler` — annotate the exception class itself with `@ResponseStatus`:

```java
@ResponseStatus(HttpStatus.NOT_FOUND)  // any throw of this exception → 404
public class OrderNotFoundException extends RuntimeException {
    public OrderNotFoundException(Long id) {
        super("Order not found: " + id);
    }
}
```

When this exception propagates, `ResponseStatusExceptionResolver` catches it and sets the HTTP status automatically. No `@ExceptionHandler` needed.

⚠️ **EXAM TRAP**: `@ResponseStatus` on an exception class only sets the status code and uses the `reason` attribute for the response message — it doesn't let you customise the response body structure. For custom error response bodies, use `@ExceptionHandler`.

---

## `ResponseStatusException` — programmatic status

Throw `ResponseStatusException` directly in controller code without creating a custom exception class:

```java
@GetMapping("/{id}")
public Order findById(@PathVariable Long id) {
    return orderService.findById(id)
        .orElseThrow(() -> new ResponseStatusException(
            HttpStatus.NOT_FOUND,
            "Order not found: " + id
        ));
}
```

Clean for simple cases — no custom exception class needed.

## `BindingResult` — handling validation without exceptions

If you want to handle validation failures yourself instead of letting Spring throw `MethodArgumentNotValidException`, add `BindingResult` as the next parameter after `@Valid`:

```java
@PostMapping
public ResponseEntity<?> create(
        @RequestBody @Valid Order order,
        BindingResult bindingResult) {  // must immediately follow @Valid param

    if (bindingResult.hasErrors()) {
        // handle errors manually — no exception thrown
        List<String> errors = bindingResult.getFieldErrors()
            .stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .toList();
        return ResponseEntity.badRequest().body(errors);
    }

    return ResponseEntity.ok(orderService.save(order));
}
```

⚠️ **EXAM TRAP**: `BindingResult` must come **immediately after** the `@Valid` parameter it corresponds to. If you put anything between them, Spring won't associate them and will throw `MethodArgumentNotValidException` regardless.

---

## Plain-English summary

- `@GetMapping`, `@PostMapping`, `@PutMapping`, `@PatchMapping`, `@DeleteMapping` — HTTP verb shortcuts. PUT is full replace, PATCH is partial update.
- `@Valid` triggers bean validation. Fails → `MethodArgumentNotValidException` → 400 by default.
- `@ExceptionHandler` in a controller — handles exceptions from that controller only.
- `@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody` — global exception handling across all controllers.
- Controller-local `@ExceptionHandler` beats global `@RestControllerAdvice` for same exception type.
- `@ResponseStatus` on exception class — simple status code mapping, no body control.
- `ResponseStatusException` — throw inline without custom exception class.
- `BindingResult` immediately after `@Valid` — handle validation manually without exception.
