# Objective 3.1.2 — DispatcherServlet Request Lifecycle

## What DispatcherServlet is

`DispatcherServlet` is Spring MVC's **front controller**. Every HTTP request to a Spring MVC application goes through it. It doesn't handle requests itself — it delegates to specialised components and orchestrates the full request/response cycle.

Spring Boot auto-configures `DispatcherServlet` and maps it to `/` by default when `spring-boot-starter-web` is on the classpath. You never declare it manually in a Boot application.

---

## The full request lifecycle — in order

```
HTTP Request
     ↓
1. DispatcherServlet receives the request
     ↓
2. HandlerMapping — finds the right handler (controller method) for the URL
     ↓
3. HandlerAdapter — invokes the handler, bridging DispatcherServlet to the handler
     ↓
4. Handler (Controller method) — executes your business logic, returns ModelAndView or ResponseEntity
     ↓
5. HandlerExceptionResolver — handles any exception thrown by the handler (if applicable)
     ↓
6. ViewResolver — resolves a logical view name to a concrete View (if not REST)
     ↓
7. View — renders the response (Thymeleaf, JSP, etc.) (if not REST)
     ↓
HTTP Response
```

For REST controllers (`@RestController`), steps 6 and 7 are replaced by `HttpMessageConverter` — the return value is serialised directly to JSON/XML and written to the response body. No view resolution needed.

---

## Each component explained

### `HandlerMapping`

Looks at the incoming request URL and HTTP method and determines which controller method should handle it.

The most common implementation is `RequestMappingHandlerMapping` — it scans all `@Controller` beans for `@RequestMapping`, `@GetMapping`, etc. annotations and builds a map of URL patterns to handler methods.

```
GET /orders/123  →  OrderController.getOrder(Long id)
POST /orders     →  OrderController.createOrder(Order order)
```

Multiple `HandlerMapping` implementations can coexist — Spring tries them in priority order and uses the first match.

---

### `HandlerAdapter`

`DispatcherServlet` doesn't call your controller method directly — it calls a `HandlerAdapter` which knows how to invoke that type of handler.

`RequestMappingHandlerAdapter` handles `@RequestMapping`-based controllers. It resolves method parameters (from path variables, request body, query params, etc.) and invokes the method.

⚠️ **EXAM TRAP**: The reason `HandlerAdapter` exists is to decouple `DispatcherServlet` from the specifics of how handlers are invoked. `DispatcherServlet` works with any handler type — annotated controllers, `HttpRequestHandler`, `Controller` interface — without knowing their invocation details.

---

### `HandlerExceptionResolver`

If a handler throws an exception, `DispatcherServlet` passes it to registered `HandlerExceptionResolver` implementations. The first one that handles the exception wins.

Built-in resolvers:
- `ExceptionHandlerExceptionResolver` — handles `@ExceptionHandler` methods in `@RestControllerAdvice`
- `ResponseStatusExceptionResolver` — handles `@ResponseStatus` annotated exceptions
- `DefaultHandlerExceptionResolver` — handles standard Spring MVC exceptions (404, 405, etc.)

This is how `@RestControllerAdvice` + `@ExceptionHandler` fits into the lifecycle — covered in 3.2.1.

---

### `ViewResolver` and `View`

For traditional MVC (non-REST), the handler returns a logical view name like `"orders/list"`. `ViewResolver` maps that string to a concrete `View` implementation — a Thymeleaf template, a JSP, a redirect, etc.

```java
@Controller  // not @RestController — returns view names
public class OrderViewController {

    @GetMapping("/orders")
    public String listOrders(Model model) {
        model.addAttribute("orders", orderService.findAll());
        return "orders/list";  // logical view name → ViewResolver maps to template
    }
}
```

For REST controllers, this step is skipped entirely — `HttpMessageConverter` writes directly to the response.

---

### `HttpMessageConverter` — the REST path

When a `@RestController` method returns an object, `RequestMappingHandlerAdapter` uses registered `HttpMessageConverter` implementations to serialise the return value.

```
Return value: Order object
     ↓
HttpMessageConverter checks Accept header: "application/json"
     ↓
MappingJackson2HttpMessageConverter serialises to JSON
     ↓
Written to response body
```

Spring Boot auto-configures `MappingJackson2HttpMessageConverter` (Jackson) when Jackson is on the classpath. You can customise it or add converters for other formats.

---

## The lifecycle for a REST request — concrete example

```java
@RestController
@RequestMapping("/orders")
public class OrderController {

    @GetMapping("/{id}")
    public ResponseEntity<Order> getOrder(@PathVariable Long id) {
        return orderService.findById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }
}
```

For `GET /orders/123`:

1. `DispatcherServlet` receives the request
2. `RequestMappingHandlerMapping` matches `/orders/123` → `OrderController.getOrder()`
3. `RequestMappingHandlerAdapter` resolves `@PathVariable Long id = 123`, invokes the method
4. Method returns `ResponseEntity<Order>`
5. No exception — skip `HandlerExceptionResolver`
6. Return value is `ResponseEntity` — skip `ViewResolver`
7. `MappingJackson2HttpMessageConverter` serialises `Order` to JSON, writes to response

## `DispatcherServlet` configuration in Boot

Boot auto-configures everything. Key properties:

```properties
spring.mvc.servlet.path=/          # DispatcherServlet mapping — default is /
spring.mvc.servlet.load-on-startup=1  # initialise at startup, not first request
```

Multiple `DispatcherServlet` instances are possible (for different URL namespaces) but rare and out of scope for the exam.

---

## Plain-English summary

- `DispatcherServlet` is the front controller — every request goes through it.
- `HandlerMapping` — finds the right controller method for the request URL.
- `HandlerAdapter` — invokes the handler, resolving parameters. Decouples `DispatcherServlet` from handler specifics.
- `HandlerExceptionResolver` — handles exceptions thrown by handlers. Where `@ExceptionHandler` fits in.
- `ViewResolver` + `View` — for traditional MVC. Maps logical view name to template.
- `HttpMessageConverter` — for REST. Serialises return value to JSON/XML. Replaces view resolution.
- Boot auto-configures `DispatcherServlet` mapped to `/`. Jackson auto-configured when on classpath.
