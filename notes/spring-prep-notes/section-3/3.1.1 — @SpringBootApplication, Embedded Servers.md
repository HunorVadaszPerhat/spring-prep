# Objective 3.1.1 — `@SpringBootApplication`, Embedded Servers

## `@SpringBootApplication` — the composed annotation

`@SpringBootApplication` is a convenience annotation that combines three annotations into one. Understanding what each component does is an explicit exam objective.

```java
@SpringBootApplication
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

The three composed annotations:

```java
@SpringBootApplication
=
@Configuration           // this class is a source of bean definitions
+ @EnableAutoConfiguration  // activate Spring Boot's auto-configuration mechanism
+ @ComponentScan         // scan this package and sub-packages for components
```

Each one does a distinct job:

**`@Configuration`** — marks the main class as a `@Configuration` class. You can declare `@Bean` methods directly in it (though in practice most beans go in separate config classes).

**`@EnableAutoConfiguration`** — tells Spring Boot to read its auto-configuration candidates and apply the ones that match the current classpath and properties. This is what configures `DataSource`, `JdbcTemplate`, `DispatcherServlet`, and hundreds of other beans automatically. Covered in depth in 6.2.2.

**`@ComponentScan`** — scans the package of the main class and all sub-packages for `@Component`, `@Service`, `@Repository`, `@Controller`, etc. This is why the main class is conventionally placed at the root package — everything below it gets scanned.

⚠️ **EXAM TRAP**: The exam may show `@SpringBootApplication` and ask what it's composed of, or show the three annotations separately and ask what single annotation replaces them. Both directions are tested.

## Customising `@SpringBootApplication`

You can pass attributes directly on `@SpringBootApplication` that delegate to its composed annotations:

```java
@SpringBootApplication(
    scanBasePackages = "com.example",        // delegates to @ComponentScan
    exclude = DataSourceAutoConfiguration.class  // delegates to @EnableAutoConfiguration
)
public class OrderApplication { }
```

This is the same as splitting them out — just more concise.

## `SpringApplication.run()`

The entry point that bootstraps the application:

```java
public static void main(String[] args) {
    SpringApplication.run(OrderApplication.class, args);
}
```

What it does:
1. Creates the `ApplicationContext`
2. Registers the main class as a configuration source
3. Triggers component scanning and auto-configuration
4. Starts the embedded server (if web dependencies are present)
5. Calls any `CommandLineRunner` or `ApplicationRunner` beans

---

## Embedded servers

Spring Boot embeds a servlet container inside the fat JAR — you don't deploy to an external server. The application starts with `java -jar`.

**Tomcat** is the default. Included automatically with `spring-boot-starter-web`.

**Switching to Jetty:**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jetty</artifactId>
</dependency>
```

**Switching to Undertow** — same pattern, replace `jetty` with `undertow`.

⚠️ **EXAM TRAP**: To switch servers you must **exclude** the default Tomcat starter first, then add the alternative. Adding Jetty without excluding Tomcat results in two server starters on the classpath — Spring Boot will still use Tomcat (it has higher auto-configuration priority) and you won't get the switch you intended.

## Configuring the embedded server

```properties
server.port=8080                    # default port
server.port=0                       # random available port (useful in tests)
server.servlet.context-path=/api    # context path prefix for all endpoints
server.tomcat.max-threads=200       # Tomcat-specific thread pool size
```

Switching port programmatically:
```java
@SpringBootApplication
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(OrderApplication.class);
        app.setDefaultProperties(Map.of("server.port", "9090"));
        app.run(args);
    }
}
```

## `CommandLineRunner` vs `ApplicationRunner`

Both run after the application context is fully started — useful for startup tasks like data seeding, cache warming, or printing startup info.

```java
@Component
public class DataSeeder implements CommandLineRunner {

    @Override
    public void run(String... args) throws Exception {
        // args = raw command-line arguments as String[]
        System.out.println("Seeding data...");
    }
}

@Component
public class StartupLogger implements ApplicationRunner {

    @Override
    public void run(ApplicationArguments args) throws Exception {
        // args = parsed ApplicationArguments object
        // args.getOptionNames() → --server.port=8080 → "server.port"
        // args.getNonOptionArgs() → positional args
        boolean debug = args.containsOption("debug");
    }
}
```

The only difference is the argument type:
- `CommandLineRunner` → `String... args` — raw strings
- `ApplicationRunner` → `ApplicationArguments args` — parsed, structured access to named options and positional args

⚠️ **EXAM TRAP**: This is a direct exam question — what is the difference between `CommandLineRunner` and `ApplicationRunner`? The answer is always the argument type. Both run at the same point in the lifecycle, both can be ordered with `@Order`.

## Multiple runners with `@Order`

```java
@Component
@Order(1)  // runs first
public class SchemaValidator implements CommandLineRunner {
    public void run(String... args) { ... }
}

@Component
@Order(2)  // runs second
public class DataSeeder implements CommandLineRunner {
    public void run(String... args) { ... }
}
```

## Plain-English summary

- `@SpringBootApplication` = `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`.
- Place main class at root package — `@ComponentScan` scans downward from there.
- Default embedded server is Tomcat. Switch to Jetty or Undertow by excluding Tomcat first then adding the alternative.
- `CommandLineRunner` → raw `String[]` args. `ApplicationRunner` → parsed `ApplicationArguments`. Both run post-startup.
- Multiple runners ordered with `@Order`. Lower number runs first.
