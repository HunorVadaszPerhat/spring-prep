# Objective 3.1.4 — Deployment: Fat JAR and WAR

## Two deployment models

Spring Boot supports two packaging approaches. The exam tests both, their differences, and when each is appropriate.

---

## Fat JAR (the Boot default)

A fat JAR (also called an uber JAR or executable JAR) packages your application code, all dependencies, and the embedded server into a single self-contained `.jar` file. You run it with `java -jar`.

```
java -jar order-service-1.0.jar
```

The embedded Tomcat (or Jetty/Undertow) starts inside the JVM process. No external server needed. This is the standard Spring Boot deployment model — one artifact, one command, runs anywhere Java is installed.

**How it works — `spring-boot-maven-plugin`:**

The standard Maven `jar` plugin creates a thin JAR with only your compiled classes. Spring Boot's plugin repackages it to include all dependencies in a nested structure:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
        </plugin>
    </plugins>
</build>
```

With `spring-boot-parent` as the parent POM, the plugin version is managed automatically — you just declare it.

The repackaged JAR structure:
```
order-service-1.0.jar
├── BOOT-INF/
│   ├── classes/          ← your compiled code
│   └── lib/              ← all dependency JARs
├── META-INF/
│   └── MANIFEST.MF       ← Main-Class: JarLauncher
└── org/springframework/boot/loader/
    └── JarLauncher.class  ← Spring's custom class loader entry point
```

⚠️ **EXAM TRAP**: The `spring-boot-maven-plugin` is what creates the fat JAR. Without it, `mvn package` produces a thin JAR with only your classes — it can't be run with `java -jar` because dependencies are missing. The plugin is **required** for executable JAR packaging.

## WAR deployment — traditional external server

If your organisation uses an external servlet container (Tomcat, JBoss, WebLogic), you can package as a WAR and deploy it there. This requires a few changes.

**Step 1: Change packaging in `pom.xml`:**
```xml
<packaging>war</packaging>
```

**Step 2: Extend `SpringBootServletInitializer`:**

This is the critical change. An external servlet container doesn't call `main()` — it looks for a `ServletContainerInitializer` to bootstrap the application. `SpringBootServletInitializer` provides that bridge.

```java
@SpringBootApplication
public class OrderApplication extends SpringBootServletInitializer {

    @Override
    protected SpringApplicationBuilder configure(SpringApplicationBuilder builder) {
        return builder.sources(OrderApplication.class);
    }

    // Keep main() for running locally as fat JAR during development
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

⚠️ **EXAM TRAP**: Extending `SpringBootServletInitializer` and overriding `configure()` is **required** for WAR deployment to an external container. Without it, the external server doesn't know how to start the Spring application. The `main()` method is ignored by external containers — only used when running locally.

**Step 3: Mark embedded Tomcat as `provided`:**

The WAR will be deployed to an external container that already has its own servlet container. You don't want to bundle Tomcat inside the WAR — mark it as `provided` so it's excluded from the WAR artifact:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-tomcat</artifactId>
    <scope>provided</scope>  <!-- excluded from WAR, provided by external container -->
</dependency>
```

## Fat JAR vs WAR — comparison

| | Fat JAR | WAR |
|---|---|---|
| Packaging | `.jar` | `.war` |
| Server | Embedded (Tomcat/Jetty/Undertow) | External (provided) |
| Run command | `java -jar app.jar` | Deploy to container |
| `SpringBootServletInitializer` | Not needed | Required |
| `spring-boot-maven-plugin` | Required | Required (still repackages) |
| `main()` used | Yes | Only for local dev |
| Cloud/container friendly | ✅ | Less common |

---

## Keeping both options open

A common pattern is to make the application deployable either way — as a fat JAR locally and as a WAR in production:

```java
@SpringBootApplication
public class OrderApplication extends SpringBootServletInitializer {

    @Override
    protected SpringApplicationBuilder configure(SpringApplicationBuilder builder) {
        return builder.sources(OrderApplication.class);
    }

    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);  // fat JAR path
    }
}
```

With `<packaging>war</packaging>` and `spring-boot-maven-plugin`, this builds a WAR that is also executable as `java -jar` — Spring Boot's plugin makes the WAR executable too.

---

## `spring-boot-maven-plugin` goals

| Goal | What it does |
|---|---|
| `spring-boot:run` | Runs the application directly via Maven — no packaging needed |
| `spring-boot:repackage` | Repackages the existing JAR/WAR as an executable artifact |
| `spring-boot:build-image` | Builds a Docker container image (OCI) |

The `repackage` goal is bound to the `package` phase automatically when the plugin is declared — running `mvn package` repackages without explicitly calling the goal.

---

## Plain-English summary

- Fat JAR = self-contained executable JAR with embedded server. Built by `spring-boot-maven-plugin`. Run with `java -jar`.
- WAR = deployed to external servlet container. Requires `<packaging>war</packaging>`, extending `SpringBootServletInitializer`, overriding `configure()`, and marking embedded Tomcat as `provided`.
- `SpringBootServletInitializer` is the bridge between the external container and Spring Boot — required for WAR, not needed for fat JAR.
- Keep `main()` even in WAR packaging — used for local fat JAR execution, ignored by external containers.
- `spring-boot-maven-plugin` is required in both cases to produce a runnable artifact.
