# Objective 4.1.2 — `@SpringBootTest` and `webEnvironment` Options

## What `@SpringBootTest` does

`@SpringBootTest` loads a full Spring application context for integration testing. Unlike slice tests (`@WebMvcTest`, `@DataJpaTest`) which load only a layer, `@SpringBootTest` starts the entire context — all beans, all auto-configuration, all layers together.

It's the integration test annotation. Use it when you need to test how components work together across multiple layers, or when you need the full context for a test.

---

## How `@SpringBootTest` finds the main class

`@SpringBootTest` walks up the package hierarchy from the test class looking for a class annotated with `@SpringBootApplication` (or `@SpringBootConfiguration`). This is why the conventional structure — main class at the root package, tests mirroring the source package structure — works automatically.

```
com.example.OrderApplication          ← @SpringBootApplication found here
com.example.service.OrderServiceTest  ← @SpringBootTest walks up, finds it
```

If no `@SpringBootApplication` class is found, the test fails at startup. You can specify it explicitly:

```java
@SpringBootTest(classes = OrderApplication.class)
class OrderServiceTest { }
```

## The four `webEnvironment` options

This is a HIGH VALUE exam topic — know each option, what it starts, and when to use it.

### `MOCK` — the default

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.MOCK)
// or just
@SpringBootTest  // MOCK is the default
class OrderServiceTest { }
```

Loads the full application context but with a **mocked** servlet environment — no real HTTP server starts. The web layer is available for testing via `MockMvc` but no actual port is opened.

Use with `@AutoConfigureMockMvc` to inject `MockMvc`:

```java
@SpringBootTest
@AutoConfigureMockMvc
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void shouldReturnOrder() throws Exception {
        mockMvc.perform(get("/orders/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.id").value(1));
    }
}
```

---

### `RANDOM_PORT`

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest { }
```

Starts a **real embedded server** on a random available port. Use this for full HTTP integration tests — real HTTP connections, real serialisation, real server lifecycle.

Inject the actual port with `@LocalServerPort`:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest {

    @LocalServerPort
    private int port;  // injected with the actual port number

    @Autowired
    private TestRestTemplate restTemplate;  // pre-configured for the test server

    @Test
    void shouldReturnOrder() {
        ResponseEntity<Order> response = restTemplate.getForEntity(
            "/orders/1",  // relative URL — TestRestTemplate knows the base URL
            Order.class
        );
        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.OK);
    }
}
```

`TestRestTemplate` is auto-configured by Boot for `RANDOM_PORT` tests. It knows the base URL and port automatically — you use relative paths.

⚠️ **EXAM TRAP**: Use `TestRestTemplate` (not `RestTemplate`) for `RANDOM_PORT` integration tests. `TestRestTemplate` is a test-friendly wrapper that handles the base URL, doesn't throw exceptions on 4xx/5xx (returns the `ResponseEntity` instead), and supports basic auth configuration.

---

### `DEFINED_PORT`

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.DEFINED_PORT)
class OrderIntegrationTest { }
```

Starts a real embedded server on the **configured port** (from `server.port` in properties, default 8080). Use this when you need to test against a specific known port — for example, when testing with a client configured for a fixed URL.

⚠️ **EXAM TRAP**: `DEFINED_PORT` can cause test failures if the port is already in use. `RANDOM_PORT` avoids this problem entirely and is preferred for most integration tests.

---

### `NONE`

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.NONE)
class OrderServiceTest { }
```

Loads the full application context but with **no web environment at all** — not even a mock one. Use this when you're testing service or data layer beans that don't need the web layer, but still want the full context (all auto-configuration, real database, etc.).

---

## The four options compared

| Option | Server | Port | Use with | Use for |
|---|---|---|---|---|
| `MOCK` (default) | Mock servlet | None | `MockMvc` | Web layer tests without real HTTP |
| `RANDOM_PORT` | Real embedded | Random | `TestRestTemplate` | Full HTTP integration tests |
| `DEFINED_PORT` | Real embedded | Configured | `TestRestTemplate` | Fixed-port integration tests |
| `NONE` | None | None | Direct bean injection | Service/data layer integration tests |

---

## `TestRestTemplate` vs `MockMvc`

| | `MockMvc` | `TestRestTemplate` |
|---|---|---|
| Web environment | `MOCK` | `RANDOM_PORT` / `DEFINED_PORT` |
| HTTP | No real HTTP | Real HTTP |
| Error handling | Throws on framework errors | Returns `ResponseEntity` for 4xx/5xx |
| Speed | Faster | Slower (real network stack) |
| Tests | Controller + web layer logic | Full end-to-end HTTP |

⚠️ **EXAM TRAP**: `MockMvc` requires `MOCK` web environment (or `@WebMvcTest`). `TestRestTemplate` requires a real server (`RANDOM_PORT` or `DEFINED_PORT`). Mixing them up is a common exam distractor.

## Context caching

Spring caches the application context between tests that share the same configuration. If two test classes have identical `@SpringBootTest` configurations, they reuse the same context — startup cost is paid once.

Context caching is invalidated when:
- `@MockBean` is used (replaces a bean — changes the context)
- `@DirtiesContext` is used (explicitly marks context as dirty)
- Different `webEnvironment` values
- Different active profiles

⚠️ **EXAM TRAP**: `@DirtiesContext` forces Spring to close and recreate the context after the test. It's expensive — avoid it unless a test actually modifies shared state. The exam may ask what `@DirtiesContext` does or when to use it.

```java
@SpringBootTest
@DirtiesContext(classMode = DirtiesContext.ClassMode.AFTER_EACH_TEST_METHOD)
class StatefulTest { }
```

## `@SpringBootTest` with `@ActiveProfiles`

```java
@SpringBootTest
@ActiveProfiles("test")  // activates test profile — loads application-test.properties
class OrderIntegrationTest { }
```

Common pattern — `application-test.properties` points to an H2 in-memory database instead of the real one, overriding production datasource config.

## Plain-English summary

- `@SpringBootTest` loads the full application context. Finds `@SpringBootApplication` by walking up the package hierarchy.
- `MOCK` (default) — mock servlet, no real HTTP. Use with `MockMvc` via `@AutoConfigureMockMvc`.
- `RANDOM_PORT` — real server on random port. Use with `TestRestTemplate`. Preferred for HTTP integration tests.
- `DEFINED_PORT` — real server on configured port. Can fail if port in use.
- `NONE` — no web environment. Full context for service/data layer tests.
- `TestRestTemplate` — test-friendly, knows base URL, doesn't throw on 4xx/5xx. Use with real server only.
- Spring caches contexts between tests with same config. `@MockBean` and `@DirtiesContext` break the cache.
