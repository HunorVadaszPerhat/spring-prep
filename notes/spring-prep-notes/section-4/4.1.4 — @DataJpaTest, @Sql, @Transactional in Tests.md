## `@DataJpaTest` — recap and deeper detail

`@DataJpaTest` is a **slice test** annotation. It loads only the JPA layer:

- JPA entities and repositories
- Spring Data JPA infrastructure
- An in-memory H2 database (by default)
- `TestEntityManager`
- Hibernate/JPA auto-configuration

It does **not** load:
- `@Service` beans
- `@Controller` beans
- `@Component` beans outside the JPA layer
- Full auto-configuration

```java
@DataJpaTest
class OrderRepositoryTest {

    @Autowired
    private OrderRepository orderRepository;

    @Autowired
    private TestEntityManager entityManager;

    @Test
    void shouldFindByStatus() {
        // arrange — persist test data
        Order order = new Order("customer-1", BigDecimal.TEN);
        entityManager.persistAndFlush(order);  // persist + flush in one call

        // act
        List<Order> pending = orderRepository.findByStatus("PENDING");

        // assert
        assertThat(pending).hasSize(1);
        assertThat(pending.get(0).getCustomerName()).isEqualTo("customer-1");
    }
}
```

## `@DataJpaTest` — the details that trip people up

**In-memory database replacement:**

`@DataJpaTest` replaces any configured `DataSource` with an embedded H2 database by default. Your `application.properties` datasource configuration is ignored.

To use your real database:
```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class OrderRepositoryTest { }
```

⚠️ **EXAM TRAP**: `@DataJpaTest` uses H2 by default — real datasource is replaced. `Replace.NONE` disables this replacement.

**`@Transactional` by default:**

`@DataJpaTest` is `@Transactional` by default — each test rolls back automatically. You don't add `@Transactional` yourself.

**`@Service` beans are not available:**

If your repository has a dependency on a `@Service` bean (unusual but possible), it won't be found in a `@DataJpaTest` context. Use `@SpringBootTest` if you need the full context, or use `@MockBean` to provide the missing dependency.

## `TestEntityManager`

A test-specific wrapper around JPA's `EntityManager`. Provides convenience methods for test setup:

```java
// persist() — persists entity, returns managed entity
Order managed = entityManager.persist(new Order("customer-1", BigDecimal.TEN));

// flush() — forces pending changes to be written to the DB within the transaction
entityManager.flush();

// persistAndFlush() — persist + flush in one call
Order managed = entityManager.persistAndFlush(new Order("customer-1", BigDecimal.TEN));

// find() — look up entity by primary key
Order found = entityManager.find(Order.class, managed.getId());

// clear() — clears the persistence context (forces fresh load on next find)
entityManager.clear();
```

**Why `flush()` matters in tests:**

JPA batches writes and may not send them to the database immediately. If you persist an entity and then call a repository query method in the same test, the entity might not be visible to the query yet. `flush()` forces the write so the query sees it.

⚠️ **EXAM TRAP**: Without `flush()`, a persist followed by a query in the same test may return stale results because JPA hasn't written to the database yet. Always `flush()` between a persist and a query in `@DataJpaTest` tests.

## `@Sql` — executing SQL scripts in tests

`@Sql` runs SQL scripts at specific lifecycle points in a test. Covered in 2.2.4 — here's the full detail.

**Basic usage:**
```java
@Test
@Sql("/test-data/insert-orders.sql")  // runs before the test method by default
void shouldFindPendingOrders() {
    List<Order> pending = orderRepository.findByStatus("PENDING");
    assertThat(pending).hasSize(3);
}
```

**Explicit execution phase:**
```java
@Test
@Sql(
    scripts = "/test-data/insert-orders.sql",
    executionPhase = Sql.ExecutionPhase.BEFORE_TEST_METHOD  // default
)
void beforeTest() { }

@Test
@Sql(
    scripts = "/test-data/cleanup.sql",
    executionPhase = Sql.ExecutionPhase.AFTER_TEST_METHOD
)
void afterTest() { }
```

**Multiple scripts:**
```java
@Test
@Sql({"/test-data/schema.sql", "/test-data/insert-orders.sql"})
void withMultipleScripts() { }
```

**Class-level `@Sql`:**
```java
@DataJpaTest
@Sql("/test-data/base-data.sql")  // runs before every test method in the class
class OrderRepositoryTest { }
```

## `@Sql` + `@Transactional` interaction

When `@Sql` is used in a `@Transactional` test, the scripts run **within the same transaction** by default:

```java
@DataJpaTest  // @Transactional by default
class OrderRepositoryTest {

    @Test
    @Sql("/test-data/insert-orders.sql")
    void test() {
        // insert-orders.sql ran inside the test transaction
        // its inserts are visible to the test
        // everything rolls back after the test — SQL inserts included
    }
}
```

This is the right behaviour for most tests — the SQL data is temporary and rolls back with the test.

If you need the SQL to run in a separate transaction (committed independently):
```java
@Test
@Sql(
    scripts = "/test-data/insert-orders.sql",
    config = @SqlConfig(transactionMode = SqlConfig.TransactionMode.ISOLATED)
)
void withIsolatedSql() { }
```

`ISOLATED` mode runs the script in its own transaction that commits. The data persists for the test but may need manual cleanup.

## `@Transactional` in tests — the `REQUIRES_NEW` reminder

Covered in 2.2.4 but worth repeating here as it directly affects `@DataJpaTest` tests:

If a repository or service method called from the test uses `REQUIRES_NEW`, that inner transaction commits independently and is **not** rolled back by the test's transaction:

```java
@DataJpaTest
class AuditTest {

    @Test
    void auditDataPersistsIndependently() {
        // if orderRepository.save() internally calls something with REQUIRES_NEW
        // that committed data stays in the DB after the test
        // the test transaction rollback doesn't reach it
    }
}
```

## Combining `@DataJpaTest` with `@ActiveProfiles`

```java
@DataJpaTest
@ActiveProfiles("test")
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class OrderRepositoryIntegrationTest {
    // uses real datasource from application-test.properties
    // test profile active
}
```

This is the pattern for testing against a real database in a CI environment — test profile points to a test database, `Replace.NONE` preserves the datasource configuration.

## Plain-English summary

- `@DataJpaTest` loads JPA layer only — entities, repositories, H2 in-memory by default, `@Transactional` by default.
- `Replace.NONE` disables H2 replacement — uses real datasource from properties.
- `TestEntityManager` — test helper for JPA setup. Always `flush()` between persist and query.
- `@Sql` runs scripts before/after test methods. Default phase is `BEFORE_TEST_METHOD`.
- `@Sql` scripts run inside the test transaction by default — rolled back with the test.
- `SqlConfig.TransactionMode.ISOLATED` runs SQL in a separate committed transaction.
- `REQUIRES_NEW` in called code commits independently — not rolled back by the test transaction.
