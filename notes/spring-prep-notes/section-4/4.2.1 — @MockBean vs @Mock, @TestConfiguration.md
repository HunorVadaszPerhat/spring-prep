# Objective 4.2.1 ‚Äî `@MockBean` vs `@Mock`, `@TestConfiguration`

## The core distinction ‚Äî context vs no context

This is one of the most tested topics in Section 4. The difference between `@MockBean` and `@Mock` is not subtle ‚Äî they operate in completely different environments.

**`@Mock`** ‚Äî Mockito annotation. Creates a mock object. No Spring context involved. Used in pure unit tests with `@ExtendWith(MockitoExtension.class)`.

**`@MockBean`** ‚Äî Spring Boot test annotation. Creates a Mockito mock AND registers it as a bean in the Spring application context, replacing any existing bean of that type. Used in Spring integration tests with `@SpringBootTest`, `@WebMvcTest`, `@DataJpaTest`.

---

## `@Mock` ‚Äî pure Mockito, no Spring

```java
@ExtendWith(MockitoExtension.class)  // Mockito only ‚Äî no Spring context
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository;  // Mockito mock ‚Äî not in any Spring context

    @Mock
    private PaymentGateway paymentGateway;

    @InjectMocks
    private OrderService orderService;  // real instance, mocks injected by Mockito

    @Test
    void shouldPlaceOrder() {
        when(orderRepository.save(any())).thenReturn(new Order());
        orderService.placeOrder(new Order());
        verify(orderRepository).save(any());
    }
}
```

No Spring context starts. Fast. Isolated. Tests one class at a time.

---

## `@MockBean` ‚Äî Mockito mock registered in Spring context

```java
@WebMvcTest(OrderController.class)  // Spring context starts ‚Äî web layer only
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean  // creates mock AND registers it in the Spring context
    private OrderService orderService;  // replaces the real OrderService bean

    @Test
    void shouldReturnOrder() throws Exception {
        Order order = new Order(1L, "customer-1", BigDecimal.TEN);
        when(orderService.findById(1L)).thenReturn(Optional.of(order));

        mockMvc.perform(get("/orders/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.customerName").value("customer-1"));
    }
}
```

`@WebMvcTest` loads the web layer but not `@Service` beans. Without `@MockBean`, Spring can't wire `OrderController` because `OrderService` is missing from the context. `@MockBean` provides it as a mock.

‚öÝÔ∏è **EXAM TRAP**: `@MockBean` replaces the real bean in the Spring context. If `OrderService` was auto-configured or component-scanned into the context, `@MockBean` removes it and puts a mock in its place. The original bean is gone for that test.

‚öÝÔ∏è **EXAM TRAP**: `@MockBean` invalidates Spring's context cache. Because it changes the context (replaces a bean), Spring can't reuse the same context for a test class that uses `@MockBean` ‚Äî it creates a new context. This makes tests with `@MockBean` slower than tests that share a cached context.

---

## `@SpyBean` ‚Äî partial mock in Spring context

The Spring Boot equivalent of Mockito's `@Spy`. Creates a real bean but allows individual methods to be stubbed. The real implementation is called for unstubbed methods.

```java
@SpringBootTest
class OrderServiceTest {

    @SpyBean
    private OrderService orderService;  // real OrderService bean, but stubbable

    @Test
    void shouldUseRealImplementationMostly() {
        doReturn(BigDecimal.ZERO).when(orderService).calculateDiscount(any());
        // all other methods call the real implementation
    }
}
```

---

## Side-by-side comparison

| | `@Mock` | `@MockBean` |
|---|---|---|
| Framework | Mockito | Spring Boot Test |
| Spring context | Not involved | Required ‚Äî registers in context |
| Replaces real bean | N/A ‚Äî no context | Yes ‚Äî replaces existing bean |
| Used with | `@ExtendWith(MockitoExtension.class)` | `@SpringBootTest`, `@WebMvcTest`, etc. |
| Context cache | No effect | Invalidates cache |
| Speed | Fast ‚Äî no context startup | Slower ‚Äî context starts |
| Counterpart | `@Spy` | `@SpyBean` |

## `@TestConfiguration`

`@TestConfiguration` defines beans that should be added to (or override beans in) the Spring context **only during tests**. It's a test-only `@Configuration` class.

```java
@TestConfiguration
public class TestPaymentConfig {

    @Bean
    public PaymentGateway paymentGateway() {
        // test-specific implementation ‚Äî in-memory, no real charges
        return new FakePaymentGateway();
    }
}
```

**Two ways to use it:**

**1. As a nested static class** ‚Äî automatically picked up by the enclosing test class:

```java
@SpringBootTest
class OrderServiceTest {

    @Autowired
    private OrderService orderService;  // gets FakePaymentGateway injected

    @TestConfiguration
    static class TestConfig {

        @Bean
        public PaymentGateway paymentGateway() {
            return new FakePaymentGateway();
        }
    }
}
```

**2. As a top-level class** ‚Äî must be explicitly imported:

```java
@TestConfiguration
public class MockServicesConfig {

    @Bean
    public PaymentGateway paymentGateway() {
        return new FakePaymentGateway();
    }
}

@SpringBootTest
@Import(MockServicesConfig.class)  // must explicitly import top-level @TestConfiguration
class OrderServiceTest { }
```

‚öÝÔ∏è **EXAM TRAP**: A nested `@TestConfiguration` class is picked up automatically. A top-level `@TestConfiguration` class must be explicitly imported with `@Import`. This asymmetry is a direct exam question.

---

## `@TestConfiguration` vs `@MockBean` ‚Äî when to use which

**Use `@MockBean`** when you want a Mockito mock ‚Äî stubbing, verification, no real implementation needed.

**Use `@TestConfiguration`** when you want a real alternative implementation ‚Äî a fake, an in-memory version, or a test double with actual logic.

```java
// @MockBean ‚Äî stub behaviour, verify interactions
@MockBean
private PaymentGateway paymentGateway;  // mock ‚Äî no real logic

// @TestConfiguration ‚Äî real test implementation
@Bean
public PaymentGateway paymentGateway() {
    return new FakePaymentGateway();  // real logic ‚Äî always returns success
}
```

## `spring-boot-starter-test` ‚Äî what it bundles

The exam asks what this starter includes. The full list:

- **JUnit 5** (Jupiter) ‚Äî test framework
- **Mockito** ‚Äî mocking
- **AssertJ** ‚Äî fluent assertions
- **Hamcrest** ‚Äî matchers
- **JSONassert** ‚Äî JSON assertions
- **JsonPath** ‚Äî JSON path expressions
- **Spring Test** ‚Äî `MockMvc`, `TestRestTemplate`, Spring test utilities
- **Spring Boot Test** ‚Äî `@SpringBootTest`, `@MockBean`, `@TestConfiguration`, slice annotations

Everything you need in one dependency. You don't add JUnit or Mockito separately when using this starter.

---

## Plain-English summary

- `@Mock` ‚Äî Mockito only, no Spring context. Fast unit tests. Paired with `@ExtendWith(MockitoExtension.class)`.
- `@MockBean` ‚Äî Mockito mock registered in Spring context. Replaces real bean. Invalidates context cache. Paired with Spring test annotations.
- `@SpyBean` ‚Äî real bean in Spring context with stubbable methods. Spring Boot's `@Spy`.
- `@TestConfiguration` ‚Äî test-only bean definitions. Nested class = automatic. Top-level class = needs `@Import`.
- `@TestConfiguration` vs `@MockBean`: use `@TestConfiguration` for real test implementations, `@MockBean` for pure mocks.
- `spring-boot-starter-test` bundles JUnit 5, Mockito, AssertJ, Hamcrest, JSONassert, JsonPath, Spring Test, Spring Boot Test.
