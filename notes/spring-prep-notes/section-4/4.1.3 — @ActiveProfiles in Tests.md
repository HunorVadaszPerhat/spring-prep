# Objective 4.1.3 — `@ActiveProfiles` in Tests

## This is a short objective

`@ActiveProfiles` in tests was covered substantially in **1.3.2** (profiles) and has already appeared in several examples throughout Sections 2, 3, and 4.1.2. This objective is about consolidating that knowledge in the test context specifically — there isn't much new ground here.

---

## What `@ActiveProfiles` does in tests

Activates one or more Spring profiles for the duration of the test class. The activated profiles determine which `@Profile`-annotated beans are registered and which profile-specific property files are loaded.

```java
@SpringBootTest
@ActiveProfiles("test")  // activates the "test" profile
class OrderServiceIntegrationTest {

    // application-test.properties is loaded
    // beans annotated @Profile("test") are registered
    // beans annotated @Profile("prod") are NOT registered
}
```

Multiple profiles:
```java
@SpringBootTest
@ActiveProfiles({"test", "mock-payments"})  // both profiles active simultaneously
class OrderServiceIntegrationTest { }
```

---

## The typical test profile pattern

The most common use in real applications and on the exam:

```
application.properties           ← base config, always loaded
application-test.properties      ← overrides for tests
```

`application-test.properties` typically overrides:
- `spring.datasource.url` → H2 in-memory instead of real database
- `spring.jpa.hibernate.ddl-auto` → `create-drop` for tests
- External service URLs → pointing to mocks or stubs

```properties
# application-test.properties
spring.datasource.url=jdbc:h2:mem:testdb
spring.datasource.driver-class-name=org.h2.Driver
spring.jpa.hibernate.ddl-auto=create-drop
payment.gateway.url=http://localhost:8089/mock-payments
```

---

## `@ActiveProfiles` vs `spring.profiles.active` in properties

In tests, always use `@ActiveProfiles` rather than setting `spring.profiles.active` in a properties file. Setting `spring.profiles.active` in `application.properties` affects all runs — tests, local dev, CI — unless overridden. `@ActiveProfiles` scopes the activation to that specific test class only.

⚠️ **EXAM TRAP**: `@ActiveProfiles` on a test class applies to that class only. It does not affect other test classes or application runs. It's the clean, test-scoped way to activate profiles.

---

## Combining with other test annotations

`@ActiveProfiles` combines cleanly with any test annotation:

```java
@DataJpaTest
@ActiveProfiles("test")
class OrderRepositoryTest { }

@WebMvcTest(OrderController.class)
@ActiveProfiles("test")
class OrderControllerTest { }

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@ActiveProfiles({"test", "integration"})
class OrderIntegrationTest { }
```

---

## Profile-specific beans in tests

You can define test-only beans scoped to a profile:

```java
@TestConfiguration
public class MockPaymentConfig {

    @Bean
    @Profile("test")  // only registered when test profile is active
    public PaymentGateway mockPaymentGateway() {
        return mock(PaymentGateway.class);
    }
}
```

When `@ActiveProfiles("test")` is on the test class, this bean is registered. In production, the real `PaymentGateway` bean from a non-test configuration is used instead.

---

## Plain-English summary

- `@ActiveProfiles("name")` on a test class activates that profile for the test context.
- Loads `application-{profile}.properties` and registers `@Profile`-annotated beans.
- Scoped to the test class — doesn't affect other tests or the application.
- Prefer `@ActiveProfiles` over `spring.profiles.active` in properties for test-specific activation.
- Combines with any test slice annotation — `@DataJpaTest`, `@WebMvcTest`, `@SpringBootTest`.
