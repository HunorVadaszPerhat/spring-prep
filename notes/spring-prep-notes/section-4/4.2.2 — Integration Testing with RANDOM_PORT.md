# Objective 4.2.2 — Integration Testing with `RANDOM_PORT`

## This objective builds on 4.1.2

The `RANDOM_PORT` web environment was introduced in 4.1.2. This objective goes deeper on the practical patterns for writing real HTTP integration tests — `TestRestTemplate` usage, asserting responses, and combining with `@MockBean`.

---

## The full pattern

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest {

    @LocalServerPort
    private int port;  // injected with actual port — useful for building absolute URLs

    @Autowired
    private TestRestTemplate restTemplate;  // pre-configured for the test server

    @Test
    void shouldCreateAndRetrieveOrder() {
        // POST — create an order
        Order newOrder = new Order("customer-1", BigDecimal.TEN);
        ResponseEntity<Order> createResponse = restTemplate.postForEntity(
            "/orders",
            newOrder,
            Order.class
        );

        assertThat(createResponse.getStatusCode()).isEqualTo(HttpStatus.CREATED);
        Order created = createResponse.getBody();
        assertThat(created.getId()).isNotNull();

        // GET — retrieve the created order
        ResponseEntity<Order> getResponse = restTemplate.getForEntity(
            "/orders/{id}",
            Order.class,
            created.getId()
        );

        assertThat(getResponse.getStatusCode()).isEqualTo(HttpStatus.OK);
        assertThat(getResponse.getBody().getCustomerName()).isEqualTo("customer-1");
    }
}
```

`TestRestTemplate` uses relative paths — it knows the base URL and port automatically. You don't need to build `http://localhost:{port}/orders` manually.

## `TestRestTemplate` — key behaviours

**Does not throw on 4xx/5xx:**

Regular `RestTemplate` throws `HttpClientErrorException` on 4xx and `HttpServerErrorException` on 5xx. `TestRestTemplate` returns the `ResponseEntity` instead — you assert on the status code rather than catching exceptions:

```java
// Regular RestTemplate — throws on 404
try {
    restTemplate.getForObject("/orders/999", Order.class);
} catch (HttpClientErrorException.NotFound e) { ... }

// TestRestTemplate — returns ResponseEntity, no throw
ResponseEntity<Order> response = testRestTemplate.getForEntity("/orders/999", Order.class);
assertThat(response.getStatusCode()).isEqualTo(HttpStatus.NOT_FOUND);
```

⚠️ **EXAM TRAP**: `TestRestTemplate` does not throw exceptions for error responses. This is a deliberate design choice for tests — you assert on the response rather than catching exceptions.

**Basic auth support:**

```java
TestRestTemplate authedTemplate = restTemplate.withBasicAuth("user", "password");
ResponseEntity<Order> response = authedTemplate.getForEntity("/admin/orders", Order.class);
```

## Combining `RANDOM_PORT` with `@MockBean`

Real HTTP integration tests often need to mock external dependencies — payment gateways, email services, third-party APIs — while testing the real HTTP and service layers:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @MockBean
    private PaymentGateway paymentGateway;  // mock external dependency

    @MockBean
    private EmailService emailService;  // don't send real emails in tests

    @Test
    void shouldPlaceOrderSuccessfully() {
        // stub external dependencies
        when(paymentGateway.charge(any())).thenReturn(true);
        doNothing().when(emailService).sendConfirmation(any());

        Order order = new Order("customer-1", BigDecimal.TEN);
        ResponseEntity<Order> response = restTemplate.postForEntity(
            "/orders", order, Order.class
        );

        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.CREATED);
        verify(paymentGateway).charge(any());
        verify(emailService).sendConfirmation(any());
    }
}
```

This tests the full HTTP stack (serialisation, controller, service layer) with real beans, while isolating external dependencies as mocks.

## Testing error responses

```java
@Test
void shouldReturn404ForMissingOrder() {
    ResponseEntity<ErrorResponse> response = restTemplate.getForEntity(
        "/orders/999",
        ErrorResponse.class
    );

    assertThat(response.getStatusCode()).isEqualTo(HttpStatus.NOT_FOUND);
    assertThat(response.getBody().getMessage()).contains("999");
}

@Test
void shouldReturn400ForInvalidOrder() {
    Order invalidOrder = new Order(null, null);  // fails @Valid constraints

    ResponseEntity<ErrorResponse> response = restTemplate.postForEntity(
        "/orders",
        invalidOrder,
        ErrorResponse.class
    );

    assertThat(response.getStatusCode()).isEqualTo(HttpStatus.BAD_REQUEST);
}
```

## `@LocalServerPort` — when you need the absolute URL

`TestRestTemplate` handles relative paths automatically. Use `@LocalServerPort` when you need the actual port — for example, when building a URL for a client configured outside of `TestRestTemplate`:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest {

    @LocalServerPort
    private int port;

    @Test
    void withAbsoluteUrl() {
        String baseUrl = "http://localhost:" + port;
        // use when you need full URL for custom HTTP clients
    }
}
```

## Database state in `RANDOM_PORT` tests

Unlike `@DataJpaTest`, `@SpringBootTest` with `RANDOM_PORT` is **not** `@Transactional` by default. The HTTP request runs in its own transaction on the server side — you can't wrap the test in a transaction and roll it back, because the server's transaction commits before the test gets the response.

This means test data **persists** between tests unless you clean it up:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest {

    @Autowired
    private OrderRepository orderRepository;

    @BeforeEach
    void setUp() {
        orderRepository.deleteAll();  // clean state before each test
    }
}
```

Or use `@Sql` with `AFTER_TEST_METHOD` phase for cleanup:

```java
@Test
@Sql(scripts = "/cleanup.sql", executionPhase = Sql.ExecutionPhase.AFTER_TEST_METHOD)
void test() { ... }
```

⚠️ **EXAM TRAP**: `@SpringBootTest` with `RANDOM_PORT` does not roll back transactions automatically. Each HTTP request the test makes commits to the database. You must manage test data cleanup yourself.

## `RANDOM_PORT` vs `MOCK` — choosing between them

| Concern | Use `MOCK` + `MockMvc` | Use `RANDOM_PORT` + `TestRestTemplate` |
|---|---|---|
| Testing HTTP serialisation | ✅ | ✅ |
| Testing real network stack | ❌ | ✅ |
| Speed | Faster | Slower |
| Automatic tx rollback | Yes (if `@Transactional`) | No |
| Security filter testing | ✅ | ✅ |
| Testing real server behaviour | ❌ | ✅ |

For most controller tests, `MOCK` + `MockMvc` is sufficient and faster. Use `RANDOM_PORT` when you specifically need to test real HTTP behaviour — redirects, connection handling, full request/response cycle.

---

## Plain-English summary

- `RANDOM_PORT` starts a real server. `TestRestTemplate` is the client — pre-configured, no throws on 4xx/5xx.
- Use relative paths with `TestRestTemplate`. Use `@LocalServerPort` only when you need the actual port number.
- Combine with `@MockBean` to mock external dependencies while testing real HTTP + service + data layers.
- `RANDOM_PORT` tests don't roll back — HTTP requests commit their transactions. Clean up with `@BeforeEach` or `@Sql`.
- `TestRestTemplate` with `.withBasicAuth()` for authenticated requests.
