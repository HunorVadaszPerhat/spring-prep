# Objective 4.1.1 — JUnit 5 + Mockito

## The testing stack

`spring-boot-starter-test` bundles everything you need for testing Spring Boot applications. When you add it to your project, you get:

- **JUnit 5** (Jupiter) — the test framework
- **Mockito** — mocking library
- **AssertJ** — fluent assertions
- **Hamcrest** — matcher library
- **JSONassert** — JSON assertion library
- **JsonPath** — JSON path expressions in assertions

This single starter covers the vast majority of testing needs. The exam expects you to know what it includes.

---

## JUnit 5 essentials

JUnit 5 is composed of three modules — JUnit Platform (runner), JUnit Jupiter (API you write tests against), and JUnit Vintage (runs JUnit 4 tests). The exam tests Jupiter.

```java
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.AfterEach;
import org.junit.jupiter.api.BeforeAll;
import org.junit.jupiter.api.AfterAll;

class OrderServiceTest {

    @BeforeAll  // runs once before all tests in the class — must be static
    static void initAll() { }

    @BeforeEach  // runs before each test method
    void init() { }

    @Test
    void shouldPlaceOrder() {
        // test body
    }

    @AfterEach  // runs after each test method
    void tearDown() { }

    @AfterAll   // runs once after all tests — must be static
    static void tearDownAll() { }
}
```

⚠️ **EXAM TRAP**: `@BeforeAll` and `@AfterAll` methods must be `static` by default in JUnit 5 — they run once for the class, not per instance. If you need non-static `@BeforeAll`, annotate the class with `@TestInstance(Lifecycle.PER_CLASS)`.

## Mockito with `@ExtendWith(MockitoExtension.class)`

JUnit 5 uses **extensions** instead of JUnit 4's `@RunWith`. To activate Mockito's annotation processing in JUnit 5:

```java
@ExtendWith(MockitoExtension.class)  // activates Mockito annotations
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository;  // Mockito creates a mock

    @Mock
    private PaymentGateway paymentGateway;

    @InjectMocks
    private OrderService orderService;  // Mockito creates this and injects mocks into it

    @Test
    void shouldPlaceOrder() {
        Order order = new Order("customer-1", BigDecimal.TEN);

        // stubbing — define what the mock returns
        when(orderRepository.save(any(Order.class))).thenReturn(order);
        when(paymentGateway.charge(any())).thenReturn(true);

        // act
        orderService.placeOrder(order);

        // verify — assert the mock was called as expected
        verify(orderRepository).save(order);
        verify(paymentGateway).charge(order);
    }
}
```

---

## `@Mock` vs `@InjectMocks`

**`@Mock`** — creates a Mockito mock of the annotated type. The mock has no real implementation — all methods return default values (`null`, `0`, `false`, empty collections) unless stubbed.

**`@InjectMocks`** — creates a real instance of the annotated type and injects all `@Mock` fields into it. Mockito tries constructor injection first, then setter injection, then field injection.

⚠️ **EXAM TRAP**: `@Mock` creates a mock with no Spring context involved. `@InjectMocks` creates the real class under test. Neither of these annotations interacts with the Spring container at all — this is pure Mockito, no Spring context started.

## Stubbing — `when().thenReturn()`

```java
// Return a value
when(orderRepository.findById(1L)).thenReturn(Optional.of(order));

// Return different values on successive calls
when(orderRepository.findById(anyLong()))
    .thenReturn(Optional.of(order))   // first call
    .thenReturn(Optional.empty());    // second call

// Throw an exception
when(orderRepository.findById(1L)).thenThrow(new RuntimeException("DB error"));

// Stub void methods
doNothing().when(orderRepository).delete(any());
doThrow(new RuntimeException()).when(orderRepository).delete(any());
```

**Argument matchers:**
```java
when(repo.findById(1L))           // exact value — matches only 1L
when(repo.findById(anyLong()))     // any long value
when(repo.save(any(Order.class))) // any Order instance
when(repo.findByName(eq("Alice"))) // explicit equality matcher
```

⚠️ **EXAM TRAP**: You cannot mix exact values and matchers in the same method call. If you use any matcher, all arguments must use matchers:

```java
// WRONG — mixing exact value and matcher
when(repo.findByNameAndStatus("Alice", any())).thenReturn(...);

// CORRECT — all matchers
when(repo.findByNameAndStatus(eq("Alice"), any())).thenReturn(...);
```

## Verification — `verify()`

```java
// Verify a method was called exactly once (default)
verify(orderRepository).save(order);

// Verify call count
verify(orderRepository, times(2)).save(any());
verify(orderRepository, never()).delete(any());
verify(orderRepository, atLeastOnce()).findById(anyLong());
verify(orderRepository, atMost(3)).findById(anyLong());

// Verify no other interactions happened
verifyNoMoreInteractions(orderRepository);

// Verify zero interactions at all
verifyNoInteractions(paymentGateway);
```

## `@Captor` — capturing arguments

When you want to assert on the exact argument passed to a mock:

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository;

    @Captor
    private ArgumentCaptor<Order> orderCaptor;

    @InjectMocks
    private OrderService orderService;

    @Test
    void shouldSaveOrderWithCorrectStatus() {
        orderService.placeOrder(new Order("customer-1", BigDecimal.TEN));

        verify(orderRepository).save(orderCaptor.capture());
        Order savedOrder = orderCaptor.getValue();

        assertThat(savedOrder.getStatus()).isEqualTo("PENDING");
        assertThat(savedOrder.getCustomerName()).isEqualTo("customer-1");
    }
}
```

## `@Spy` — partial mocks

`@Spy` creates a real object but allows individual methods to be stubbed. Unstubbed methods call the real implementation.

```java
@Spy
private OrderValidator orderValidator = new OrderValidator(); // real object

@Test
void test() {
    // real method called for unstubbed methods
    doReturn(false).when(orderValidator).isValid(any()); // stub just this one
}
```

Use sparingly — if you need a spy, it often signals a design problem.

## AssertJ — fluent assertions

`spring-boot-starter-test` includes AssertJ. The exam expects you to recognise the fluent style:

```java
import static org.assertj.core.api.Assertions.*;

assertThat(order).isNotNull();
assertThat(order.getStatus()).isEqualTo("PENDING");
assertThat(orders).hasSize(3).contains(order);
assertThat(order.getTotal()).isGreaterThan(BigDecimal.ZERO);

// Exception assertion
assertThatThrownBy(() -> orderService.placeOrder(null))
    .isInstanceOf(IllegalArgumentException.class)
    .hasMessage("Order cannot be null");
```

## Plain-English summary

- `spring-boot-starter-test` bundles JUnit 5, Mockito, AssertJ, Hamcrest, JSONassert, JsonPath.
- `@ExtendWith(MockitoExtension.class)` activates Mockito annotations in JUnit 5.
- `@Mock` — Mockito mock, no Spring context. `@InjectMocks` — real instance with mocks injected.
- `@BeforeAll` / `@AfterAll` must be `static` by default.
- `when().thenReturn()` for stubbing. `verify()` for interaction assertions.
- Can't mix exact values and matchers in the same stubbing call — use `eq()` for equality matching.
- `@Captor` captures arguments passed to mocks for assertion.
- No Spring context is started with pure Mockito tests — fast, isolated unit tests.
