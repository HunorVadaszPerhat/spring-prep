# Objective 5.2 — Authentication and Authorization Configuration

## The two APIs — exam context

This objective requires holding two APIs in your head simultaneously. The exam was written against Spring Boot 2.5 / Spring Security 5.5, where `WebSecurityConfigurerAdapter` was the standard approach. That class was deprecated in Spring Security 5.7 (Boot 2.7) and removed in Spring Security 6 (Boot 3.x).

📝 **EXAM NOTE**: Expect questions using `WebSecurityConfigurerAdapter`, `antMatchers()`, and `@EnableGlobalMethodSecurity`. Do not modernise these in your head — answer based on what the code shows.

✅ **Modern approach**: `SecurityFilterChain` bean, `requestMatchers()`, `@EnableMethodSecurity`.

## The exam approach — `WebSecurityConfigurerAdapter`

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/public/**").permitAll()
                .antMatchers("/admin/**").hasRole("ADMIN")
                .antMatchers(HttpMethod.GET, "/orders/**").hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated()
                .and()
            .formLogin()
                .loginPage("/login")
                .permitAll()
                .and()
            .httpBasic()
                .and()
            .logout()
                .permitAll();
    }

    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        auth.inMemoryAuthentication()
            .withUser("user")
                .password("{noop}password")  // {noop} = no password encoding
                .roles("USER")
                .and()
            .withUser("admin")
                .password("{noop}admin")
                .roles("ADMIN");
    }
}
```

Two `configure()` overloads:
- `configure(HttpSecurity http)` — HTTP security rules: which URLs require what roles, login/logout config
- `configure(AuthenticationManagerBuilder auth)` — authentication source: where to look up users

⚠️ **EXAM TRAP**: `antMatchers()` rules are evaluated **in order** — the first matching rule wins. More specific rules must come before more general ones. `anyRequest().authenticated()` must always be last — it's the catch-all.

```java
// CORRECT — specific before general
.antMatchers("/admin/**").hasRole("ADMIN")
.antMatchers("/public/**").permitAll()
.anyRequest().authenticated()

// WRONG — anyRequest() catches everything, rules below never reached
.anyRequest().authenticated()
.antMatchers("/public/**").permitAll()  // never evaluated
```

## The modern approach — `SecurityFilterChain` bean

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers(HttpMethod.GET, "/orders/**").hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults())
            .httpBasic(Customizer.withDefaults());

        return http.build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        UserDetails user = User.withUsername("user")
            .password("{noop}password")
            .roles("USER")
            .build();

        UserDetails admin = User.withUsername("admin")
            .password("{noop}admin")
            .roles("ADMIN")
            .build();

        return new InMemoryUserDetailsManager(user, admin);
    }
}
```

Key differences from the exam approach:
- No class inheritance — just a `@Bean` method returning `SecurityFilterChain`
- `authorizeHttpRequests()` instead of `authorizeRequests()`
- `requestMatchers()` instead of `antMatchers()`
- `UserDetailsService` declared as a `@Bean` instead of configured via `AuthenticationManagerBuilder`

## URL matcher patterns

Both `antMatchers()` and `requestMatchers()` accept Ant-style patterns:

```java
"/orders"        // exact match — only /orders
"/orders/"       // exact match with trailing slash
"/orders/*"      // one path segment — /orders/123 but NOT /orders/123/items
"/orders/**"     // any depth — /orders/123, /orders/123/items, etc.
"/orders/?"      // single character wildcard
```

⚠️ **EXAM TRAP**: `/orders/*` matches one additional path segment only. `/orders/**` matches any depth. The exam tests this distinction.

## HTTP Basic vs Form Login

**HTTP Basic** — credentials sent in the `Authorization: Basic base64(user:password)` header on every request. Stateless. No login page. Used for API authentication.

```java
// Exam style
.httpBasic()

// Modern style
.httpBasic(Customizer.withDefaults())
```

**Form Login** — browser-based login form. Spring Security redirects unauthenticated requests to a login page. Session-based.

```java
// Exam style — custom login page
.formLogin()
    .loginPage("/login")
    .loginProcessingUrl("/authenticate")  // form POST target
    .defaultSuccessUrl("/dashboard")
    .failureUrl("/login?error")
    .permitAll()

// Modern style
.formLogin(Customizer.withDefaults())  // uses Spring Security's default login page
```

## Common authorisation expressions

```java
.permitAll()                           // everyone — authenticated or not
.denyAll()                             // nobody
.authenticated()                       // any authenticated user
.anonymous()                           // unauthenticated users only
.hasRole("ADMIN")                      // ROLE_ADMIN authority
.hasAnyRole("USER", "ADMIN")           // ROLE_USER or ROLE_ADMIN
.hasAuthority("ROLE_ADMIN")            // exact authority string
.hasAnyAuthority("ROLE_USER", "ROLE_ADMIN")
.hasIpAddress("192.168.1.0/24")        // IP range
.access("hasRole('ADMIN') and hasIpAddress('192.168.1.0/24')")  // SpEL expression
```

## Password encoding

Spring Security requires passwords to be encoded. The `{noop}` prefix tells Spring to use no encoding — only for testing, never production.

```java
// Production — BCrypt encoding
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

// In-memory with BCrypt
UserDetails user = User.withUsername("user")
    .password(passwordEncoder().encode("password"))
    .roles("USER")
    .build();

// {noop} in tests/examples only
.password("{noop}password")
```

⚠️ **EXAM TRAP**: Without a `PasswordEncoder` bean, Spring Security throws `IllegalArgumentException` when it tries to match passwords — unless you use `{noop}`. The encoder is required in production config.

## CSRF protection

Spring Security enables CSRF protection by default for state-changing requests (POST, PUT, DELETE). For REST APIs that use stateless authentication (tokens), CSRF is typically disabled:

```java
// Exam style
http.csrf().disable()

// Modern style
http.csrf(csrf -> csrf.disable())
```

⚠️ **EXAM TRAP**: CSRF is enabled by default. POST requests from forms in tests will fail with 403 Forbidden unless CSRF is disabled or a CSRF token is included. This is a common source of test failures when adding Spring Security to an existing application.

## Stateless session management

For REST APIs, disable session creation — authentication is handled per-request via tokens:

```java
// Exam style
http.sessionManagement()
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS)

// Modern style
http.sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
```

## Custom `UserDetailsService`

For database-backed authentication:

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {

    @Autowired
    private UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String username)
            throws UsernameNotFoundException {
        User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException(
                "User not found: " + username));

        return org.springframework.security.core.userdetails.User
            .withUsername(user.getUsername())
            .password(user.getPassword())  // already encoded in DB
            .roles(user.getRole())
            .build();
    }
}
```

Spring Security auto-detects a `UserDetailsService` bean and uses it for authentication — no explicit wiring needed.

## Version reference — exam vs modern

| Concept | Exam (Boot 2.5) | Modern (Boot 3.x) |
|---|---|---|
| Config class | `extends WebSecurityConfigurerAdapter` | `SecurityFilterChain @Bean` |
| URL matchers | `antMatchers()` | `requestMatchers()` |
| Auth rules method | `authorizeRequests()` | `authorizeHttpRequests()` |
| Method security | `@EnableGlobalMethodSecurity` | `@EnableMethodSecurity` |
| In-memory users | `auth.inMemoryAuthentication()` | `InMemoryUserDetailsManager @Bean` |

## Plain-English summary

- Exam uses `WebSecurityConfigurerAdapter` with `configure(HttpSecurity)` and `configure(AuthenticationManagerBuilder)`.
- Modern uses `SecurityFilterChain @Bean` and `UserDetailsService @Bean` — no inheritance.
- `antMatchers()` (exam) / `requestMatchers()` (modern) — Ant patterns, evaluated in order, specific before general.
- `anyRequest().authenticated()` must be last — it's the catch-all.
- `hasRole("X")` adds `ROLE_` prefix. `hasAuthority("X")` uses exact string.
- HTTP Basic = stateless, no login page. Form Login = session-based, login page.
- `{noop}` prefix = no password encoding — tests only.
- CSRF enabled by default — disable for stateless REST APIs.
- `PasswordEncoder` bean required for production password matching.
