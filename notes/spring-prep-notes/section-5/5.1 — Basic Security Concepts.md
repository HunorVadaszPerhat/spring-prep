# Objective 5.1 — Basic Security Concepts

## Authentication vs Authorisation

These two terms are frequently conflated. The exam tests whether you can distinguish them precisely.

**Authentication** — establishing *who* you are. Verifying identity. "Is this person who they claim to be?" The result is a confirmed identity — a `Principal`.

**Authorisation** — determining *what* you are allowed to do. "Does this authenticated person have permission to perform this action?" The result is access granted or denied.

The sequence is always: authenticate first, then authorise. You can't decide what someone is allowed to do until you know who they are.

```
Request arrives
    ↓
Authentication — verify identity (username/password, token, certificate)
    ↓
Principal established — "this is Alice"
    ↓
Authorisation — does Alice have ROLE_ADMIN to access /admin/orders?
    ↓
Access granted or denied
```

## Core abstractions

### `Principal`

Represents the currently authenticated user. In Java security, `Principal` is a standard interface (`java.security.Principal`) with one method: `getName()`.

In Spring Security, the authenticated user is typically represented as a `UsernamePasswordAuthenticationToken` which implements both `Authentication` and `Principal`.

```java
@GetMapping("/profile")
public String getProfile(Principal principal) {
    return "Hello, " + principal.getName();  // Spring injects the current user
}
```

Spring MVC automatically resolves `Principal` as a method parameter by pulling it from the security context.

---

### `GrantedAuthority`

Represents a permission or role granted to a user. Spring Security uses `GrantedAuthority` to represent both roles and fine-grained permissions.

```java
// GrantedAuthority is typically a role string
// Roles conventionally prefixed with "ROLE_"
new SimpleGrantedAuthority("ROLE_ADMIN")
new SimpleGrantedAuthority("ROLE_USER")
new SimpleGrantedAuthority("ORDER_READ")   // fine-grained permission
```

⚠️ **EXAM TRAP**: Roles in Spring Security are `GrantedAuthority` instances with the `ROLE_` prefix by convention. When you use `hasRole("ADMIN")` in security config, Spring automatically prepends `ROLE_` — so it matches `ROLE_ADMIN`. When you use `hasAuthority("ROLE_ADMIN")`, no prefix is added — you must provide the full string including `ROLE_`.

```java
// These are equivalent:
hasRole("ADMIN")           // matches GrantedAuthority "ROLE_ADMIN"
hasAuthority("ROLE_ADMIN") // matches GrantedAuthority "ROLE_ADMIN"

// These are NOT equivalent:
hasRole("ROLE_ADMIN")      // looks for "ROLE_ROLE_ADMIN" — double prefix, bug!
hasAuthority("ADMIN")      // looks for "ADMIN" — no ROLE_ prefix, won't match role
```

---

### `Authentication`

Represents the authentication token — holds the principal, credentials, and granted authorities. Extends `Principal`.

Key implementations:
- `UsernamePasswordAuthenticationToken` — username/password authentication
- `AnonymousAuthenticationToken` — unauthenticated users (Spring Security gives them an anonymous token)

```java
Authentication auth = SecurityContextHolder.getContext().getAuthentication();

String username = auth.getName();                    // principal name
Object credentials = auth.getCredentials();          // password (often cleared after auth)
Collection<? extends GrantedAuthority> authorities = auth.getAuthorities(); // roles/permissions
boolean authenticated = auth.isAuthenticated();
```

---

### `SecurityContext`

Holds the `Authentication` object for the current user. One `SecurityContext` per thread.

```java
SecurityContext context = SecurityContextHolder.getContext();
Authentication auth = context.getAuthentication();
```

---

### `SecurityContextHolder`

The static gateway to the current thread's `SecurityContext`. Spring Security stores the authenticated user here after successful authentication, and the framework reads from here for authorisation decisions.

```java
// Get current authentication
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String currentUser = auth.getName();

// Clear security context (logout)
SecurityContextHolder.clearContext();
```

**Storage strategy** — `SecurityContextHolder` uses a strategy to store the context. The default is `ThreadLocal` — each thread gets its own security context. This is why the security context is available anywhere in the same thread without passing it around explicitly.

⚠️ **EXAM TRAP**: The default `SecurityContextHolder` strategy is `MODE_THREADLOCAL`. In async scenarios where work moves to different threads, the security context is not automatically propagated. Know that this limitation exists — `MODE_INHERITABLETHREADLOCAL` propagates to child threads.

## How Spring Security plugs into the request lifecycle

Spring Security is implemented as a chain of servlet `Filter`s — the `SecurityFilterChain`. It intercepts requests before they reach the `DispatcherServlet`.

```
HTTP Request
    ↓
SecurityFilterChain (filters execute in order)
    ├── UsernamePasswordAuthenticationFilter — handles form login
    ├── BasicAuthenticationFilter — handles HTTP Basic
    ├── BearerTokenAuthenticationFilter — handles JWT/OAuth2
    ├── ExceptionTranslationFilter — converts security exceptions to HTTP responses
    └── FilterSecurityInterceptor — makes the final authorisation decision
    ↓
DispatcherServlet → Controller
```

The key point: security happens **before** the request reaches your controller. If authentication or authorisation fails, the request never gets to `DispatcherServlet`.

## `UserDetails` and `UserDetailsService`

`UserDetails` represents the user information loaded from a data source (database, in-memory, LDAP):

```java
public interface UserDetails {
    String getUsername();
    String getPassword();
    Collection<? extends GrantedAuthority> getAuthorities();
    boolean isAccountNonExpired();
    boolean isAccountNonLocked();
    boolean isCredentialsNonExpired();
    boolean isEnabled();
}
```

`UserDetailsService` loads `UserDetails` by username — this is what Spring Security calls during authentication:

```java
public interface UserDetailsService {
    UserDetails loadUserByUsername(String username) throws UsernameNotFoundException;
}
```

Spring Security calls `loadUserByUsername()`, gets the `UserDetails`, verifies the password, and if successful, creates an `Authentication` token and stores it in the `SecurityContextHolder`.

## Plain-English summary

- **Authentication** = who are you? Produces a `Principal`.
- **Authorisation** = what can you do? Uses `GrantedAuthority`.
- `Principal` — the authenticated user's identity. Available as a controller method parameter.
- `GrantedAuthority` — a role or permission. Roles prefixed with `ROLE_` by convention.
- `hasRole("X")` adds `ROLE_` prefix automatically. `hasAuthority("X")` uses exact string.
- `Authentication` — holds principal, credentials, and authorities.
- `SecurityContext` — holds the `Authentication` for the current thread.
- `SecurityContextHolder` — static access to the current thread's `SecurityContext`. Default strategy is `ThreadLocal`.
- Spring Security filters run before `DispatcherServlet` — security is enforced before your controller sees the request.
