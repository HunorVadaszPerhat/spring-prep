# Objective 5.3 — Method-Level Security

## What method-level security adds

URL-based security in `SecurityFilterChain` protects endpoints at the HTTP level. Method-level security adds a second layer — it protects individual service methods regardless of how they're called. A method can be secured even if called from another service, a scheduled task, or a message listener — not just from an HTTP request.

---

## Enabling method security

📝 **EXAM NOTE**: Two annotations exist — the old one the exam tests and the modern replacement.

**Exam style (Boot 2.5 / Spring Security 5.x):**
```java
@Configuration
@EnableGlobalMethodSecurity(
    prePostEnabled = true,   // enables @PreAuthorize and @PostAuthorize
    securedEnabled = true,   // enables @Secured
    jsr250Enabled = true     // enables @RolesAllowed (JSR-250)
)
public class SecurityConfig extends WebSecurityConfigurerAdapter { }
```

Each feature is disabled by default — you must explicitly enable the ones you use.

**Modern style (Boot 3.x / Spring Security 6.x):**
```java
@Configuration
@EnableMethodSecurity(
    // prePostEnabled = true by default in modern — no need to set it
    securedEnabled = true,   // still opt-in
    jsr250Enabled = true     // still opt-in
)
public class SecurityConfig { }
```

⚠️ **EXAM TRAP**: With `@EnableGlobalMethodSecurity`, `prePostEnabled` is **false by default** — you must explicitly set it to `true`. Forgetting this means `@PreAuthorize` and `@PostAuthorize` annotations are silently ignored. This is a direct exam question.

---

## `@PreAuthorize` — check before method executes

Evaluates a SpEL expression before the method runs. If the expression returns false, `AccessDeniedException` is thrown and the method never executes.

```java
@Service
public class OrderService {

    // Simple role check
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(Long id) {
        orderRepository.deleteById(id);
    }

    // Multiple roles
    @PreAuthorize("hasAnyRole('USER', 'ADMIN')")
    public List<Order> findAllOrders() {
        return orderRepository.findAll();
    }

    // Using method parameter in expression — # prefix accesses parameter by name
    @PreAuthorize("hasRole('ADMIN') or #customerId == authentication.principal.username")
    public List<Order> findOrdersByCustomer(String customerId) {
        return orderRepository.findByCustomerId(customerId);
    }

    // Checking authority (exact string, no ROLE_ prefix added)
    @PreAuthorize("hasAuthority('ORDER_DELETE')")
    public void cancelOrder(Long id) { ... }
}
```

The SpEL expressions have access to:
- `authentication` — the current `Authentication` object
- `principal` — the current `Principal`
- `#paramName` — method parameters by name

⚠️ **EXAM TRAP**: `#paramName` syntax accesses method parameters by name. This requires the parameter names to be available at runtime — either via `-parameters` compiler flag or debug information. In Spring Boot, this works out of the box.

---

## `@PostAuthorize` — check after method executes

Evaluates after the method returns. Used when the authorisation decision depends on the return value. If the expression returns false, `AccessDeniedException` is thrown after the method has already executed — the return value is discarded.

```java
@Service
public class OrderService {

    // Check return value — returnObject is the special variable
    @PostAuthorize("returnObject.customerName == authentication.principal.username")
    public Order findById(Long id) {
        return orderRepository.findById(id).orElseThrow();
        // if the returned order doesn't belong to the current user → AccessDeniedException
    }
}
```

`returnObject` is the special SpEL variable that holds the method's return value.

⚠️ **EXAM TRAP**: `@PostAuthorize` runs the method first, then checks. If the method has side effects (writes to DB, sends emails), those side effects have already happened when the exception is thrown. Use `@PreAuthorize` when you want to prevent execution entirely.

---

## `@Secured` — simple role-based security

Simpler than `@PreAuthorize` — takes a list of role strings. No SpEL. Requires `securedEnabled = true`.

```java
@Secured("ROLE_ADMIN")               // single role — note: full string including ROLE_
public void deleteOrder(Long id) { ... }

@Secured({"ROLE_USER", "ROLE_ADMIN"}) // either role — OR logic
public List<Order> findAllOrders() { ... }
```

⚠️ **EXAM TRAP**: `@Secured` requires the full role string including `ROLE_` prefix — it does NOT add the prefix automatically like `hasRole()`. This is different from `@PreAuthorize("hasRole('ADMIN')")`.

---

## `@RolesAllowed` — JSR-250 standard annotation

The Java standard equivalent of `@Secured`. Requires `jsr250Enabled = true`. Same behaviour as `@Secured` but uses the JSR-250 standard annotation.

```java
import javax.annotation.security.RolesAllowed;  // Boot 2.x
// import jakarta.annotation.security.RolesAllowed;  // Boot 3.x

@RolesAllowed("ROLE_ADMIN")
public void deleteOrder(Long id) { ... }

@RolesAllowed({"ROLE_USER", "ROLE_ADMIN"})
public List<Order> findAllOrders() { ... }
```

Functionally identical to `@Secured`. The difference is source — JSR-250 standard vs Spring-specific.

---

## Where to put method security annotations

Method security works on any Spring-managed bean. Most commonly on `@Service` classes — securing the service layer means the security applies regardless of how the service is called:

```java
@Service
public class OrderService {
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(Long id) { ... }
}
```

You can also put annotations on `@Controller` methods — but this is less common and slightly redundant if you already have URL-level security. Service-level security is the safer choice.

⚠️ **EXAM TRAP**: Method security uses AOP proxies — all the proxy limitations from 1.6.2 apply. Self-invocation bypasses the proxy, so `@PreAuthorize` on a method called from within the same class is silently ignored.

```java
@Service
public class OrderService {

    public void processOrder(Order order) {
        deleteOrder(order.getId()); // self-invocation — bypasses proxy
                                    // @PreAuthorize on deleteOrder is ignored
    }

    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(Long id) { ... }
}
```

---

## Method security on interfaces vs implementations

When a class implements an interface, Spring Security recommends putting method security annotations on the **concrete class**, not the interface. The proxy wraps the concrete class — annotations on the interface may not be reliably picked up depending on the proxy type.

```java
// Preferred — annotate the implementation
@Service
public class OrderServiceImpl implements OrderService {

    @Override
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(Long id) { ... }
}
```

## Handling `AccessDeniedException`

When a security check fails, Spring Security throws `AccessDeniedException`. In a web context, this is caught by Spring Security's `ExceptionTranslationFilter` and translated to:
- **401 Unauthorized** — if the user is not authenticated
- **403 Forbidden** — if the user is authenticated but lacks permission

You can also handle it in `@RestControllerAdvice`:

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AccessDeniedException.class)
    @ResponseStatus(HttpStatus.FORBIDDEN)
    public ErrorResponse handleAccessDenied(AccessDeniedException ex) {
        return new ErrorResponse("ACCESS_DENIED", ex.getMessage());
    }
}
```

📝 **EXAM NOTE**: Be cautious — if you handle `AccessDeniedException` in `@RestControllerAdvice`, it may bypass Spring Security's standard handling. Generally it's better to let Spring Security handle it via `ExceptionTranslationFilter`.

## Comparison of method security annotations

| Annotation | Fires | SpEL | Role prefix | Requires enabling |
|---|---|---|---|---|
| `@PreAuthorize` | Before method | ✅ | None added — use `hasRole()` for auto-prefix | `prePostEnabled = true` |
| `@PostAuthorize` | After method | ✅ | None added | `prePostEnabled = true` |
| `@Secured` | Before method | ❌ | Must include `ROLE_` manually | `securedEnabled = true` |
| `@RolesAllowed` | Before method | ❌ | Must include `ROLE_` manually | `jsr250Enabled = true` |

---

## Plain-English summary

- `@EnableGlobalMethodSecurity(prePostEnabled = true)` (exam) / `@EnableMethodSecurity` (modern) activates method security.
- `prePostEnabled` is false by default in the exam API — must explicitly enable it.
- `@PreAuthorize` — SpEL before method. Access method params with `#paramName`. Access return with `returnObject` (in `@PostAuthorize`).
- `@PostAuthorize` — SpEL after method. Method already executed when check fires — side effects already happened.
- `@Secured` — simple role list, no SpEL, requires full `ROLE_` prefix manually.
- `@RolesAllowed` — JSR-250 equivalent of `@Secured`.
- Proxy rules apply — self-invocation bypasses `@PreAuthorize` silently.
- Failed check → `AccessDeniedException` → 401 (unauthenticated) or 403 (unauthorised).
