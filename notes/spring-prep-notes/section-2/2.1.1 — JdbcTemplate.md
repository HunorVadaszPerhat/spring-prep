# Objective 2.1.1 — JdbcTemplate

## What problem JdbcTemplate solves

Raw JDBC is verbose and error-prone. Every query requires the same boilerplate: get a connection, create a statement, execute, iterate the ResultSet, handle checked exceptions, close everything in a finally block. Miss any step and you leak resources or swallow errors.

`JdbcTemplate` handles all of that infrastructure. You provide the SQL and the logic for mapping results — it handles connections, statement creation, execution, exception translation, and cleanup. Every resource is closed automatically, even if an exception is thrown.

---

## Setup

With Spring Boot, `JdbcTemplate` is auto-configured when `spring-boot-starter-jdbc` (or `spring-boot-starter-data-jpa`) is on the classpath and a `DataSource` bean exists. You just inject it:

```java
@Repository
public class OrderRepository {

    private final JdbcTemplate jdbcTemplate;

    public OrderRepository(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }
}
```

Without Boot, declare it as a bean:
```java
@Bean
public JdbcTemplate jdbcTemplate(DataSource dataSource) {
    return new JdbcTemplate(dataSource);
}
```

## The four core methods

### `query()` — returns multiple rows

```java
public List<Order> findAll() {
    return jdbcTemplate.query(
        "SELECT id, customer_name, total FROM orders",
        (rs, rowNum) -> new Order(
            rs.getLong("id"),
            rs.getString("customer_name"),
            rs.getBigDecimal("total")
        )
    );
}
```

The second argument is a `RowMapper<T>` — a lambda that converts one `ResultSet` row into one object. `JdbcTemplate` calls it once per row and collects the results into a `List<T>`.

With named parameters (requires `NamedParameterJdbcTemplate`):
```java
// Standard JdbcTemplate uses positional ? parameters
List<Order> orders = jdbcTemplate.query(
    "SELECT * FROM orders WHERE status = ? AND customer_id = ?",
    (rs, rowNum) -> mapOrder(rs),
    "PENDING", customerId  // varargs — positional, left to right
);
```

---

### `queryForObject()` — returns exactly one row

```java
public Order findById(Long id) {
    return jdbcTemplate.queryForObject(
        "SELECT id, customer_name, total FROM orders WHERE id = ?",
        (rs, rowNum) -> new Order(
            rs.getLong("id"),
            rs.getString("customer_name"),
            rs.getBigDecimal("total")
        ),
        id  // parameter value
    );
}
```

For scalar values (count, single column):
```java
int count = jdbcTemplate.queryForObject(
    "SELECT COUNT(*) FROM orders WHERE status = ?",
    Integer.class,  // expected return type
    "PENDING"
);
```

⚠️ **EXAM TRAP**: `queryForObject()` throws `EmptyResultDataAccessException` if zero rows are returned, and `IncorrectResultSizeDataAccessException` if more than one row is returned. It expects **exactly one row**. Use `query()` when the result might be empty.

---

### `update()` — INSERT, UPDATE, DELETE

```java
public int save(Order order) {
    return jdbcTemplate.update(
        "INSERT INTO orders (customer_name, total, status) VALUES (?, ?, ?)",
        order.getCustomerName(),
        order.getTotal(),
        order.getStatus()
    );
    // returns number of rows affected
}

public int updateStatus(Long id, String status) {
    return jdbcTemplate.update(
        "UPDATE orders SET status = ? WHERE id = ?",
        status, id
    );
}

public int delete(Long id) {
    return jdbcTemplate.update(
        "DELETE FROM orders WHERE id = ?",
        id
    );
}
```

`update()` returns an `int` — the number of rows affected. Use it for any write operation (INSERT, UPDATE, DELETE).

---

### `batchUpdate()` — multiple updates in one batch

```java
public int[] saveAll(List<Order> orders) {
    return jdbcTemplate.batchUpdate(
        "INSERT INTO orders (customer_name, total) VALUES (?, ?)",
        orders,
        orders.size(),
        (ps, order) -> {
            ps.setString(1, order.getCustomerName());
            ps.setBigDecimal(2, order.getTotal());
        }
    );
    // returns int[] — rows affected per statement
}
```

More efficient than calling `update()` in a loop — sends all statements to the database in one round trip.

## Exception translation

`JdbcTemplate` catches checked `SQLException` and re-throws it as a subclass of `DataAccessException` — Spring's unchecked exception hierarchy. This is the same exception translation that `@Repository` activates (covered in 1.4.4).

```
DataAccessException (unchecked)
├── EmptyResultDataAccessException    — queryForObject() found 0 rows
├── IncorrectResultSizeDataAccessException — queryForObject() found >1 row
├── DuplicateKeyException             — unique constraint violation
├── DataIntegrityViolationException   — foreign key, not-null violation
└── CannotGetJdbcConnectionException  — connection failure
```

You don't need to catch checked exceptions or handle connection cleanup — `JdbcTemplate` handles both.

---

## Plain-English summary

- `query()` — maps multiple rows to a list via `RowMapper`. Returns empty list if no rows.
- `queryForObject()` — maps exactly one row. Throws if zero or more than one row found.
- `update()` — INSERT, UPDATE, DELETE. Returns rows affected.
- `batchUpdate()` — multiple writes in one database round trip. Returns int array.
- Parameters are positional `?` — passed as varargs after the SQL and mapper.
- `JdbcTemplate` handles connection management, resource cleanup, and exception translation automatically.
