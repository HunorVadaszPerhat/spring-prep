# Objective 2.2.4 — Transactions in Tests

## The core behaviour

When you put `@Transactional` on a test method (or test class), Spring wraps the entire test in a transaction and **automatically rolls it back** after the test completes — whether it passed, failed, or threw an exception.

This means your test can write to the database, assert against that data, and leave the database in exactly the same state it started in. No cleanup code needed. No test ordering issues from leftover data.

```java
@SpringBootTest
@Transactional  // each test method runs in a transaction that rolls back after
class OrderRepositoryTest {

    @Autowired
    private OrderRepository orderRepository;

    @Test
    void shouldSaveOrder() {
        Order order = new Order("customer-1", BigDecimal.TEN);
        orderRepository.save(order);

        Order found = orderRepository.findById(order.getId()).orElseThrow();
        assertThat(found.getCustomerName()).isEqualTo("customer-1");

        // transaction rolls back here — order is NOT in the database after this test
    }
}
```

---

## Rollback is unconditional

Unlike application `@Transactional` where rollback depends on exception type, test rollback is unconditional. It doesn't matter if the test passes, fails, or throws a checked exception — the transaction always rolls back.

⚠️ **EXAM TRAP**: The rollback rules from 2.2.3 (checked vs unchecked, `rollbackFor`) do **not** apply to test transactions. Spring's test framework handles the rollback directly, bypassing the normal rollback rule evaluation.

---

## `@Commit` — override the automatic rollback

If you want a test transaction to commit instead of roll back, annotate the test method with `@Commit`:

```java
@SpringBootTest
@Transactional
class OrderRepositoryTest {

    @Test
    @Commit  // overrides automatic rollback — data persists after this test
    void shouldPersistOrder() {
        orderRepository.save(new Order("customer-1", BigDecimal.TEN));
        // transaction commits — use this only when you need to verify
        // data survives a real commit (e.g. testing DB constraints)
    }
}
```

Use `@Commit` sparingly — it leaves data in the database and can affect other tests.

Equivalent annotation: `@Rollback(false)` does the same thing:
```java
@Rollback(false)  // same as @Commit
void shouldPersistOrder() { ... }
```

`@Rollback(true)` is the explicit default — same as having no annotation.

---

## `@Rollback` at the class level

You can control rollback behaviour at the class level and override per method:

```java
@SpringBootTest
@Transactional
@Rollback(false)  // all tests in this class commit by default
class SchemaValidationTest {

    @Test
    void testConstraint() { ... }  // commits

    @Test
    @Rollback  // this one rolls back
    void testWithRollback() { ... }
}
```

## `@DataJpaTest` — the slice test for JPA

`@DataJpaTest` is a test slice annotation that loads only the JPA layer — entities, repositories, and an in-memory database. It does not load `@Service`, `@Controller`, or the full application context.

```java
@DataJpaTest  // loads JPA layer only, uses H2 in-memory by default
class OrderRepositoryTest {

    @Autowired
    private OrderRepository orderRepository;

    @Autowired
    private TestEntityManager entityManager; // helper for JPA test setup

    @Test
    void shouldFindByCustomerName() {
        entityManager.persist(new Order("customer-1", BigDecimal.TEN));
        entityManager.flush();

        List<Order> orders = orderRepository.findByCustomerName("customer-1");
        assertThat(orders).hasSize(1);
    }
}
```

`@DataJpaTest` is `@Transactional` by default — each test rolls back automatically. You don't need to add `@Transactional` yourself.

⚠️ **EXAM TRAP**: `@DataJpaTest` uses an **in-memory database** (H2) by default, replacing any real `DataSource` configured in `application.properties`. To use the real database in a `@DataJpaTest`, add `@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)`.

## `@Sql` — executing SQL before or after tests

`@Sql` runs SQL scripts at specific points in the test lifecycle — useful for setting up test data or cleaning up after `@Commit` tests.

```java
@SpringBootTest
@Transactional
class OrderServiceTest {

    @Test
    @Sql("/test-data/orders.sql")  // runs before the test method
    void shouldProcessExistingOrder() {
        // orders.sql has inserted test data
        Order order = orderService.findById(1L);
        assertThat(order).isNotNull();
    }

    @Test
    @Sql(
        scripts = "/test-data/orders.sql",
        executionPhase = Sql.ExecutionPhase.BEFORE_TEST_METHOD  // default
    )
    void withExplicitPhase() { ... }

    @Test
    @Sql(
        scripts = "/test-data/cleanup.sql",
        executionPhase = Sql.ExecutionPhase.AFTER_TEST_METHOD  // runs after
    )
    void withCleanup() { ... }
}
```

`@Sql` can be placed at the class level to apply to all test methods.

⚠️ **EXAM TRAP**: When `@Sql` is used with `@Transactional`, the SQL scripts run **within the same transaction** as the test by default. That means the inserted data is visible to the test but rolled back with it afterward. If you need the SQL to run outside the transaction, use `@Sql(config = @SqlConfig(transactionMode = SqlConfig.TransactionMode.ISOLATED))`.

## `@ActiveProfiles` in tests

Revisiting from 1.3.2 — use `@ActiveProfiles` to activate a specific profile for a test class. Common pattern: a `test` profile that points to an in-memory database or mock services.

```java
@SpringBootTest
@ActiveProfiles("test")  // activates application-test.properties
@Transactional
class OrderServiceIntegrationTest { ... }
```

---

## Transaction propagation in tests — a subtle point

When a test method is `@Transactional` and it calls a service method that is also `@Transactional(propagation = REQUIRED)`, the service method **joins the test's transaction**. This means the service method's work is included in the rollback.

But if the service method uses `REQUIRES_NEW`, it runs in a **separate** transaction that commits independently — that data **won't** be rolled back by the test transaction:

```java
@SpringBootTest
@Transactional
class AuditTest {

    @Test
    void auditLogShouldPersistEvenOnRollback() {
        orderService.placeOrder(order); 
        // placeOrder calls saveAuditLog() with REQUIRES_NEW
        // that audit log commits independently and is NOT rolled back
        // it remains in the database after the test
    }
}
```

⚠️ **EXAM TRAP**: `REQUIRES_NEW` in a service method called from a `@Transactional` test will commit its data even though the test rolls back. This can leave data in the database after tests that use `REQUIRES_NEW` internally.

---

## Plain-English summary

- `@Transactional` on a test = automatic rollback after each test. Unconditional — no rollback rules apply.
- `@Commit` or `@Rollback(false)` overrides automatic rollback. Use sparingly.
- `@DataJpaTest` loads JPA layer only, uses H2 in-memory, is `@Transactional` by default.
- `@Sql` runs SQL scripts before or after test methods. Runs inside the test transaction by default.
- `REQUIRES_NEW` in called service methods commits independently — that data survives the test rollback.
- `@ActiveProfiles("test")` activates test-specific configuration.
