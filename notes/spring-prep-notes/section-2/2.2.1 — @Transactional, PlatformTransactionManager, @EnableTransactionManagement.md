# Objective 2.2.1 — `@Transactional`, `PlatformTransactionManager`, `@EnableTransactionManagement`

## What transaction management means

A transaction is a unit of work that either completes entirely or not at all. If you're updating three tables and the third one fails, you want the first two rolled back — not left in a half-updated state. Spring's transaction management handles the open/commit/rollback lifecycle so your business logic doesn't have to.

---

## `PlatformTransactionManager`

The core Spring abstraction for transaction management. Spring provides implementations for different technologies:

| Implementation | Use with |
|---|---|
| `DataSourceTransactionManager` | Plain JDBC / `JdbcTemplate` |
| `JpaTransactionManager` | JPA / Hibernate |
| `JtaTransactionManager` | Distributed transactions (XA) |

With Spring Boot, the right implementation is auto-configured based on what's on the classpath. If you have JPA, you get `JpaTransactionManager`. If you have only JDBC, you get `DataSourceTransactionManager`. You rarely declare this manually.

---

## `@EnableTransactionManagement`

Activates Spring's annotation-driven transaction infrastructure — specifically, it registers a `BeanPostProcessor` that wraps `@Transactional` beans in a proxy.

```java
@Configuration
@EnableTransactionManagement  // activates @Transactional processing
public class AppConfig {

    @Bean
    public PlatformTransactionManager transactionManager(DataSource dataSource) {
        return new DataSourceTransactionManager(dataSource);
    }
}
```

With Spring Boot, `@EnableTransactionManagement` is auto-configured — you don't need to declare it. But you need to know it exists and what it does for the exam.

---

## `@Transactional`

Declares that a method (or all methods in a class) should run inside a transaction. Spring's proxy intercepts the call, begins a transaction, invokes your method, then commits or rolls back.

```java
@Service
public class OrderService {

    @Transactional  // default settings — REQUIRED propagation, rolls back on RuntimeException
    public void placeOrder(Order order) {
        orderRepository.save(order);
        inventoryService.reserve(order.getItems());
        paymentService.charge(order.getPayment());
        // if any of these throw a RuntimeException, everything rolls back
    }
}
```

`@Transactional` on the class level applies to all public methods in the class:

```java
@Service
@Transactional  // all public methods are transactional by default
public class OrderService {

    public void placeOrder(Order order) { ... }  // transactional

    @Transactional(readOnly = true)  // method-level overrides class-level
    public Order findOrder(Long id) { ... }
}
```

---

## `@Transactional` attributes

```java
@Transactional(
    propagation = Propagation.REQUIRED,      // default
    isolation = Isolation.DEFAULT,           // default — uses DB default
    readOnly = false,                        // default
    timeout = -1,                            // default — no timeout
    rollbackFor = Exception.class,           // roll back on this exception type
    noRollbackFor = OptimisticLockException.class  // don't roll back on this
)
```

**`readOnly = true`** — hints to the persistence provider and database that no writes will occur. Can enable optimisations (flush skipping in JPA, read replicas). Doesn't enforce read-only at the Java level — it's a hint, not a constraint.

**`timeout`** — seconds before the transaction is automatically rolled back. `-1` means no limit.

---

## Declarative vs programmatic transactions

**Declarative** (the `@Transactional` approach above) — Spring handles open/commit/rollback via proxy. Your code has no transaction management boilerplate. This is the standard approach.

**Programmatic** — you control the transaction explicitly using `TransactionTemplate`:

```java
@Service
public class OrderService {

    private final TransactionTemplate transactionTemplate;

    public OrderService(PlatformTransactionManager transactionManager) {
        this.transactionTemplate = new TransactionTemplate(transactionManager);
    }

    public void placeOrder(Order order) {
        transactionTemplate.execute(status -> {
            orderRepository.save(order);
            inventoryService.reserve(order.getItems());
            return null;
            // throw any exception to trigger rollback
            // or call status.setRollbackOnly() to roll back without throwing
        });
    }
}
```

**When to use programmatic**: when you need fine-grained control over transaction boundaries within a single method — for example, committing part of the work and then starting a new transaction. The exam knows both approaches exist; declarative is the primary focus.

---

## The proxy mechanics — same rules as AOP

`@Transactional` is implemented as an AOP aspect. All the proxy limitations from 1.5.3 and 1.6.2 apply:

⚠️ **EXAM TRAP**: `@Transactional` on a **private method** is silently ignored — the proxy can't intercept it.

⚠️ **EXAM TRAP**: **Self-invocation** — a method calling another `@Transactional` method in the same class bypasses the proxy. The called method's transaction settings are ignored.

⚠️ **EXAM TRAP**: `@Transactional` only works on **Spring-managed beans**. Objects created with `new` don't have proxies.

---

## Where to put `@Transactional` — service vs repository

The exam-standard answer: put `@Transactional` on the **service layer**, not the repository layer. The service method represents a business unit of work that may span multiple repository calls — you want one transaction wrapping all of them.

```java
@Service
public class OrderService {

    @Transactional  // one transaction covers both repository calls
    public void transferOrder(Long fromId, Long toId) {
        Order order = orderRepository.findById(fromId);
        orderRepository.delete(fromId);
        orderRepository.save(order.withNewId(toId));
    }
}
```

If you put `@Transactional` only on repository methods, each repository call gets its own transaction — there's no guarantee they're atomic together.

---

## Plain-English summary

- `PlatformTransactionManager` — Spring's transaction abstraction. Boot auto-configures the right implementation.
- `@EnableTransactionManagement` — activates `@Transactional` processing. Auto-configured by Boot.
- `@Transactional` — declarative transaction. Proxy opens, commits, or rolls back around your method.
- Class-level `@Transactional` applies to all public methods. Method-level overrides class-level.
- `readOnly = true` is a hint for optimisation — not a Java enforcement.
- Proxy rules apply: private methods and self-invocation silently bypass `@Transactional`.
- `TransactionTemplate` for programmatic control. Know it exists.
- Put `@Transactional` on the service layer — one transaction per business operation.
