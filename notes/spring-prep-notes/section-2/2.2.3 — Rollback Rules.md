# Objective 2.2.3 — Rollback Rules

## The default rollback behaviour

Spring's default rollback rule is deliberately narrow:

- **Unchecked exceptions** (`RuntimeException` and its subclasses, plus `Error`) → **automatic rollback**
- **Checked exceptions** (anything that extends `Exception` but not `RuntimeException`) → **no rollback, commits instead**

This default mirrors the original EJB convention. The reasoning was that checked exceptions represent anticipated business conditions (order not found, insufficient stock) — situations you expected and handled. Unchecked exceptions represent programming errors or unexpected failures — situations that should abort the transaction.

```java
@Transactional
public void placeOrder(Order order) throws InsufficientStockException {
    orderRepository.save(order);
    
    if (!inventoryService.reserve(order)) {
        throw new InsufficientStockException("Not enough stock"); // checked
        // transaction COMMITS — no rollback by default
    }
    
    paymentService.charge(order); // if this throws RuntimeException → ROLLBACK
}
```

⚠️ **EXAM TRAP**: Checked exceptions do **not** trigger rollback by default. This catches a lot of developers off guard and is one of the top 10 exam traps. The transaction commits even though an exception was thrown.

---

## `rollbackFor` — roll back on checked exceptions

Override the default by explicitly listing exceptions that should trigger a rollback:

```java
@Transactional(rollbackFor = InsufficientStockException.class)
public void placeOrder(Order order) throws InsufficientStockException {
    orderRepository.save(order);
    
    if (!inventoryService.reserve(order)) {
        throw new InsufficientStockException("Not enough stock");
        // NOW triggers rollback — because rollbackFor says so
    }
}
```

You can list multiple exception types:
```java
@Transactional(rollbackFor = {InsufficientStockException.class, PaymentException.class})
```

Or roll back on any exception including checked:
```java
@Transactional(rollbackFor = Exception.class)  // rolls back on anything
```

---

## `noRollbackFor` — suppress rollback on unchecked exceptions

The inverse — prevent rollback for specific unchecked exceptions that you want to treat as non-fatal:

```java
@Transactional(noRollbackFor = OptimisticLockException.class)
public void updateOrder(Order order) {
    orderRepository.save(order);
    // if OptimisticLockException is thrown, transaction COMMITS
    // (you're saying: this exception is expected and handled, don't roll back)
}
```

---

## Combining both

```java
@Transactional(
    rollbackFor = BusinessException.class,        // checked — DO roll back
    noRollbackFor = StaleDataException.class       // unchecked — DON'T roll back
)
public void processOrder(Order order) throws BusinessException { ... }
```

⚠️ **EXAM TRAP**: If both `rollbackFor` and `noRollbackFor` match the same exception (via inheritance), the **more specific** rule wins. If they're equally specific, `noRollbackFor` wins.

---

## Rollback rules with exception hierarchies

The rules apply to subclasses too:

```java
@Transactional(rollbackFor = Exception.class)
public void doWork() throws Exception { }
// rolls back on ANY exception — checked or unchecked
// because all exceptions are subclasses of Exception
```

```java
@Transactional(rollbackFor = BusinessException.class)
public void doWork() throws OrderException { }
// OrderException extends BusinessException
// → rollback applies because OrderException IS-A BusinessException
```

---

## Programmatic rollback — `TransactionStatus`

Sometimes you want to trigger a rollback without throwing an exception — for example, after a validation check inside a loop. Use `TransactionStatus.setRollbackOnly()`:

```java
@Autowired
private TransactionTemplate transactionTemplate;

public void processOrders(List<Order> orders) {
    transactionTemplate.execute(status -> {
        for (Order order : orders) {
            if (!isValid(order)) {
                status.setRollbackOnly(); // marks for rollback without throwing
                return null;
            }
            orderRepository.save(order);
        }
        return null;
    });
}
```

With declarative `@Transactional`, you can inject `TransactionAspectSupport` to access the current status — but this is rarely needed and mostly out of scope for the exam. Know that `setRollbackOnly()` exists as a concept.

---

## `@Transactional` on test methods — automatic rollback

Covered more in 2.2.4, but worth anchoring here: `@Transactional` on a test method rolls back automatically after the test completes — regardless of whether the test passed or failed, and regardless of what exceptions were thrown. The rollback rules above don't apply in test context — rollback is unconditional.

---

## Summary of rollback behaviour

| Exception type | Default behaviour | Override with |
|---|---|---|
| `RuntimeException` (unchecked) | Rollback | `noRollbackFor` to suppress |
| `Error` | Rollback | `noRollbackFor` to suppress |
| Checked `Exception` | **Commit** | `rollbackFor` to trigger rollback |

---

## Plain-English summary

- Default: unchecked exceptions roll back, checked exceptions **do not**.
- `rollbackFor` — add specific checked exceptions to the rollback set.
- `noRollbackFor` — remove specific unchecked exceptions from the rollback set.
- Rules apply to subclasses via inheritance. More specific rule wins on conflict.
- `setRollbackOnly()` — programmatic rollback without throwing an exception.
- Test `@Transactional` always rolls back unconditionally — rollback rules don't apply.
