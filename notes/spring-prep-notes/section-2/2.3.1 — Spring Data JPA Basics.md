# Objective 2.3.1 — Spring Data JPA Basics

## What Spring Data JPA adds on top of JPA

JPA (Java Persistence API) is the standard Java specification for ORM — mapping objects to relational database tables. Hibernate is the most common JPA implementation. Spring Data JPA sits on top of JPA and eliminates the boilerplate of writing `EntityManager` calls manually. You define a repository interface, Spring generates the implementation at startup.

The exam tests the essentials — entity mapping annotations, basic relationships, auto-configuration properties, and fetch types. Deep JPA internals (Criteria API, second-level cache, cascade subtleties) are explicitly out of scope.

---

## Entity mapping — the essentials

```java
@Entity                          // marks this class as a JPA entity
@Table(name = "orders")          // optional — defaults to class name "Order"
public class Order {

    @Id                          // primary key
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // DB auto-increment
    private Long id;

    @Column(name = "customer_name", nullable = false)
    private String customerName;

    @Column(precision = 10, scale = 2)
    private BigDecimal total;

    private String status;       // @Column optional — defaults to field name

    // JPA requires a no-arg constructor
    protected Order() { }

    public Order(String customerName, BigDecimal total) {
        this.customerName = customerName;
        this.total = total;
        this.status = "PENDING";
    }

    // getters...
}
```

**`@GeneratedValue` strategies:**
- `IDENTITY` — relies on database auto-increment column (MySQL, PostgreSQL serial)
- `SEQUENCE` — uses a database sequence (PostgreSQL, Oracle)
- `AUTO` — JPA picks a strategy based on the database dialect (default)
- `TABLE` — uses a separate table to simulate sequences (rarely used)

⚠️ **EXAM TRAP**: JPA requires a **no-arg constructor** on every entity — it uses reflection to instantiate entities when loading from the database. It doesn't need to be public — `protected` is fine and is actually the recommended visibility (prevents accidental use in application code).

## Basic relationships

**`@ManyToOne` — many orders belong to one customer:**

```java
@Entity
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)  // recommended — load customer only when accessed
    @JoinColumn(name = "customer_id")   // foreign key column in orders table
    private Customer customer;
}
```

**`@OneToMany` — one customer has many orders:**

```java
@Entity
public class Customer {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
    // mappedBy = the field on the OTHER side that owns the relationship
    private List<Order> orders = new ArrayList<>();
}
```

`mappedBy` tells JPA that the `Order.customer` field owns the foreign key — `Customer` is the inverse side. The table structure is defined by the owning side (`@ManyToOne`).

## Fetch types — `LAZY` vs `EAGER`

**`EAGER`** — load the related entity immediately when the parent is loaded. One query (or a join) fetches everything.

**`LAZY`** — load the related entity only when it's first accessed. A separate query fires at that point.

Default fetch types by relationship:
| Relationship | Default fetch |
|---|---|
| `@ManyToOne` | `EAGER` |
| `@OneToOne` | `EAGER` |
| `@OneToMany` | `LAZY` |
| `@ManyToMany` | `LAZY` |

⚠️ **EXAM TRAP**: `@ManyToOne` defaults to `EAGER` — this surprises people. Every time you load an `Order`, JPA also loads the `Customer` by default. In a list of 100 orders, that's 100 extra queries for customers (the N+1 problem). Best practice is to override with `fetch = FetchType.LAZY` explicitly.

⚠️ **EXAM TRAP**: Accessing a `LAZY` association outside of a transaction throws `LazyInitializationException` — the persistence context is closed by the time you try to access the data. This is a classic Spring Data JPA production bug.

## Spring Boot auto-configuration for JPA

Spring Boot auto-configures JPA when `spring-boot-starter-data-jpa` is on the classpath. Key properties:

```properties
# DataSource
spring.datasource.url=jdbc:postgresql://localhost:5432/orders
spring.datasource.username=postgres
spring.datasource.password=secret

# JPA / Hibernate
spring.jpa.hibernate.ddl-auto=update      # schema management
spring.jpa.show-sql=true                  # log SQL statements
spring.jpa.properties.hibernate.format_sql=true  # pretty-print SQL
spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect
```

**`spring.jpa.hibernate.ddl-auto` values:**
- `none` — do nothing to the schema
- `validate` — validate schema matches entities, throw if not
- `update` — update schema to match entities (adds columns, doesn't drop)
- `create` — drop and recreate schema on startup
- `create-drop` — create on startup, drop on shutdown (default for embedded DBs)

⚠️ **EXAM TRAP**: `create-drop` is the default when using an embedded database (H2). `none` is the safe default for production. Using `create` or `create-drop` in production drops your data on restart.

## Entity scanning

Spring Boot scans for `@Entity` classes in the same package as the main application class and sub-packages — the same rule as `@ComponentScan`. Entities don't need any extra annotation to be found; `@Entity` is enough.

If your entities live outside the main package, configure explicitly:
```java
@SpringBootApplication
@EntityScan("com.example.domain")  // explicit entity scanning
public class Application { }
```

## `TestEntityManager` in `@DataJpaTest`

Covered briefly in 2.2.4 — `TestEntityManager` is a test-specific wrapper around `EntityManager` that makes it easier to set up test data:

```java
@DataJpaTest
class OrderRepositoryTest {

    @Autowired
    private TestEntityManager entityManager;

    @Autowired
    private OrderRepository orderRepository;

    @Test
    void shouldFindByStatus() {
        entityManager.persist(new Order("customer-1", BigDecimal.TEN));
        entityManager.flush();  // write to DB within the transaction

        List<Order> pending = orderRepository.findByStatus("PENDING");
        assertThat(pending).hasSize(1);
    }
}
```

`flush()` forces JPA to write pending changes to the database within the current transaction — necessary when you want to query for data you just persisted in the same test.

## Plain-English summary

- `@Entity` marks a class as a JPA entity. `@Id` marks the primary key. `@GeneratedValue` auto-generates it.
- JPA requires a no-arg constructor on every entity — `protected` visibility is fine.
- `@ManyToOne` owns the foreign key. `@OneToMany` uses `mappedBy` to point back to the owning side.
- `@ManyToOne` defaults to `EAGER`. `@OneToMany` defaults to `LAZY`. Override `@ManyToOne` to `LAZY` in practice.
- `spring.jpa.hibernate.ddl-auto` controls schema management. `create-drop` for dev/test, `none` for prod.
- Boot scans for entities in the main package and sub-packages. Use `@EntityScan` to override.
- `LazyInitializationException` = accessing a lazy association outside a transaction.
