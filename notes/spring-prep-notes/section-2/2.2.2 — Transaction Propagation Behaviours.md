# Objective 2.2.2 ŌĆö Transaction Propagation Behaviours

## What propagation controls

When a `@Transactional` method calls another `@Transactional` method, propagation defines what happens to the transaction context. Does the second method join the existing transaction? Start a new one? Refuse to run if one exists?

This is a HIGH VALUE topic ŌĆö the exam tests specific propagation behaviours and their edge cases in detail.

---

## The six propagation behaviours

### `REQUIRED` ŌĆö the default

Join the existing transaction if one exists. Start a new one if there isn't one.

```java
@Transactional(propagation = Propagation.REQUIRED)  // this is the default
public void placeOrder(Order order) {
    orderRepository.save(order);        // participates in caller's transaction
    paymentService.charge(order);       // also participates in same transaction
}
```

Both the outer and inner method share **one** transaction. If the inner method rolls back, the entire transaction rolls back ŌĆö including the outer method's work.

ŌÜĀ’ĖÅ **EXAM TRAP**: With `REQUIRED`, if the inner method marks the transaction for rollback (by throwing an unchecked exception), the **outer** method cannot commit ŌĆö even if it catches the exception. The transaction is marked `rollbackOnly` and will roll back regardless.

```java
@Transactional
public void outer() {
    try {
        inner(); // throws RuntimeException, marks tx as rollbackOnly
    } catch (RuntimeException e) {
        // caught the exception ŌĆö but too late
        // transaction is already marked rollbackOnly
        // outer() will still roll back on commit attempt
    }
    // Spring throws UnexpectedRollbackException here
}

@Transactional(propagation = Propagation.REQUIRED)
public void inner() {
    throw new RuntimeException("failed");
}
```

---

### `REQUIRES_NEW` ŌĆö suspend and start fresh

Suspend the existing transaction (if any) and start a brand new, independent transaction. The new transaction commits or rolls back independently.

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveAuditLog(AuditEntry entry) {
    auditRepository.save(entry);
    // this transaction commits independently
    // even if the caller's transaction rolls back, this audit log is saved
}
```

Use case: audit logging, notifications ŌĆö work that must persist even if the main transaction fails.

ŌÜĀ’ĖÅ **EXAM TRAP**: `REQUIRES_NEW` **suspends** the outer transaction ŌĆö it doesn't commit it. The outer transaction is paused while the inner one runs, then resumed afterward. If the inner transaction rolls back, the outer transaction is unaffected (and vice versa).

---

### `NESTED` ŌĆö savepoint within the existing transaction

Executes within the existing transaction but marks a **savepoint** at the start. If the nested transaction rolls back, it rolls back only to the savepoint ŌĆö not the entire outer transaction.

```java
@Transactional(propagation = Propagation.NESTED)
public void processLineItem(LineItem item) {
    // runs inside the outer transaction
    // if this fails, only this line item's work is rolled back
    // the outer transaction can continue with other line items
}
```

ŌÜĀ’ĖÅ **EXAM TRAP**: `NESTED` requires a `DataSourceTransactionManager` with a JDBC connection that supports savepoints. It does **not** work with JPA/Hibernate transaction managers. If there's no existing transaction, it behaves like `REQUIRED` and starts a new one.

---

### `SUPPORTS` ŌĆö optional transaction

Run within a transaction if one exists. Run without a transaction if there isn't one.

```java
@Transactional(propagation = Propagation.SUPPORTS)
public Order findOrder(Long id) {
    // runs in transaction if caller has one
    // runs without transaction if caller doesn't
    return orderRepository.findById(id);
}
```

Use case: read operations that are fine with or without a transaction context.

---

### `NOT_SUPPORTED` ŌĆö suspend any existing transaction

Suspend any existing transaction and run without a transaction. Resume the outer transaction afterward.

```java
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void sendNotification(Notification notification) {
    // explicitly runs outside any transaction
    // even if called from within a transaction
    notificationService.send(notification);
}
```

Use case: operations that shouldn't be part of a transaction ŌĆö like sending an email (you don't want the email send to block a commit, and you can't roll back an email).

---

### `NEVER` ŌĆö forbid transactions

Throw an exception if a transaction exists. Run without a transaction if there isn't one.

```java
@Transactional(propagation = Propagation.NEVER)
public void batchExport(ExportRequest request) {
    // throws IllegalTransactionStateException if called within a transaction
}
```

Use case: operations that must never run inside a transaction ŌĆö bulk exports, long-running reads.

---

### `MANDATORY` ŌĆö require an existing transaction

Throw an exception if no transaction exists. Join the existing one if there is one.

```java
@Transactional(propagation = Propagation.MANDATORY)
public void updateInventory(Order order) {
    // throws IllegalTransactionStateException if called without an active transaction
    // enforces that callers must always provide a transaction context
}
```

Use case: helper methods that should only be called from within a transaction ŌĆö enforces correct usage at runtime.

## The three most tested ŌĆö side by side

| Propagation | Existing tx exists | No tx exists |
|---|---|---|
| `REQUIRED` | Join it | Start new |
| `REQUIRES_NEW` | Suspend it, start new | Start new |
| `NESTED` | Savepoint within it | Start new |
| `SUPPORTS` | Join it | Run without tx |
| `NOT_SUPPORTED` | Suspend it, run without tx | Run without tx |
| `NEVER` | Throw exception | Run without tx |
| `MANDATORY` | Join it | Throw exception |

## The `rollbackOnly` trap ŌĆö deeper detail

This is the most subtle `REQUIRED` edge case:

```java
@Transactional
public void processOrders(List<Order> orders) {
    for (Order order : orders) {
        try {
            processOne(order); // REQUIRED ŌĆö joins outer transaction
        } catch (Exception e) {
            log.warn("Failed order: " + order.getId());
            // you think you've handled it ŌĆö but the transaction is now rollbackOnly
        }
    }
    // Spring throws UnexpectedRollbackException on commit
    // ALL orders roll back, not just the failed one
}

@Transactional(propagation = Propagation.REQUIRED)
public void processOne(Order order) {
    throw new RuntimeException("failed");
}
```

The fix: use `REQUIRES_NEW` on `processOne()` so each order runs in its own independent transaction. A failure in one doesn't affect the others.

---

## Plain-English summary

- `REQUIRED` ŌĆö join existing or start new. Both methods share one transaction. Inner rollback marks entire tx as rollbackOnly.
- `REQUIRES_NEW` ŌĆö suspend existing, start independent new tx. Inner and outer commit/rollback independently.
- `NESTED` ŌĆö savepoint within existing tx. Inner rollback goes back to savepoint only. JDBC only ŌĆö no JPA.
- `SUPPORTS` ŌĆö use tx if available, proceed without if not.
- `NOT_SUPPORTED` ŌĆö suspend any tx, run without.
- `NEVER` ŌĆö throw if tx exists.
- `MANDATORY` ŌĆö throw if no tx exists.
- The exam focuses on `REQUIRED`, `REQUIRES_NEW`, and `NESTED` ŌĆö know their behaviour cold.
