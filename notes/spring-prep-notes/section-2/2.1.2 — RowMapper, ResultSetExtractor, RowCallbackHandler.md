# Objective 2.1.2 — RowMapper, ResultSetExtractor, RowCallbackHandler

## Why three different interfaces exist

All three deal with processing a `ResultSet` — the raw data coming back from a SQL query. The difference is in **what you're producing** and **who controls the iteration**.

---

## `RowMapper<T>` — one row to one object

You've already seen this in 2.1.1. `RowMapper` converts a single row into a single object. `JdbcTemplate` controls the iteration — it calls your `RowMapper` once per row and collects the results.

```java
public interface RowMapper<T> {
    T mapRow(ResultSet rs, int rowNum) throws SQLException;
}
```

```java
// As a named class
public class OrderRowMapper implements RowMapper<Order> {
    @Override
    public Order mapRow(ResultSet rs, int rowNum) throws SQLException {
        return new Order(
            rs.getLong("id"),
            rs.getString("customer_name"),
            rs.getBigDecimal("total")
        );
    }
}

// Usage
List<Order> orders = jdbcTemplate.query("SELECT * FROM orders", new OrderRowMapper());

// Or as a lambda (more common)
List<Order> orders = jdbcTemplate.query(
    "SELECT * FROM orders",
    (rs, rowNum) -> new Order(rs.getLong("id"), rs.getString("customer_name"), rs.getBigDecimal("total"))
);
```

**Use when**: you want to map each row to an object and get back a `List<T>`. This covers the vast majority of query use cases.

---

## `ResultSetExtractor<T>` — you control the whole ResultSet

`ResultSetExtractor` gives you the entire `ResultSet` at once and expects you to return a single result — which can be any type, including a complex aggregate built from multiple rows.

```java
public interface ResultSetExtractor<T> {
    T extractData(ResultSet rs) throws SQLException, DataAccessException;
}
```

```java
// Building a Map from the entire ResultSet
Map<Long, List<Order>> ordersByCustomer = jdbcTemplate.query(
    "SELECT * FROM orders ORDER BY customer_id",
    rs -> {
        Map<Long, List<Order>> result = new LinkedHashMap<>();
        while (rs.next()) {  // YOU control the iteration
            Long customerId = rs.getLong("customer_id");
            Order order = new Order(rs.getLong("id"), rs.getString("customer_name"));
            result.computeIfAbsent(customerId, k -> new ArrayList<>()).add(order);
        }
        return result;
    }
);
```

Notice the `while (rs.next())` — you're responsible for iterating the `ResultSet` yourself. `RowMapper` never exposes this; `ResultSetExtractor` always requires it.

⚠️ **EXAM TRAP**: With `ResultSetExtractor`, **you** call `rs.next()`. With `RowMapper`, `JdbcTemplate` calls it for you. Forgetting to iterate in a `ResultSetExtractor` means you process zero rows.

**Use when**: you need to build a single complex result from multiple rows — a grouped map, a tree structure, or any aggregate that doesn't map cleanly to a flat list.

---

## `RowCallbackHandler` — process rows, return nothing

`RowCallbackHandler` is called once per row like `RowMapper`, but returns nothing. It's for side effects — writing to a file, streaming to an output, accumulating into a stateful object.

```java
public interface RowCallbackHandler {
    void processRow(ResultSet rs) throws SQLException;
}
```

```java
// Streaming results to a file — no return value needed
List<String> reportLines = new ArrayList<>();

jdbcTemplate.query(
    "SELECT customer_name, total FROM orders WHERE status = ?",
    rs -> {
        // called once per row, void return
        reportLines.add(rs.getString("customer_name") + ": " + rs.getBigDecimal("total"));
    },
    "PENDING"
);
```

`JdbcTemplate` controls iteration (like `RowMapper`), but your callback produces no return value. The result is accumulated in an external data structure or side effect.

**Use when**: you want row-by-row processing without producing a return value — streaming, writing, accumulating into external state.

---

## The key distinction — who controls iteration and what's returned

| Interface | Controls iteration | Returns |
|---|---|---|
| `RowMapper<T>` | `JdbcTemplate` | One object per row → collected into `List<T>` |
| `ResultSetExtractor<T>` | You (`rs.next()`) | One result for the whole ResultSet (any type) |
| `RowCallbackHandler` | `JdbcTemplate` | Nothing (`void`) |

---

## Which `JdbcTemplate` method uses which

```java
// RowMapper
List<Order> orders = jdbcTemplate.query(sql, rowMapper);
Order order = jdbcTemplate.queryForObject(sql, rowMapper, args);

// ResultSetExtractor
Map<Long, List<Order>> grouped = jdbcTemplate.query(sql, resultSetExtractor);

// RowCallbackHandler
jdbcTemplate.query(sql, rowCallbackHandler);  // void return
```

All three use the same `query()` method on `JdbcTemplate` — Spring picks the right overload based on the type passed in.

---

## `BeanPropertyRowMapper` — built-in convenience mapper

Worth knowing for the exam — Spring provides a ready-made `RowMapper` that maps columns to bean properties by name (with relaxed matching: `customer_name` → `customerName`):

```java
List<Order> orders = jdbcTemplate.query(
    "SELECT * FROM orders",
    new BeanPropertyRowMapper<>(Order.class)
);
```

No custom mapping code needed when column names match field names. Uses reflection under the hood — slightly slower than a hand-written mapper but convenient for simple cases.

---

## Plain-English summary

- `RowMapper<T>` — one row in, one object out. `JdbcTemplate` iterates. Returns `List<T>`. Use for standard row-to-object mapping.
- `ResultSetExtractor<T>` — whole `ResultSet` in, one result out (any type). You iterate with `rs.next()`. Use for grouped or aggregate results.
- `RowCallbackHandler` — one row in, nothing out. `JdbcTemplate` iterates. Use for side effects — streaming, writing, accumulating externally.
- All three used via `jdbcTemplate.query()` — Spring picks the overload by type.
- `BeanPropertyRowMapper` — built-in mapper, matches column names to bean properties automatically.
