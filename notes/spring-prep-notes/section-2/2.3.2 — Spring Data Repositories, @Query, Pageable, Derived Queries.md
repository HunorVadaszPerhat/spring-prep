# Objective 2.3.2 ŌĆö Spring Data Repositories, `@Query`, `Pageable`, Derived Queries

## The repository abstraction

Spring Data JPA generates repository implementations at startup. You define an interface extending one of Spring Data's base interfaces, declare your query methods, and Spring writes the implementation ŌĆö no SQL, no `EntityManager` boilerplate.

---

## The repository hierarchy

```java
Repository<T, ID>                    // marker interface ŌĆö no methods
ŌööŌöĆŌöĆ CrudRepository<T, ID>            // save, findById, findAll, delete, count, exists
    ŌööŌöĆŌöĆ PagingAndSortingRepository<T, ID>  // adds findAll(Pageable), findAll(Sort)
        ŌööŌöĆŌöĆ JpaRepository<T, ID>     // adds flush, saveAndFlush, deleteInBatch, getOne
```

In practice, extend `JpaRepository` ŌĆö it gives you everything:

```java
@Repository  // optional ŌĆö Spring Data detects repository interfaces automatically
public interface OrderRepository extends JpaRepository<Order, Long> {
    // CRUD methods inherited ŌĆö no implementation needed
}
```

`JpaRepository<Order, Long>` ŌĆö first type parameter is the entity, second is the primary key type.

ŌÜĀ’ĖÅ **EXAM TRAP**: You don't need `@Repository` on Spring Data repository interfaces ŌĆö Spring Data registers them automatically. The annotation doesn't hurt but isn't required. It's different from plain `@Repository` on a hand-written DAO class.

## Derived query methods ŌĆö HIGH VALUE

Spring Data reads method names and translates them into JPQL at startup. This is one of the most tested features in Section 2.

**The naming convention:**

```
find + By + [property] + [keyword] + [property] + ...
```

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    // SELECT o FROM Order o WHERE o.customerName = ?1
    List<Order> findByCustomerName(String customerName);

    // SELECT o FROM Order o WHERE o.status = ?1 AND o.total > ?2
    List<Order> findByStatusAndTotalGreaterThan(String status, BigDecimal total);

    // SELECT o FROM Order o WHERE o.customerName LIKE ?1
    List<Order> findByCustomerNameContaining(String keyword);

    // SELECT o FROM Order o WHERE o.createdAt BETWEEN ?1 AND ?2
    List<Order> findByCreatedAtBetween(LocalDateTime from, LocalDateTime to);

    // SELECT o FROM Order o WHERE o.status IN ?1
    List<Order> findByStatusIn(List<String> statuses);

    // SELECT o FROM Order o WHERE o.customer.name = ?1 (nested property)
    List<Order> findByCustomerName(String name);

    // SELECT COUNT(o) FROM Order o WHERE o.status = ?1
    long countByStatus(String status);

    // DELETE FROM Order o WHERE o.status = ?1
    @Modifying
    @Transactional
    void deleteByStatus(String status);
}
```

**Common keywords:**

| Keyword | JPQL equivalent |
|---|---|
| `And` | `AND` |
| `Or` | `OR` |
| `Is`, `Equals` | `=` |
| `Between` | `BETWEEN` |
| `LessThan` | `<` |
| `GreaterThan` | `>` |
| `IsNull` | `IS NULL` |
| `IsNotNull` | `IS NOT NULL` |
| `Like` | `LIKE` |
| `Containing` | `LIKE %value%` |
| `StartingWith` | `LIKE value%` |
| `EndingWith` | `LIKE %value` |
| `In` | `IN` |
| `NotIn` | `NOT IN` |
| `OrderBy` | `ORDER BY` |
| `Not` | `!=` |

ŌÜĀ’ĖÅ **EXAM TRAP**: Spring Data parses the method name at startup. A typo or reference to a non-existent property throws `PropertyReferenceException` at startup ŌĆö not at query time. The exam may ask when this error is detected.

## `@Query` ŌĆö explicit JPQL or native SQL

When derived query names get too long or complex, use `@Query` to write the query explicitly:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    // JPQL ŌĆö uses entity class name and field names, not table/column names
    @Query("SELECT o FROM Order o WHERE o.status = :status AND o.total > :minTotal")
    List<Order> findExpensivePendingOrders(
        @Param("status") String status,
        @Param("minTotal") BigDecimal minTotal
    );

    // Positional parameters ŌĆö ?1, ?2 refer to method param position
    @Query("SELECT o FROM Order o WHERE o.customerName = ?1")
    List<Order> findByCustomer(String customerName);

    // Native SQL ŌĆö uses actual table and column names
    @Query(value = "SELECT * FROM orders WHERE status = :status",
           nativeQuery = true)
    List<Order> findByStatusNative(@Param("status") String status);
}
```

ŌÜĀ’ĖÅ **EXAM TRAP**: JPQL uses **entity class names and field names** ŌĆö not table names and column names. `FROM Order` not `FROM orders`. `o.customerName` not `o.customer_name`. Native queries (`nativeQuery = true`) use the actual SQL table and column names.

---

## `@Modifying` ŌĆö for UPDATE and DELETE queries

Write operations (`UPDATE`, `DELETE`) with `@Query` require `@Modifying`:

```java
@Modifying
@Transactional
@Query("UPDATE Order o SET o.status = :status WHERE o.id = :id")
int updateStatus(@Param("id") Long id, @Param("status") String status);
```

ŌÜĀ’ĖÅ **EXAM TRAP**: `@Modifying` is required for any `@Query` that writes data. Without it, Spring Data treats the query as a SELECT and throws an exception. Also needs `@Transactional` ŌĆö either on the method or inherited from a `@Transactional` service calling it.

## `Pageable` and `Page<T>` ŌĆö pagination

Pass a `Pageable` parameter to get paginated results:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    // Returns a Page ŌĆö includes total count, total pages, current page data
    Page<Order> findByStatus(String status, Pageable pageable);

    // Returns a Slice ŌĆö lighter, no total count query
    Slice<Order> findByCustomerName(String name, Pageable pageable);

    // Returns List ŌĆö just the data, no pagination metadata
    List<Order> findByStatus(String status, Pageable pageable);
}
```

Creating a `Pageable`:

```java
// Page 0, 10 items per page, sorted by total descending
Pageable pageable = PageRequest.of(0, 10, Sort.by(Sort.Direction.DESC, "total"));

Page<Order> page = orderRepository.findByStatus("PENDING", pageable);

// Page metadata
int totalPages = page.getTotalPages();
long totalElements = page.getTotalElements();
List<Order> orders = page.getContent();
boolean hasNext = page.hasNext();
```

ŌÜĀ’ĖÅ **EXAM TRAP**: Pages are **zero-indexed**. Page 0 is the first page. `PageRequest.of(0, 10)` gives you items 1ŌĆō10.

**`Page` vs `Slice`:**
- `Page<T>` ŌĆö fires a COUNT query to get total elements. More expensive but gives you total pages.
- `Slice<T>` ŌĆö no COUNT query. Only knows if there's a next page. More efficient for infinite scroll.

---

## `Sort` ŌĆö standalone sorting

For sorting without pagination:

```java
// Sort by single field
List<Order> orders = orderRepository.findByStatus("PENDING",
    Sort.by("total"));

// Sort descending
List<Order> orders = orderRepository.findByStatus("PENDING",
    Sort.by(Sort.Direction.DESC, "total"));

// Multiple sort fields
List<Order> orders = orderRepository.findByStatus("PENDING",
    Sort.by("customerName").and(Sort.by(Sort.Direction.DESC, "total")));
```

---

## Combining derived queries with `Pageable`

```java
// Derived query + pagination together
Page<Order> findByStatusOrderByTotalDesc(String status, Pageable pageable);
```

The `OrderBy` in the method name and the `Sort` in `Pageable` can coexist ŌĆö method name sort applies first.

---

## Plain-English summary

- Extend `JpaRepository<Entity, ID>` ŌĆö inherits all CRUD methods, no implementation needed.
- `@Repository` optional on Spring Data interfaces ŌĆö Spring Data finds them automatically.
- Derived query methods ŌĆö Spring parses the method name into JPQL at startup. Typos fail at startup, not runtime.
- `@Query` ŌĆö explicit JPQL (entity/field names) or native SQL (table/column names). Named params with `@Param`.
- `@Modifying` + `@Transactional` required for UPDATE/DELETE `@Query` methods.
- `Pageable` ŌåÆ pass `PageRequest.of(page, size, sort)`. Pages are zero-indexed.
- `Page<T>` includes total count. `Slice<T>` doesn't ŌĆö more efficient for large datasets.
