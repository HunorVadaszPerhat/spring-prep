# Objective 2.1.3 — DataAccessException Hierarchy

## The problem Spring solves here

Different persistence technologies throw different exceptions. Raw JDBC throws checked `SQLException`. Hibernate throws `HibernateException`. JPA throws `PersistenceException`. If your code catches these directly, it's coupled to the specific persistence technology — swap from JDBC to JPA and you're rewriting exception handling everywhere.

Spring solves this by mapping all persistence exceptions into a single, unified, **unchecked** exception hierarchy rooted at `DataAccessException`. Your code catches Spring exceptions, not technology-specific ones. Swap the persistence layer and the exception handling stays the same.

---

## Why unchecked matters

`SQLException` is a checked exception — Java forces you to either catch it or declare it in your method signature. This pollutes every layer of your application with persistence concerns.

`DataAccessException` extends `RuntimeException` — unchecked. You handle it where it makes sense, not where the compiler forces you. Most persistence errors are unrecoverable anyway (connection failures, constraint violations) — forcing the caller to handle them at every call site adds noise without value.

---

## The hierarchy

```
RuntimeException
└── DataAccessException
    ├── NonTransientDataAccessException   — retrying won't help
    │   ├── DataIntegrityViolationException  — FK violation, not-null, check constraint
    │   │   └── DuplicateKeyException        — unique constraint violation
    │   ├── DataRetrievalFailureException
    │   │   ├── EmptyResultDataAccessException    — queryForObject() got 0 rows
    │   │   └── IncorrectResultSizeDataAccessException — wrong number of rows
    │   └── PermissionDeniedDataAccessException  — no DB permission
    ├── TransientDataAccessException      — retrying might help
    │   ├── TransientDataAccessResourceException — transient connection issue
    │   └── ConcurrencyFailureException          — deadlock, optimistic lock failure
    └── NonTransientDataAccessResourceException
        └── CannotGetJdbcConnectionException     — connection pool exhausted
```

You don't need the full tree memorised — know the key leaves the exam tests.

---

## The exceptions the exam cares about

**`EmptyResultDataAccessException`** — thrown by `queryForObject()` when the query returns zero rows. The name says it — the result set was empty.

**`IncorrectResultSizeDataAccessException`** — thrown by `queryForObject()` when the query returns more than one row (or the wrong number in general). `EmptyResultDataAccessException` is actually a subclass of this one.

**`DuplicateKeyException`** — unique constraint or primary key violation. Spring translates the database-specific error code into this.

**`DataIntegrityViolationException`** — parent of `DuplicateKeyException`. Covers foreign key violations, not-null violations, check constraint failures.

**`CannotGetJdbcConnectionException`** — connection pool exhausted or database unreachable.

---

## How exception translation works

Two things must be in place for translation to happen:

**1. Use `JdbcTemplate`** — it wraps every database call in a try/catch and runs the exception through a `SQLExceptionTranslator`. Raw JDBC code does not get translation.

**2. Annotate the repository with `@Repository`** — this activates a `PersistenceExceptionTranslationPostProcessor` (a `BeanPostProcessor`) that translates exceptions thrown from `@Repository`-annotated beans. Without `@Repository`, exceptions from JPA or Hibernate won't be translated.

```java
@Repository  // activates exception translation
public class OrderRepository {

    private final JdbcTemplate jdbcTemplate;

    public Order findById(Long id) {
        return jdbcTemplate.queryForObject(  // JdbcTemplate translates internally
            "SELECT * FROM orders WHERE id = ?",
            (rs, rowNum) -> mapOrder(rs),
            id
        );
        // if no row found: EmptyResultDataAccessException (unchecked)
        // not SQLException (checked)
    }
}
```

⚠️ **EXAM TRAP**: `JdbcTemplate` translates `SQLException` internally by itself — you don't need `@Repository` for JDBC translation. But for JPA/Hibernate exceptions to be translated, `@Repository` is required because those exceptions escape the template and need the `BeanPostProcessor` to catch and translate them.

## Catching specific exceptions

Because the hierarchy is unchecked, you catch only what you can handle:

```java
public Order findById(Long id) {
    try {
        return jdbcTemplate.queryForObject(
            "SELECT * FROM orders WHERE id = ?",
            (rs, rowNum) -> mapOrder(rs),
            id
        );
    } catch (EmptyResultDataAccessException e) {
        return null;  // or Optional.empty(), or throw a domain exception
    }
}
```

You could also catch `DataAccessException` to handle any persistence error generically — but catching the specific subclass is cleaner.

---

## Plain-English summary

- `DataAccessException` is an unchecked hierarchy that wraps all persistence-technology-specific exceptions.
- Benefit: swap JDBC for JPA and exception handling code doesn't change.
- Benefit: unchecked — no forced catch blocks at every call site.
- Key exceptions: `EmptyResultDataAccessException` (zero rows), `IncorrectResultSizeDataAccessException` (wrong row count), `DuplicateKeyException` (unique violation), `DataIntegrityViolationException` (constraint violation), `CannotGetJdbcConnectionException` (no connection).
- `JdbcTemplate` translates `SQLException` automatically. `@Repository` is needed for JPA/Hibernate exception translation.
